## 📝 简答题题库

**Q：请在 PVP、SMP、MPP、DSM 和 COW 中任选一种，简要描述其特点。**\
以 PVP 为例: 
* 系统中包含了少量高性能专门设计定制的向量处理器 VP，每个至少具有 1Gflops 的处理能力。
* 存储器以兆字节每秒（MB/s）的速度向处理器提供数据。
* 向量处理器 VP 和共享存储模块通过高带宽的交叉开关网络互连。
* 通常不使用高速缓存，而是使用大量的向量寄存器 and 指令缓冲器。

**Q：从存储角度来看，MIMD 结构可以分为哪几类？**
* 单地址空间共享存储：包含均匀存储访问（如 PVP、SMP）和非均匀存储访问（如 DSM）。
* 多地址空间非共享存储：所有存储器都是私有的（如 MPP、COW）。

**Q：什么是多处理机 Cache 一致性问题？**
* 允许共享数据进入 Cache，就可能出现多个处理器的 Cache 中都有同一存储块的副本。
* 当其中某个处理器对其 Cache 中的数据进行修改后，就会使得其 Cache 中的数据与其他 Cache 中的数据不一致，从而产生一致性问题。

**Q：请比较说明写作废协议和写更新协议性能上的差别。**
* 在对同一个数据进行多次写操作而中间无读操作的情况下，写更新协议需进行多次写广播操作，而写作废协议只需一次作废操作。
* 在对同一 Cache 块的多个字进行写操作的情况下，写更新协议对于每一个写操作都要进行一次广播，而写作废协议仅在对该块的第一次写时进行作废操作即可。
* 写作废是针对 Cache 块进行操作，而写更新则是针对字（或字节）进行。
* 延迟时间对比：从处理器 A 进行写操作后，到处理器 B 能读到该写入数据之间的延迟时间，写更新协议的延迟时间较小。

**Q：请简述监听式协议的原理。**
* Cache 通常连在共享存储器的总线上。
* 当某个 Cache 需要访问存储器时，它会把请求放到总线上广播出去。
* 其他各个 Cache 控制器通过监听总线（它们一直在监听），来判断它们是否有总线上请求的数据块。
* 如果有，就进行相应的操作（如作废本地副本或提供数据）。

**Q：请简述目录式协议的原理。**
* 物理存储器中数据块的共享状态被保存在一个称为目录（Directory）的数据结构中，用于集中管理共享信息。
* 避免了总线广播操作，提升了系统的可扩放性。

**Q：请解释，在目录式协议中，什么是本地节点、宿主节点、远程节点和共享集合？**
* 本地节点：发出访问请求的节点。
* 宿主节点：包含所访问的存储单元及其目录项的节点。
* 远程节点：Cache 中拥有该块的副本的节点。
* 共享集合：使用位向量记录拥有其副本的处理器的集合。

**Q：请简述目录式协议中，目录的三种结构。**
* 全映像目录：每一个目录项都包含一个 N 位的位向量，其每一位对应于一个处理机。
* 有限映像目录：采用位数固定的目录项目，通过对同一数据块在所有 Cache 中的副本总数进行限制来实现。
* 链式目录：用一个目录指针链表来表示共享集合。当一个数据块的副本数增加（或减少）时，指针链表随之变长（或变短）。

## 🔢 计算题专项

### 题型说明
本章计算题通常考察**监听式 Cache 一致性协议的状态转换表填写**。题目会给定处理器的 Cache 初始状态及一系列访存指令，要求逐步推导每个指令的本地请求、状态转换、本地及总线操作，并给出最终的所有 Cache 状态。

### 解题方法与步骤

1.  **判定命中与失效**：根据给定的读写地址计算块号及标记，比对目标 CPU 当前 Cache 的标记。若一致且状态非无效（`00`），则为命中（Hit）；否则为失效（Miss）。

2.  **确定本地状态跳变**：根据监听式写作废（Write Invalidate）协议的三状态机：
    * 读失效（Read Miss）：无效 `00` -> 共享 `10`。
    * 写命中（Write Hit）：共享 `10` -> 修改 `11`。
    * 写失效（Write Miss）：无效 `00` -> 修改 `11`。
    * 读命中（Read Hit）：共享 `10` -> 共享 `10`。

3.  **确定本地及总线操作**：
    * 若 Read Miss，发出 `将ReadMiss放到总线上`（对应选项 1）。
    * 若 Write Miss，发出 `将WriteMiss放到总线上`（对应选项 2）。
    * 若 S 状态下 Write Hit，发出 `将Invalidate放总线上`（对应选项 3）。
    * 若正常命中，只需 `读本地数据 / 写本地数据`。

4.  **同步更新远端 Cache**：每完成一步访存，必须检查总线上的操作（如 Invalidate 或 WriteMiss），若总线发出作废信号，其他所有拥有该块的 CPU Cache 对应行的状态必须强行置为 `00`（无效）。

### 课件与作业例题精解

**背景条件：**
某多机系统为共享主存均匀存储访问系统，配置如下：采用监听式写作废协议和写回法，4 个节点 (CPU00~CPU11)。Cache 块数为 16，大小 16B，直接映射。Cache 状态定义：`Invalid (00)`，`Shared (10)`，`Modified (11)`。本地操作选项：(1) 将 ReadMiss 放总线；(2) 将 WriteMiss 放总线；(3) 将 Invalidate 放总线；(4) 读本地数据；(5) 写本地数据。

**【初始状态】**
CPU00: 块0H=`12FFH / 00`, 块1H=`28BEH / 10`
CPU01: 块0H=`0023H / 11`, 块1H=`34EAH / 10`
CPU10: 块0H=`0124H / 10`, 块1H=`28BEH / 10`
CPU11: 块0H=`8A7FH / 00`, 块1H=`34EAH / 10`

**步骤分析与填表：**

| 访存请求 | 本地请求 | 状态转换 | 本地操作 |
| :--- | :--- | :--- | :--- |
| **访存请求1：CPU00 读 002302H** | ReadMiss | 00 10 | 1 |
| **访存请求2：CPU01 写 34EA1FH** | WriteHit | 10 11 | 3 |
| **访存请求3：CPU10 读 28BE1AH** | ReadHit | 10 10 | 4 |
| **访存请求4：CPU11 写 28BE11H** | WriteMiss | 00 11 | 2 |

**访存动作结束后，四个节点 Cache 状态结果：**

| 节点 | 块 0H 标记 | 块 0H 状态 | 块 1H 标记 | 块 1H 状态 |
| :--- | :--- | :--- | :--- | :--- |
| **CPU00** | 0023H | 10 | 28BEH | 00 |
| **CPU01** | 0023H | 10 | 34EAH | 11 |
| **CPU10** | 0124H | 10 | 28BEH | 00 |
| **CPU11** | 8A7FH | 00 | 28BEH | 11 |

### ⚠️ 易错提醒

* **被动失效遗漏**：当某个 CPU 发生 `WriteHit` (S->M) 或 `WriteMiss` (I->M) 时，它会在总线上发送作废信号，此时**必须**检查其他 CPU 对应位置是否拥有相同标记的数据块。如果有，必须同步将其状态修改为 `00`（Invalid），极易在最终状态填表中遗漏更新其他 CPU。
* **状态转换终点写错**：注意 `WriteMiss` 发生时，状态是直接从 `00` 跃迁到 `11`（修改），而非经过 `10`。
