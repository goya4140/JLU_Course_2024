# 监听协议可视化操作设计

## 设计目标

参考 Tomasulo 算法可视化项目的操作方式，监听协议可视化应当从“静态填表题”变成“逐步推演工具”。用户不只是看答案，而是按访存序列一步一步观察：

- 地址如何拆成 Tag / Index / Offset。
- 本地 Cache 如何判断 ReadHit、ReadMiss、WriteHit、WriteMiss。
- 哪些动作会放到共享总线上。
- 其他 CPU 如何监听总线并更新状态。
- 最终状态表为什么得到这些 Tag 和 MSI 状态。

本工具建议采用“左侧例题配置 + 中央系统状态 + 右侧推演解释”的三栏布局，并保留 Tomasulo 项目的“重新开始、单步、自动运行、视图切换”交互习惯。

## 页面结构

### 顶部控制区

放置全局操作按钮和当前进度：

- `重新开始`：恢复例题初始 Cache 状态。
- `上一步`：回退到上一阶段，方便纠错。
- `单步`：推进一个微步骤。
- `自动运行`：连续播放完整推演。
- `暂停`：停止自动播放。
- `显示答案`：直接展示所有空缺答案。
- `练习模式 / 讲解模式` 切换：
  - 练习模式隐藏部分结论，让用户先判断。
  - 讲解模式自动展示每一步原因。

监听协议不像 Tomasulo 那样按“周期”推进，建议顶部进度显示为：

```text
访存请求 2 / 4 · 阶段：本地命中判断
```

## 主要视图设计

### 1. 左侧：例题输入与配置面板

对应 Tomasulo 项目的“程序”和“机器配置”区域。

建议包含四块：

#### 系统配置

显示并允许修改：

- CPU 数量：默认 4。
- Cache 块数：默认 16。
- 块大小：默认 16B。
- 主存大小：默认 16MB。
- 映射方式：直接映射。
- 协议：MSI。
- 写策略：写作废 + 写回。
- 数据提供策略：就近提供。

配置区要实时给出地址位数拆解：

```text
地址长度 24 位 = Tag 16 位 + Index 4 位 + Offset 4 位
```

#### 初始 Cache 状态

用可编辑表格呈现题目给出的初态：

| CPU | 块号 | Tag | 状态 |
| --- | --- | --- | --- |
| CPU00 | 0H | 12FFH | I / 00 |
| CPU00 | 1H | 28BEH | S / 10 |
| CPU01 | 0H | 0023H | M / 11 |
| CPU01 | 1H | 34EAH | S / 10 |

状态单元格建议用颜色编码：

- I / 00：灰色。
- S / 10：蓝色。
- M / 11：橙红色。

#### 访存请求序列

显示例题四条请求，并支持增删改：

```text
1. CPU00 READ  002302H
2. CPU01 WRITE 34EA1FH
3. CPU10 READ  28BE1AH
4. CPU11 WRITE 28BE11H
```

当前正在执行的请求高亮。

#### 本地操作选项

固定展示题目选项，方便学生把可视化和填空题对应起来：

1. 将 ReadMiss 放到总线上
2. 将 WriteMiss 放到总线上
3. 将 Invalidate 放到总线上
4. 读本地数据
5. 写本地数据

练习模式下，用户需要自己点选本地请求、本地操作和状态转换，再由系统判定。

### 2. 中央：系统状态可视化

这是页面核心，建议分为“CPU Cache 阵列 + 共享总线 + 主存”。

#### CPU Cache 阵列

四个 CPU 以 2 x 2 网格展示：

```text
CPU00        CPU01
CPU10        CPU11
```

每个 CPU 卡片内只显示题目相关块，默认显示 0H 和 1H：

```text
CPU01
┌────┬───────┬──────┐
│ 块号 │ Tag   │ MSI  │
├────┼───────┼──────┤
│ 0H │ 0023H │ M/11 │
│ 1H │ 34EAH │ S/10 │
└────┴───────┴──────┘
```

可视化重点：

- 当前发起请求的 CPU 外框高亮。
- 当前访问的 Index 行高亮。
- Tag 匹配的单元格用描边提示。
- 状态变化用动画或闪烁提示，例如 `M/11 -> S/10`。
- 被 Invalidate 的块从蓝色/红色淡出为灰色。

#### 共享总线

放在四个 CPU 中间或下方，用一条横向总线表示。

总线上显示当前动作：

```text
Bus: ReadMiss(Tag=0023H, Index=0H) from CPU00
```

总线动作颜色建议：

- ReadMiss：青色。
- WriteMiss：紫色。
- Invalidate：红色。
- WriteBack：黄色。
- Local Read / Local Write：不上总线，可在 CPU 内部显示，不占用总线。

#### 主存与数据提供者

在总线旁边展示数据来源：

```text
数据来源：CPU01 Cache M 块写回并提供
```

或者：

```text
数据来源：主存提供
```

这能专门解决易错点：“有 M 状态副本时不能直接认为主存最新”。

### 3. 右侧：推演说明与填空答案

对应 Tomasulo 项目右侧的事件列表和状态表。

建议包含三个 Tab：

#### Tab A：步骤解释

按当前微步骤显示事件：

```text
访存请求 1：CPU00 读 002302H

1. 地址拆解
   Offset = 2H，Index = 0H，Tag = 0023H

2. 本地检查
   CPU00 的 0H 块 Tag=12FFH，状态=I，因此读缺失。

3. 本地请求
   ReadMiss

4. 总线动作
   将 ReadMiss 放到总线上，答案 (3)=1。

5. 监听响应
   CPU01 的 0H 块 Tag=0023H 且状态=M。
   CPU01 写回并降级为 S。
```

#### Tab B：填空表

直接映射题目空缺：

| 空 | 当前值 | 来源 |
| --- | --- | --- |
| (1) | ReadMiss | 请求 1 本地检查 |
| (2) | 00 10 | CPU00 I 读缺失后转 S |
| (3) | 1 | ReadMiss 上总线 |

练习模式下可以先显示空框，用户提交后再给出正确答案。

#### Tab C：状态转移规则

显示 MSI 规则速查表，并高亮当前触发的规则。

本地请求规则：

| 当前状态 | CPU 读 | CPU 写 |
| --- | --- | --- |
| I | ReadMiss，变 S | WriteMiss，变 M |
| S | ReadHit，保持 S | WriteHit，发 Invalidate，变 M |
| M | ReadHit，保持 M | WriteHit，保持 M |

监听规则：

| 当前状态 | 监听 ReadMiss | 监听 WriteMiss / Invalidate |
| --- | --- | --- |
| S | 保持 S | 变 I |
| M | 写回，变 S | 写回，变 I |
| I | 无动作 | 无动作 |

## 推进粒度设计

为了让用户真正学会推演，建议每条访存请求拆成 6 个微步骤。

### 微步骤 1：地址拆解

展示十六进制地址如何拆分：

```text
002302H
Tag    = 0023H
Index  = 0H
Offset = 2H
```

视觉上可以把地址分段染色：

```text
[0023][0][2]
 Tag  Index Offset
```

### 微步骤 2：本地 Cache 查找

高亮发起 CPU 的对应 Index 行，比较 Tag 和状态：

```text
目标 Tag = 0023H
本地 Tag = 12FFH
状态 = I
结论：ReadMiss
```

### 微步骤 3：本地请求分类

生成填空项，例如：

```text
(1) ReadMiss
```

同时在规则表中高亮 `I + CPU读 -> ReadMiss -> S`。

### 微步骤 4：本地状态变化与操作选择

展示本地状态转换：

```text
CPU00[0H]：I/00 -> S/10
Tag：12FFH -> 0023H
本地操作：(3) 1，将 ReadMiss 放到总线上
```

### 微步骤 5：总线广播与监听

总线动画从发起 CPU 扩散到其他 CPU。

所有其他 CPU 对同一 Index 做 Tag 检查：

```text
CPU01[0H] Tag=0023H，状态=M，监听命中
CPU10[0H] Tag=0124H，不匹配
CPU11[0H] Tag=8A7FH，不匹配
```

### 微步骤 6：监听者状态更新

展示被影响的 CPU：

```text
CPU01[0H]：M/11 -> S/10
原因：别人 ReadMiss，M 持有者必须写回并共享。
```

完成后进入下一条访存请求。

## 针对本例题的四步演示脚本

### 请求 1：CPU00 读 002302H

- 地址拆解：Tag=0023H，Index=0H，Offset=2H。
- CPU00 本地 0H 块 Tag=12FFH，状态=I，因此 ReadMiss。
- CPU00 发 ReadMiss，上总线选项为 1。
- CPU00 更新为 Tag=0023H，状态 S/10。
- CPU01 的 0H 块 Tag=0023H，状态 M/11，监听到 ReadMiss 后写回并降级为 S/10。

### 请求 2：CPU01 写 34EA1FH

- 地址拆解：Tag=34EAH，Index=1H，Offset=FH。
- CPU01 本地 1H 块 Tag=34EAH，状态=S，因此 WriteHit。
- S 状态写命中不能直接写，必须发 Invalidate。
- CPU01 状态 S/10 -> M/11，本地操作选 3。
- CPU11 的 1H 块 Tag=34EAH，状态=S，监听到 Invalidate 后变 I/00。

### 请求 3：CPU10 读 28BE1AH

- 地址拆解：Tag=28BEH，Index=1H，Offset=AH。
- CPU10 本地 1H 块 Tag=28BEH，状态=S，因此 ReadHit。
- 状态 S/10 保持 S/10。
- 本地操作选 4，读本地数据。
- 无总线动作，其他 CPU 不变化。

### 请求 4：CPU11 写 28BE11H

- 地址拆解：Tag=28BEH，Index=1H，Offset=1H。
- CPU11 本地 1H 块状态已为 I，且 Tag=34EAH，不匹配，因此 WriteMiss。
- CPU11 发 WriteMiss，上总线选项为 2。
- CPU11 装入并修改该块，Tag=28BEH，状态 M/11。
- CPU00 和 CPU10 的 1H 块 Tag=28BEH，状态=S，监听到 WriteMiss 后变 I/00。

## 交互模式建议

### 讲解模式

适合初学者：

- 点击单步后自动给出结论。
- 每一步右侧显示规则依据。
- 状态变化自动高亮。
- 填空表实时补全。

### 练习模式

适合做题训练：

每个请求让用户依次选择：

- 本地请求：ReadHit / ReadMiss / WriteHit / WriteMiss。
- 状态转换：例如 `00 10`。
- 本地操作：1 到 5。
- 哪些 CPU 会被监听影响。

系统即时反馈：

```text
正确：S 状态写命中需要发 Invalidate，不能直接写本地。
```

或：

```text
再想想：CPU01[0H] 是 M 状态，它持有最新数据，所以 CPU00 的 ReadMiss 会触发 CPU01 写回并降级。
```

### 回放模式

适合课堂展示：

- 自动播放完整流程。
- 每条请求停顿 2 到 3 秒。
- 总线动作以动画形式显示。
- 最后停在最终状态表和完整答案。

## 推荐实现优先级

第一阶段先做固定例题播放器：

- 固定四条请求。
- 支持重新开始、单步、自动运行。
- 显示 Cache 状态、总线动作、步骤解释和最终答案。

第二阶段加入练习模式：

- 让用户填写每一步本地请求、状态转换、本地操作。
- 给出即时反馈。
- 支持显示/隐藏答案。

第三阶段加入自定义题目：

- 可编辑初始 Cache 表。
- 可编辑访存请求序列。
- 自动计算 Tag / Index / Offset。
- 自动推演最终状态。

第四阶段扩展协议：

- 加入 MESI。
- 加入写更新协议对比。
- 加入目录协议对比视图。

## 视觉风格

可以沿用 Tomasulo 项目的深色教学面板风格，但监听协议更适合强调“广播”和“状态颜色”：

- 背景：深色。
- CPU 卡片：深蓝灰面板。
- 总线：横向亮线。
- I：灰色。
- S：蓝色。
- M：橙红色。
- 当前发起者：青色描边。
- 监听命中者：黄色描边。
- 被作废者：红色闪烁后变灰。

整体不要做成纯表格页面。表格是题目形式，但可视化重点应该是：谁发请求、总线上发生了什么、谁监听后改变。

## 最终页面一句话定位

这个监听协议可视化工具应当像 Tomasulo 项目一样支持逐步播放，但它的核心不是“周期时序”，而是“每次访存请求引发的一次全系统一致性协商”：本地判断、总线广播、其他 Cache 监听、状态收敛。
