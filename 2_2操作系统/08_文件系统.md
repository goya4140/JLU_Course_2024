## 0. 本章定位与考试价值

如果第 6、7 章回答的是“进程看到的地址如何映射到内存/外存”，第 8 章回答的就是：

> 用户看到的是一个个有名字的文件；操作系统内部真正管理的是目录项、FCB/inode、磁盘块、打开文件表和空闲空间。

也就是说，文件系统把“磁盘上的块”包装成“用户可理解、可命名、可保护、可共享、可持久保存的文件”。

本章的考试特点很明显：

1. 概念题多：文件、目录、FCB、inode、文件保护、共享、打开文件表。
2. 结构题多：顺序/链接/索引文件结构，单级/二级/多级目录，UNIX 文件表关系。
3. 流程题多：`open/read/write/close/link/unlink` 到底改哪些表、查哪些结构。
4. 计算题少但容易考：索引结构的最大文件长度，尤其是真题出现过的“两级索引 + 块号大小”。
5. 易错点集中：目录项不是目录本身；删除文件不等于删除目录；`unlink` 不一定立刻释放文件；系统打开文件表和用户打开文件表不是一张表。

## 1. 文件与文件系统：把磁盘块变成“有名字的信息对象”

### 1.1 为什么需要文件系统

直觉解释：

如果没有文件系统，用户面对的不是“`report.docx`”“`main.c`”“`video.mp4`”，而是一堆磁盘块编号：

```text
第 8172 块、第 8173 块、第 9201 块……
```

这显然不可用。用户真正想做的是：

1. 用名字保存一段完整信息。
2. 下次能通过名字找到它。
3. 不关心它分散在哪些磁盘块。
4. 能控制谁可以读、写、执行。
5. 多个进程可以共享或同时访问。
6. 系统崩溃后文件还能尽量保持一致。

所以文件系统的本质是：

> 在外存块之上，提供命名、组织、查找、保护、共享和持久化管理。

课内定义：

- 文件：具有符号名的、逻辑上完整的一组信息项序列。
- 文件系统：文件以及管理文件所需的程序和数据结构的总体。

考试判断：

- 【重点】文件不是单纯的磁盘块集合，而是“有名字、有逻辑完整性、可由系统管理的信息对象”。
- 【易错】文件系统不只是“文件的集合”，还包括管理文件的目录、FCB/inode、打开文件表、空闲空间管理程序等。

### 1.2 UNIX 的三类文件

课件中给出的 UNIX 文件类型主要有三类：

| 类型 | 含义 | 直觉 |
|---|---|---|
| 普通文件 | 用户数据或程序文件 | 文本、程序、数据 |
| 目录文件 | 由目录项组成的文件 | 保存“文件名 -> 文件号”的表 |
| 特殊文件 | 设备文件 | 把设备也当作文件访问 |

【重点】设备作为文件管理的好处：

1. 接口统一：读文件、写设备都可以抽象为类似的读写操作。
2. 保护统一：可以用文件保护机制保护设备。
3. 命名统一：设备也能出现在目录树中。

考试判断：

- 看到“UNIX 把设备看作文件”，不要理解成设备真的存储在普通文件里，而是说操作系统给设备提供文件式接口。
- 特殊文件更偏第 9 章 I/O 管理，但它在第 8 章出现，是为了说明文件系统接口的统一性。

## 2. 文件访问方式：顺序访问与随机访问

### 2.1 顺序访问

直觉解释：

顺序访问像读纸质书：

```text
从第一页读到第二页，再读第三页……
```

你可以从头开始，也可以从某个位置继续，但访问模式通常沿着文件顺序前进。

课内定义：

- 顺序访问：按照文件中信息的排列顺序依次访问。
- 可以从文件开始处访问，也可以从中间某处开始继续访问。

考试判断：

- 【重点】顺序访问适合日志、文本流、批处理数据等。
- 它的优点是实现简单、连续读写效率高。
- 缺点是不适合频繁按编号或关键字直接定位。

### 2.2 随机访问

直觉解释：

随机访问像查字典：

```text
我不想从 A 翻到 Z，我想直接查某个词。
```

课内定义：

- 随机访问：可以根据记录号、关键字或偏移量直接访问文件中的某个位置。

考试判断：

- 【重点】随机访问通常要求文件物理组织或索引结构支持直接定位。
- 顺序结构天然适合顺序访问；索引结构更适合随机访问。
- 链接结构虽然增长方便，但随机访问较差。

## 3. 文件组织：逻辑上怎么看，物理上怎么放

文件组织要分两层：

1. 逻辑组织：用户/程序看到文件内部是什么结构。
2. 物理组织：操作系统把文件内容放到磁盘块上的方式。

### 3.1 逻辑组织：记录文件与流式文件

#### 3.1.1 记录文件

直觉解释：

记录文件像学生信息表：

```text
学号 | 姓名 | 成绩
001  | 张三 | 90
002  | 李四 | 85
```

每一行是一条记录。

课内定义：

- 记录文件：由若干逻辑记录组成的文件。
- 记录可以是定长记录，也可以是变长记录。

考试判断：

- 定长记录：第 k 条记录位置容易计算。
- 变长记录：更灵活，但定位和管理更复杂。

#### 3.1.2 流式文件

直觉解释：

流式文件像一条连续字节流：

```text
byte0 byte1 byte2 byte3 ...
```

程序按字节偏移读写，而不是按“第几条记录”读写。

课内定义：

- 流式文件：文件被看成字节序列。
- UNIX、Windows 中大量普通文件都可看成流式文件。

考试判断：

- 【重点】流式文件不强调记录边界。
- 文件系统只负责字节序列；具体格式由应用程序解释。

### 3.2 物理组织总览

物理组织回答：

> 文件的逻辑记录或字节序列如何映射到磁盘块？

课内主要讲四类：

| 物理组织 | 核心思想 | 优点 | 缺点 |
|---|---|---|---|
| 顺序结构 | 文件占用连续磁盘块 | 顺序访问快，节省指针空间 | 文件增长困难，可能产生外部碎片 |
| 链接结构 | 文件块分散，用指针串起来 | 文件增长方便，无需连续空间 | 随机访问慢，指针有开销 |
| 索引结构 | 用索引块保存数据块号 | 支持随机访问，增长较方便 | 索引块占空间 |
| Hash 结构 | 根据关键字映射位置 | 查找快 | 冲突处理复杂 |

## 4. 顺序结构、链接结构、索引结构、Hash 结构

### 4.1 顺序结构

直觉解释：

顺序结构像把一本书的所有页连续放在一起：

```text
文件 A: [100][101][102][103][104]
```

只要知道起始块号和长度，就能一路读下去。

课内定义：

- 顺序结构：文件占用一组连续的磁盘块。

优点：

1. 顺序访问速度快。
2. 随机访问也容易：第 k 块 = 起始块 + k。
3. 不需要为每个块保存指针，空间开销小。

缺点：

1. 文件长度变化困难。
2. 创建时需要预估文件大小。
3. 外存中需要找到足够大的连续空闲区。

考试判断：

- 【重点】看到“连续磁盘块”“顺序读写快”“扩展困难”，对应顺序结构。
- 【易错】顺序结构不是只能顺序访问，它也可以随机定位；只是文件增长困难。

### 4.2 链接结构

直觉解释：

链接结构像把书页分散放在不同位置，每页写着“下一页在哪里”：

```text
文件 A: [100] -> [508] -> [17] -> [309]
```

课内定义：

- 链接结构：文件的数据块不要求连续，每个块中保存指向下一块的指针。

优点：

1. 文件容易增长。
2. 不需要连续空间。
3. 外存利用率较高。

缺点：

1. 随机访问慢：访问第 k 块必须从头沿链走。
2. 指针占用部分块空间。
3. 指针损坏可能导致后续块丢失。

考试判断：

- 【重点】看到“文件增长方便”“随机访问慢”，对应链接结构。
- 【易错】链接结构适合顺序访问，不适合直接访问第 k 块。

### 4.3 索引结构

直觉解释：

索引结构像书的目录页：

```text
索引块: [100, 508, 17, 309, ...]
数据块: 100, 508, 17, 309
```

文件数据块可以分散，但有一个索引块统一记录它们的块号。

课内定义：

- 索引结构：为文件建立索引块，索引块中保存文件各数据块的块号。

优点：

1. 支持随机访问。
2. 文件不要求连续存放。
3. 文件增长比顺序结构灵活。

缺点：

1. 索引块本身占用外存空间。
2. 小文件也可能需要索引块，空间利用率可能下降。
3. 大文件需要多级索引或混合索引。

考试判断：

- 【必会】看到“随机访问快 + 非连续分配 + 索引块开销”，对应索引结构。
- 索引结构是本章计算题最常见的来源。

### 4.4 Hash 结构

直觉解释：

Hash 结构像用关键字直接算位置：

```text
位置 = H(文件名或记录关键字)
```

如果两个关键字算到同一个位置，就发生冲突，需要冲突处理。

课内定义：

- Hash 结构：通过 Hash 函数把关键字映射到存储位置。
- 冲突可用顺序探查等方法处理。

考试判断：

- 【重点】课件提到 Hash 结构，习题课也强调目录查找中的 Hash 查找。
- 在文件系统中，Hash 常用于目录项查找，而不是所有文件都必须按 Hash 存储。

## 5. UNIX 混合索引结构与文件长度计算

### 5.1 为什么需要多级索引

单级索引的问题：

如果一个索引块只能放 256 个块号，那么一个文件最多只能直接指向 256 个数据块。小文件够用，大文件不够。

解决方法：

1. 对小文件使用直接地址，少走索引层次。
2. 对中等文件使用一级间接索引。
3. 对大文件使用二级、三级间接索引。

这就是 UNIX 文件系统中常见的混合索引思想。

### 5.2 UNIX 课件中的结构

课件给出 UNIX 文件物理结构的思路：

```text
直接地址 + 一级间接 + 二级间接 + 三级间接
```

在课件给出的例子中，如果一个索引块能保存 256 个块号，则最大可寻址数据块数可以写成：

```text
10 + 256 + 256^2 + 256^3
```

含义：

| 部分 | 可指向的数据块数 |
|---|---:|
| 直接地址 | 10 |
| 一级间接 | 256 |
| 二级间接 | 256^2 |
| 三级间接 | 256^3 |

考试判断：

- 【重点】这是课件中的 UNIX 混合索引结构，不要和真题中的“纯两级索引”混淆。
- 混合索引的好处是：小文件访问快，大文件也能支持。

### 5.3 【必会】真题型：两级索引最大文件长度

真题形式：

> 某系统采用两级索引文件结构，磁盘块大小 1KB，每个磁盘块号占 4B，则该系统的最大单个文件长度是多少？

解题模板：

1. 先算一个索引块能放多少个块号：

```text
索引项数 = 磁盘块大小 / 块号大小
        = 1KB / 4B
        = 1024B / 4B
        = 256
```

2. 两级索引能指向的数据块数：

```text
256 * 256 = 65536 块
```

3. 每个数据块大小 1KB，所以最大文件长度：

```text
65536 * 1KB = 65536KB = 64MB
```

答案：

```text
64MB
```

【题型】索引最大长度计算通式：

```text
每个索引块可放指针数 p = 块大小 / 指针大小
k 级索引可寻址数据块数 = p^k
最大文件长度 = p^k * 块大小
```

【易错】三类常见错误：

1. 忘记把 1KB 换成 1024B。
2. 把“两级索引”误算成 `2 * 256`。
3. 把索引块也算进文件数据长度。

补充判断：

- 如果题目说“UNIX 混合索引”，要把直接、一级、二级、三级分别加起来。
- 如果题目只说“两级索引文件结构”，通常按纯二级索引计算，即 `p^2 * block_size`。

## 6. 文件目录：文件名如何找到文件实体

### 6.1 为什么需要目录

直觉解释：

磁盘上真正管理文件时，系统更关心文件号、inode、FCB、数据块地址；但用户只记得文件名。

目录的作用就是建立映射：

```text
文件名 -> 文件号/FCB/inode -> 文件数据块
```

没有目录，文件名就找不到文件实体。

### 6.2 FCB：文件控制块

课内定义：

- FCB，即文件控制块，是文件存在的标志。
- FCB 中保存操作系统管理文件所需的全部信息。

常见字段：

| 字段 | 含义 |
|---|---|
| 文件名 | 用户看到的名字 |
| 文件号 | 系统内部标识 |
| 文件主 | 文件所有者 |
| 文件类型 | 普通/目录/特殊等 |
| 文件属性 | 只读、隐藏等 |
| 共享信息 | 是否共享、链接数等 |
| 文件长度 | 文件大小 |
| 文件地址 | 数据块位置或索引位置 |
| 时间信息 | 创建、修改、访问时间 |
| 保护信息 | 权限、口令等 |

考试判断：

- 【必会】FCB 是文件存在的标志。
- 【易错】FCB 不是文件数据本身，而是文件的管理信息。

### 6.3 目录项、目录文件、文件目录

这组概念很容易混。

| 概念 | 含义 | 直觉 |
|---|---|---|
| 目录项 | 目录文件中的一项 | 一条“文件名 -> 文件号/FCB”的记录 |
| 目录文件 | 由目录项组成的文件 | 一个特殊文件 |
| 文件目录 | 目录结构整体 | 文件命名与查找系统 |
| FCB | 文件管理信息块 | 文件的“档案袋” |

【必会】课件表述：

- 目录项是目录文件中的一个项目，其内容为 FCB。
- UNIX 改进后，目录项常只保存“文件名 + 文件号”，完整 FCB 主部放在 inode 区。

考试判断：

- 看到“目录项”，想到“一个文件在目录中的登记项”。
- 看到“目录文件”，想到“目录本身也是一种文件”。
- 看到“FCB/inode”，想到“文件管理信息，不是文件内容”。

## 7. 目录结构：单级、二级、多级

### 7.1 单级目录

直觉解释：

所有文件放在一张大表中：

```text
/:
  a.txt
  b.c
  data.bin
```

优点：

- 结构简单。

缺点：

1. 文件不能重名。
2. 查找效率低。
3. 不利于分组。
4. 不利于多用户保护。

考试判断：

- 【重点】看到“所有用户共享一个目录”“文件名不能重复”，对应单级目录。

### 7.2 二级目录

直觉解释：

每个用户有自己的目录：

```text
/userA/a.txt
/userB/a.txt
```

这样不同用户可以使用相同文件名。

优点：

1. 不同用户可有相同文件名。
2. 查找范围缩小，效率提高。
3. 有一定用户隔离能力。

缺点：

- 仍然缺乏灵活分组能力。

考试判断：

- 【重点】二级目录解决“不同用户重名”问题，但不充分解决“层次化分组”问题。

### 7.3 多级目录

直觉解释：

多级目录就是树形目录：

```text
/
├── usr
│   └── wang
│       └── d1
│           └── f1
└── tmp
```

优点：

1. 支持文件分类组织。
2. 支持路径名。
3. 支持当前目录。
4. 查找和保护更灵活。

考试判断：

- 【重点】UNIX 采用多级树形目录。
- 多级目录引入绝对路径和相对路径。

## 8. 路径查找与目录查找算法

### 8.1 从根目录查找与从当前目录查找

习题课明确回顾：

- 查找路径：根目录、当前目录。

直觉解释：

绝对路径从根目录开始：

```text
/usr/users/wang/d1/f1
```

相对路径从当前目录开始：

```text
d1/f1
```

考试判断：

- 【重点】绝对路径不依赖当前工作目录。
- 相对路径依赖当前工作目录。
- `chdir` 改变当前工作目录后，相对路径解释会改变。

### 8.2 目录查找算法

习题课明确列出：

- 顺序查找
- Hash 查找
- 对分查找

#### 顺序查找

从目录项开头逐项比较文件名。

优点：

- 简单。

缺点：

- 目录很大时效率低。

#### Hash 查找

根据文件名计算 Hash 值，快速定位可能位置。

优点：

- 平均查找快。

缺点：

- 需要处理冲突。

#### 对分查找

目录项按文件名排序后，可以二分查找。

优点：

- 查找效率较高。

前提：

- 目录项必须有序。

考试判断：

- 【重点】Hash 查找适合大目录快速定位，但要考虑冲突。
- 【易错】对分查找不是任何目录都能直接用，必须目录项有序。

## 9. 目录改进：次部、主部与 inode

### 9.1 为什么目录需要改进

朴素做法：

目录项中直接保存完整 FCB。

问题：

1. FCB 很大，目录文件也很大。
2. 查找文件名时，其实只需要比较文件名，却不得不读入大量 FCB 信息。
3. 目录顺序查找变慢。
4. 文件共享和链接实现不方便。

改进思想：

> 把“查找文件名必需的信息”和“文件管理详细信息”分开。

### 9.2 次部与主部

课件中的目录改进：

| 部分 | 内容 | 放在哪里 |
|---|---|---|
| FCB 次部 | 文件名、文件号 | 目录文件中 |
| FCB 主部 | 文件其他管理信息、链接计数等 | 外存 inode 区 |

UNIX 中目录项可以很小，例如只保存：

```text
(文件名, 文件号)
```

完整 inode 放在 inode 区。

优点：

1. 目录文件变小。
2. 顺序查找目录更快。
3. 多个目录项可以指向同一个 inode，便于实现链接共享。

考试判断：

- 【必会】目录项变小是为了加快目录查找。
- 【必会】主部保存链接计数，支持文件共享和 `unlink`。
- 【易错】UNIX 目录项不是完整 FCB，而是“文件名 + inode 号”。

### 9.3 UNIX 卷结构

课件中的 UNIX 卷布局：

```text
引导块 | 专用块/超级块 | inode 区 | 文件存储区
```

| 区域 | 作用 |
|---|---|
| 引导块 | 启动相关 |
| 超级块 | 文件系统整体管理信息 |
| inode 区 | 保存 FCB 主部，即 inode |
| 文件存储区 | 保存目录文件和普通文件数据 |

考试判断：

- 【重点】inode 在 inode 区；文件数据在文件存储区。
- 目录文件本身也是文件，它的数据块中存放目录项。

## 10. 文件共享：多个名字如何指向同一文件

### 10.1 为什么需要共享

文件共享有两个主要作用：

1. 节省外存空间：多个用户共享同一份文件，不必复制多份。
2. 支持进程间通信：例如 UNIX 管道可看成一种特殊文件式通信机制。

课件给出的共享方式：

1. 公共目录。
2. 共享说明。
3. 链接。

### 10.2 链接：多个目录项指向同一个 inode

直觉解释：

硬链接像给同一个文件实体起了多个名字：

```text
/usr/users/wang/d1/f1  ----\
                            -> inode 15 -> 数据块
/usr/users/li/f2       ----/
```

课件示例：

```c
link("/usr/users/wang/d1/f1", "/usr/users/li/f2");
```

执行后：

- 新目录中增加一个目录项。
- 新目录项的文件号指向原文件的 inode。
- inode 的链接计数加 1。

### 10.3 `unlink`：删除名字，不一定删除文件实体

课件示例：

```c
unlink("/usr/users/wang/d1/f1");
```

直觉解释：

`unlink` 的直接含义不是“立刻摧毁文件数据”，而是：

> 删除某个目录项，也就是删除文件的一个名字。

流程：

1. 找到路径名对应的目录项。
2. 找到对应 inode。
3. 删除目录项。
4. inode 链接计数减 1。
5. 如果链接计数变为 0，才释放文件占用的数据块和 inode。

【必会】考试判断：

- 一个文件可以有多个名字。
- `unlink` 删除的是目录项。
- 只有链接计数归零时，文件实体才会被撤销。
- 如果仍有其他目录项指向该 inode，文件数据不能释放。

【易错】如果文件已经被进程打开，实际系统还可能等打开引用也释放后再回收；课内重点是链接计数和目录项关系。

## 11. 文件删除流程与真题判断

### 11.1 删除文件时系统会做什么

删除文件常见流程：

1. 根据路径查找目录。
2. 找到与该文件关联的目录项。
3. 删除目录项。
4. 修改 FCB/inode 中的链接计数。
5. 若链接计数为 0，释放文件数据块。
6. 若无其他引用，释放 FCB/inode。
7. 必要时清理相关缓存或打开表项。

### 11.2 【必会】真题型：删除文件不可能做什么

真题形式大意：

> 用户执行删除文件的过程中，操作系统不可能执行的操作是：
>
> A. 删除此文件所在的目录  
> B. 删除与此文件关联的目录项  
> C. 删除与此文件对应的文件控制块  
> D. 释放与此文件关联的内存缓冲区

合理判断：

- B：可能。删除文件最核心动作之一就是删除目录项。
- C：可能。当没有其他链接或引用时，可以释放 FCB/inode。
- D：可能。系统可能清理与文件相关的缓冲或缓存，但不是每次必做。
- A：不可能。删除一个文件不等于删除它所在的目录。

答案：

```text
A
```

【易错】题干问的是“不可能执行”，不是“不一定执行”。  
如果问“不一定执行”，那释放 FCB、释放缓冲区都可能依赖具体条件；但删除文件所在目录显然不是删除该文件的正常操作。

## 12. 文件保护、保密与安全

课件把这三个概念区分开：

| 概念 | 关注点 | 直觉 |
|---|---|---|
| 文件保护 | 防止未授权访问 | 谁能读写执行 |
| 文件保密 | 防止信息泄露 | 即使拿到也看不懂 |
| 文件安全 | 防止文件被破坏 | 防自然/人为损坏 |

### 12.1 访问类型

课件列出的访问类型包括：

| 类型 | 含义 |
|---|---|
| Read | 读 |
| Write | 写 |
| Execute | 执行 |
| Append | 追加 |
| Delete | 删除 |
| List | 列目录 |

考试判断：

- 【重点】访问控制不是只有读/写/执行，也可能包含追加、删除、列目录等。

### 12.2 访问控制矩阵

直觉解释：

访问控制矩阵像一张用户-文件权限表：

```text
          file1     file2
userA     R W       R
userB     R         R W E
```

优点：

- 表达精确。

缺点：

- 用户和文件很多时，矩阵巨大，存储开销大，管理复杂。

考试判断：

- 【重点】访问控制矩阵精确但不经济。
- 实际系统常用访问控制表、能力表或简化权限位。

### 12.3 UNIX 权限：`i_mode`

UNIX 使用 owner/group/others 三类用户，每类有读、写、执行权限：

```text
owner:  r w x
group:  r w x
others: r w x
```

判断顺序：

1. 如果当前用户是文件主，按 owner 权限判断。
2. 否则如果当前用户属于文件组，按 group 权限判断。
3. 否则按 others 权限判断。

相关系统调用：

- `creat(filename, mode)`：创建文件时设置权限。
- `chmod`：修改权限。

考试判断：

- 【重点】UNIX 权限是课件重点，但期末更可能考概念判断而非复杂八进制权限计算。
- 【易错】权限属于 inode/FCB 主部中的管理信息，而不是目录项中的文件名信息。

### 12.4 口令、加密与备份

口令保护：

- 简单。
- 但口令可能泄露，系统管理员也可能看到相关信息。

加密：

- 保存时加密。
- 读取时用密钥解密。
- 安全性较强，但有时间开销。

备份：

- 防止文件因硬件故障、误操作、灾害等被破坏。
- 包括完全转储和增量转储。

考试判断：

- 【重点】保密更强调防泄露，加密是保密手段。
- 安全更强调防破坏，备份是安全手段。

## 13. 文件打开：为什么需要打开文件表

### 13.1 直接每次按路径访问为什么慢

如果每次 `read` 都从路径名开始查找：

```text
/usr/users/wang/d1/f1
```

系统每读一次都要：

1. 从根目录开始查。
2. 查 `usr`。
3. 查 `users`。
4. 查 `wang`。
5. 查 `d1`。
6. 找到 `f1`。
7. 读 FCB/inode。
8. 再定位数据块。

这太慢。

所以文件访问一般先 `open`：

```text
open: 路径名 -> 打开文件表项 -> 文件描述符
read/write: 文件描述符 -> 打开文件表项 -> inode -> 数据块
```

### 13.2 系统打开文件表与用户打开文件表

课件给出两类表：

| 表 | 范围 | 主要内容 |
|---|---|---|
| 系统打开文件表 | 全系统共享 | FCB 主部、文件号、共享计数、修改标志 |
| 用户打开文件表 | 每个进程一张 | 文件描述符、打开方式、读写指针、系统表入口 |

直觉解释：

- 系统打开文件表：系统层面“这个文件已经被打开了几次”。
- 用户打开文件表：某个进程层面“我用哪个 fd 访问这个文件，当前读写到哪里”。

【必会】考试判断：

- 用户打开文件表是每进程的。
- 系统打开文件表是全系统的。
- 文件描述符 fd 是用户打开文件表中的下标或入口标识。
- 多个进程打开同一文件，可能共享系统打开文件表项，也可能各有文件表项，取决于实现和打开方式；课件重点是存在“共享计数”。

### 13.3 为什么读写指针通常在用户打开文件表中

两个进程同时打开同一个文件：

```text
进程 A 读到第 100 字节
进程 B 读到第 500 字节
```

它们的读写位置可以不同。  
所以读写指针常放在和进程打开实例相关的表项中。

考试判断：

- 【重点】读写指针描述的是“本次打开的当前位置”，不是文件永久属性。
- 文件长度、数据块地址属于 FCB/inode；读写指针属于打开文件状态。

## 14. UNIX 的 `u_ofile`、`file`、`inode`

习题课明确要求掌握 UNIX 文件实现中的三张表：

```text
u_ofile  ->  file  ->  inode
```

### 14.1 三张表的含义

| 结构 | 范围 | 作用 |
|---|---|---|
| `u_ofile` | 每个进程 | 文件描述符表 |
| `file` | 系统范围 | 打开文件表，保存打开方式、引用计数、偏移量等 |
| `inode` | 系统范围/内存 inode 表 | 文件 FCB 主部，保存文件属性和地址 |

关系：

```text
fd
 |
 v
u_ofile[fd]  --->  file 表项  --->  inode 表项  --->  磁盘数据块
```

### 14.2 `file` 表项

课件中的 `file` 结构包含：

| 字段 | 含义 |
|---|---|
| `f_flag` | 打开方式/状态标志 |
| `f_count` | 引用计数 |
| `f_inode` | 指向 inode |
| `f_offset` | 当前读写偏移 |

考试判断：

- 【必会】`file` 表项描述“打开实例”。
- `f_count` 表示有多少引用指向该 `file` 表项。
- `f_inode` 把打开文件表和文件实体连接起来。

### 14.3 inode 表项

课件中的 inode 字段包括：

| 字段 | 含义 |
|---|---|
| `i_count` | 内存 inode 引用计数 |
| `i_dev` | 设备号 |
| `i_number` | inode 编号 |
| `i_mode` | 类型与权限 |
| `i_nlink` | 链接计数 |
| `i_uid` | 文件主 |
| `i_gid` | 文件组 |
| `i_size` | 文件大小 |
| `i_addr` | 文件块地址/索引地址 |
| 时间字段 | 访问、修改等时间 |

【必会】两个计数不要混：

| 计数 | 含义 |
|---|---|
| `f_count` | 有多少引用指向这个打开文件表项 |
| `i_count` | 内存 inode 被引用次数 |
| `i_nlink` | 磁盘目录项硬链接数 |

【易错】`i_nlink` 是“有多少目录项指向该 inode”，不是“有多少进程打开该文件”。

## 15. 文件系统调用流程

### 15.1 `creat`

作用：

创建一个新文件，并通常以写方式打开。

课件流程：

1. 分配 FCB 主部/inode。
2. 初始化文件大小、权限、文件主、文件组、链接计数等。
3. 在最终目录中填写目录项，即文件名和文件号。
4. 建立打开文件表项。
5. 返回文件描述符。

考试判断：

- 【重点】创建文件既要分配 inode，也要在目录中增加目录项。
- 新文件的链接计数通常初始化为 1。

### 15.2 `open`

作用：

把路径名转换成后续读写可用的文件描述符。

课件流程：

1. 根据路径名查找目录。
2. 找到 FCB 主部/inode。
3. 检查访问权限。
4. 如果系统打开文件表中已有该文件，增加共享计数；否则建立系统表项。
5. 在进程用户打开文件表中分配一项。
6. 返回 fd。

【必会】考试判断：

- `open` 之后，后续 `read/write` 不再每次使用路径名，而是使用 fd。
- `open` 要检查权限。
- `open` 会影响打开文件表。

### 15.3 `close`

作用：

关闭进程中的一个文件描述符。

流程：

1. 根据 fd 找到用户打开文件表项。
2. 找到系统打开文件表项。
3. 减少共享计数/引用计数。
4. 如果计数变为 0，必要时把修改过的 FCB/inode 写回外存。
5. 清除用户打开文件表项。

考试判断：

- 【重点】`close` 不是简单丢掉 fd，还可能触发写回。
- 如果还有其他打开引用，不能把共享的系统表项直接释放。

### 15.4 `seek`

作用：

改变文件当前读写位置。

流程：

1. 根据 fd 找到用户打开文件表项。
2. 检查偏移是否合法。
3. 设置读写指针。

考试判断：

- 【重点】`seek` 改变的是打开文件状态中的读写位置，不直接读写磁盘数据。

### 15.5 `read`

作用：

从文件读数据到用户缓冲区。

简化流程：

1. fd -> 用户打开文件表。
2. 用户表 -> 系统打开文件表。
3. 系统表 -> FCB/inode。
4. 根据文件偏移和地址信息计算物理块位置。
5. 发起 I/O，把数据读入系统缓冲区。
6. 拷贝到用户缓冲区。
7. 更新读写偏移。

考试判断：

- 【重点】`read` 使用 fd，不使用路径名。
- 逻辑位置到物理块位置的转换依赖文件物理组织。

### 15.6 `write`

作用：

把用户缓冲区数据写入文件。

简化流程：

1. fd -> 打开文件表 -> inode。
2. 检查写权限。
3. 根据偏移找到或分配数据块。
4. 写入系统缓冲区或设备 I/O 队列。
5. 更新文件大小、修改标志、时间等。

考试判断：

- 【重点】写文件可能导致分配新磁盘块。
- 写文件可能先写缓冲区，之后再真正写磁盘，这和延迟写有关。

### 15.7 `link`

作用：

给已有文件增加一个新名字。

流程：

1. 查找旧路径，找到 inode。
2. 查找新路径所在目录。
3. 在新目录中增加目录项。
4. inode 的链接计数加 1。

考试判断：

- 【必会】`link` 不复制文件数据。
- 【必会】`link` 增加目录项和链接计数。

### 15.8 `unlink`

作用：

删除一个文件名。

流程：

1. 查找路径对应目录项。
2. 删除目录项。
3. inode 链接计数减 1。
4. 如果链接计数为 0，释放文件块和 inode。

考试判断：

- 【必会】`unlink` 不一定立刻释放数据块。
- 【易错】删除文件名不等于删除所在目录。

## 16. 外存空闲空间管理

文件系统不仅要记录“哪些块属于哪个文件”，还要记录：

> 哪些磁盘块现在是空闲的，可以分配给新文件？

课件讲了几种方法：

1. 空闲块表。
2. 空闲块链。
3. 位示图。
4. UNIX 成组链接。

### 16.1 空闲块表

思想：

把空闲区的起始块号和长度记录在表中。

适合：

- 连续分配。

优点：

- 能快速找到一段连续空闲区。

缺点：

- 表需要维护。
- 空闲区分裂、合并会带来管理开销。

### 16.2 空闲块链

思想：

把所有空闲块用指针链起来。

优点：

- 分配一个块简单。

缺点：

- 难以快速找到连续空闲块。
- 链很长时管理效率低。

### 16.3 位示图

思想：

用一位表示一个磁盘块是否空闲：

```text
0 表示空闲，1 表示占用
```

或者反过来，具体取决于系统定义。

优点：

1. 表示紧凑。
2. 便于寻找连续空闲块。
3. 适合用位运算加速。

缺点：

- 位示图本身要占用空间。
- 大磁盘位图也可能较大，需要缓存。

考试判断：

- 【重点】位示图是非常常见的空闲空间管理方法。
- 判断时先看题目规定 0/1 含义，不要默认。

## 17. UNIX 成组链接

习题课明确强调：

> UNIX 外存管理：成组链接。

### 17.1 为什么需要成组链接

普通空闲块链的问题：

如果每次分配一个空闲块都要读链表中的下一个块，I/O 次数多。

成组链接的思想：

> 把空闲块分组，每组记录一批空闲块号，并把组与组链接起来。

课件中每组可记录 100 个空闲块号，超级块中缓存当前一组空闲块号。

### 17.2 关键数据结构

超级块中有：

| 字段 | 含义 |
|---|---|
| `s_nfree` | 当前缓存的空闲块号个数 |
| `s_free[100]` | 当前缓存的一组空闲块号 |

直觉：

```text
超级块缓存一组空闲块号
如果这组快用完，就从下一组的链块中读入新的 100 个块号
```

### 17.3 分配空闲块

课件规则：

如果：

```text
s_nfree > 1
```

则：

```text
取出 s_free[--s_nfree]
```

如果：

```text
s_nfree == 1
```

则：

1. `s_free[0]` 指向的块不是普通空闲数据块，而是下一组空闲块号的登记块。
2. 读入该块中的空闲块号信息，补充到超级块。
3. 再分配相应块。

考试判断：

- 【必会】`s_nfree == 1` 是成组链接分配中的特殊情况。
- 最后一个块号通常用于找到下一组，而不只是简单分配完。

### 17.4 回收空闲块

课件规则：

如果：

```text
s_nfree < 100
```

则：

```text
s_free[s_nfree++] = 释放块号
```

如果：

```text
s_nfree == 100
```

则：

1. 把当前超级块中的 `s_nfree` 和 `s_free` 写入刚释放的块。
2. 让刚释放的块成为新的组链块。
3. `s_free[0] = 刚释放块号`。
4. `s_nfree = 1`。

考试判断：

- 【必会】成组链接把“链表”和“数组缓存”结合起来。
- 【易错】回收时如果当前组满了，不是丢弃块号，而是把原缓存写入新释放块，让它成为新组。

### 17.5 空闲 inode 管理

课件还提到：

- 超级块中缓存一部分空闲 inode 号。
- 分配 inode 时先从缓存取。
- 如果缓存为空，则扫描 inode 区，找出一批 `i_nlink = 0` 的 inode。
- 释放 inode 时，如果缓存有空位就放回；没有空位可不立即放入缓存。

考试判断：

- 【重点】inode 是否空闲可看链接计数等状态。
- 空闲块和空闲 inode 都属于文件系统资源管理。

## 18. 管道：文件系统中的进程通信接口

管道虽然和进程通信有关，但课件在 UNIX 文件系统接口中讲到 `pipe`。

### 18.1 管道的直觉

管道像一个内核维护的临时文件/缓冲区：

```text
进程 A 写入 ---> pipe ---> 进程 B 读出
```

创建管道后通常返回两个文件描述符：

```text
fd[0]：读端
fd[1]：写端
```

### 18.2 课件中的管道实现要点

课件描述：

1. 分配一个 inode。
2. `i_count = 2`，因为读端和写端都引用它。
3. 分配两个 file 表项：一个读，一个写。
4. 分配两个 `u_ofile` 表项。
5. 返回两个 fd。

同步规则：

- 读指针追上写指针，读者等待。
- 写指针追上读指针，写者等待。
- 一端关闭时，另一端要被唤醒或收到相应信号。

考试判断：

- 【重点】管道体现“把通信也做成文件接口”的思想。
- 详细同步机制属于第 4 章通信/同步，本章只把它作为文件系统接口理解。

## 19. 日志结构文件系统

习题课明确列出：

- 日志结构文件系统。

### 19.1 为什么提出日志结构文件系统

背景：

1. CPU 和内存速度提升很快。
2. 磁盘随机写仍然慢。
3. 小文件、小块写入会造成大量寻道和旋转延迟。
4. 延迟写虽然提高性能，但崩溃时可能造成一致性问题。

日志结构文件系统的直觉：

> 不在磁盘各处随机修改，而是把修改攒起来，像写日志一样顺序追加到磁盘。

### 19.2 基本思想

LSFS 把磁盘看成一个日志：

1. 新数据先进入内存缓冲。
2. 积累成一个 segment。
3. 顺序写入磁盘尾部。
4. segment 中可以包含数据块、inode、目录块等。
5. 通过 inode map 找到最新 inode 位置。

### 19.3 清理器

因为不断追加，旧版本数据会变成无效块。  
清理器负责：

1. 扫描 segment。
2. 找出仍然有效的块。
3. 把有效块复制到新 segment。
4. 释放旧 segment。

考试判断：

- 【重点】日志结构文件系统的关键词是“顺序写”“段”“清理器”“inode map”。
- 它不是简单的操作日志，而是把文件系统更新本身组织成日志式追加。

## 20. 内存映射文件

习题课明确列出：

- 内存映射文件。

### 20.1 为什么需要内存映射文件

普通文件访问：

```text
open -> read/write -> 系统调用 -> 缓冲区拷贝 -> 用户空间
```

问题：

1. 每次读写需要系统调用。
2. 可能发生数据在内核缓冲区和用户缓冲区之间的拷贝。
3. 频繁访问大文件时开销较大。

内存映射文件的直觉：

> 把文件的一段映射到进程虚拟地址空间，之后像访问内存一样访问文件。

### 20.2 基本步骤

课件给出的过程：

1. `open`：打开文件。
2. `map`：建立文件到虚拟地址空间的映射。
3. 使用：通过普通内存读写指令访问。
4. `unmap`：解除映射。
5. `close`：关闭文件。

考试判断：

- 【重点】内存映射文件把文件 I/O 转化为内存访问形式。
- 真正的数据调入、写回仍由操作系统和虚拟存储机制配合完成。
- 它和第 7 章虚拟存储有关，但在本章作为文件接口扩展理解。

## 21. VFS、Ext2fs 与文件系统效率

### 21.1 VFS：统一不同文件系统接口

课件讲 Linux 文件系统时提到 VFS。

直觉解释：

Linux 可以同时支持多种文件系统：

```text
Ext2/Ext3/Ext4, FAT, ISO9660, NFS ...
```

如果每种文件系统都让应用直接面对不同接口，编程会非常痛苦。

VFS 的作用：

> 在具体文件系统之上提供统一抽象，让上层用统一接口访问不同文件系统。

考试判断：

- 【重点】VFS 是抽象层，不是某一种磁盘格式。

### 21.2 Ext2fs 的基本思想

课件提到 Ext2fs 块组结构：

```text
引导块
超级块
组描述符
块位图
inode 位图
inode 表
数据块
```

核心思想：

- 把磁盘分成块组。
- 每个块组管理自己的一部分 inode 和数据块。
- 用位图管理空闲块和空闲 inode。
- 尽量把相关数据放近，减少磁盘臂移动。

考试判断：

- 【重点】Ext2fs 常用位图管理空闲块。
- 尽量连续或邻近分配可以提升磁盘访问效率。

### 21.3 文件系统效率手段

课件提到的效率改进包括：

| 方法 | 作用 |
|---|---|
| 缓存 | 减少实际磁盘访问 |
| 预读 | 顺序访问时提前读后续块 |
| 延迟写 | 合并写请求，减少小写入 |
| 连续/邻近分配 | 减少寻道 |
| 扫描顺序优化 | 减少磁臂移动 |

考试边界：

- 缓冲/缓存和磁盘调度细节会在第 9 章展开。
- 本章只需理解它们是文件系统提高效率的手段。

## 22. 课内题型归纳与逐题解析

### 22.1 【题型】索引文件最大长度

题目模型：

> 块大小 B，每个块号占 P 字节，采用 k 级索引，问最大文件长度。

解题步骤：

```text
1. 每个索引块可放指针数 n = B / P
2. k 级索引可指向数据块数 = n^k
3. 最大文件长度 = n^k * B
```

例题：

```text
B = 1KB = 1024B
P = 4B
k = 2
n = 1024 / 4 = 256
最大长度 = 256^2 * 1KB = 64MB
```

应试口诀：

```text
先算一块放几个号，再看几级乘几次，最后乘数据块大小。
```

### 22.2 【题型】物理组织优缺点判断

常见问法：

> 哪种文件结构随机访问快？哪种文件结构扩展方便？哪种结构需要连续空间？

快速判断：

| 问法 | 答案倾向 |
|---|---|
| 连续空间、顺序访问最快 | 顺序结构 |
| 文件增长方便、不要求连续 | 链接结构 |
| 随机访问好、需要索引块 | 索引结构 |
| 按关键字快速定位 | Hash 结构 |

### 22.3 【题型】目录与 FCB/inode

常见问法：

> 目录项保存什么？FCB 是什么？UNIX 目录改进为什么能提高查找效率？

答题模板：

```text
FCB 保存文件管理信息，是文件存在的标志。
目录文件由目录项组成，用于把文件名映射到文件号/FCB/inode。
UNIX 将 FCB 分为次部和主部：次部即文件名和文件号放在目录中，
主部即 inode 放在 inode 区。这样目录项变小，顺序查找更快，
也便于多个目录项链接到同一个 inode。
```

### 22.4 【题型】打开文件表关系

常见问法：

> 用户打开文件表和系统打开文件表有什么区别？UNIX 的 `u_ofile/file/inode` 是什么关系？

答题模板：

```text
用户打开文件表属于每个进程，文件描述符 fd 指向其中的表项；
系统打开文件表属于全系统，保存打开文件的共享信息、打开方式、偏移量或引用计数；
FCB/inode 保存文件实体的管理信息，如权限、长度、地址、链接计数。
UNIX 中 fd -> u_ofile -> file -> inode。
```

### 22.5 【题型】`link/unlink`

常见问法：

> `link` 和 `unlink` 对目录项、inode、链接计数有什么影响？

答题模板：

```text
link 给已有文件增加一个新目录项，新目录项指向同一 inode，
并使 inode 的链接计数加 1，不复制文件数据。

unlink 删除一个目录项，使 inode 的链接计数减 1。
只有当链接计数减为 0 时，才释放文件数据块和 inode。
```

### 22.6 【题型】删除文件流程

常见问法：

> 删除文件时，操作系统不可能做什么？

快速判断：

```text
会删除目录项；
可能释放 FCB/inode；
可能释放数据块；
可能清理相关缓冲；
不会删除该文件所在的目录。
```

### 22.7 【题型】成组链接

常见问法：

> UNIX 如何管理空闲磁盘块？`s_nfree` 和 `s_free` 有什么作用？

答题模板：

```text
UNIX 使用成组链接管理空闲块。超级块中保存当前一组空闲块号，
用 s_nfree 表示数量，用 s_free 保存块号。分配时优先从当前组取；
当 s_nfree 为 1 时，s_free[0] 指向下一组登记块，需要读入下一组。
回收时若当前组未满，直接加入 s_free；若已满，则把当前组信息写入
刚释放的块，并让该块成为新的组链块。
```

## 23. 易混概念总表

| 易混概念 | 区别 |
|---|---|
| 文件 vs 磁盘块 | 文件是逻辑信息对象；磁盘块是物理存储单位 |
| 文件系统 vs 文件集合 | 文件系统还包括目录、FCB/inode、打开表、空闲空间管理等 |
| 逻辑组织 vs 物理组织 | 逻辑组织描述用户看到的记录/字节流；物理组织描述磁盘块存放方式 |
| 顺序访问 vs 顺序结构 | 顺序访问是访问方式；顺序结构是物理组织方式 |
| 随机访问 vs 索引结构 | 随机访问是需求；索引结构是支持随机访问的一种实现 |
| 目录项 vs 目录文件 | 目录项是一条记录；目录文件由许多目录项组成 |
| 文件目录 vs 目录文件 | 文件目录是目录体系；目录文件是具体存放目录项的文件 |
| FCB vs 文件内容 | FCB 是管理信息；文件内容是用户数据 |
| FCB 次部 vs 主部 | 次部通常是文件名和文件号；主部是权限、长度、地址、链接数等 |
| inode 号 vs 文件名 | inode 号标识文件实体；文件名是目录中的名字 |
| 硬链接 vs 复制文件 | 硬链接增加目录项；复制文件会产生新的文件实体和数据副本 |
| `link` vs `unlink` | `link` 增加目录项和链接计数；`unlink` 删除目录项并减少链接计数 |
| `i_nlink` vs `i_count` | `i_nlink` 是目录项链接数；`i_count` 是内存 inode 引用数 |
| 用户打开文件表 vs 系统打开文件表 | 前者每进程一张；后者全系统共享 |
| 文件描述符 vs inode | fd 是进程访问打开文件的句柄；inode 是文件实体管理信息 |
| `open` vs `read` | `open` 建立访问通道并返回 fd；`read` 通过 fd 读取数据 |
| `seek` vs `read` | `seek` 改读写位置；`read` 真正读数据 |
| 文件保护 vs 文件保密 | 保护防未授权访问；保密防信息泄露 |
| 文件安全 vs 文件保护 | 安全防破坏；保护防非法访问 |
| 位示图 vs 成组链接 | 位示图用位表示空闲状态；成组链接用分组块号链管理空闲块 |
| 日志结构文件系统 vs 操作日志 | LSFS 把文件系统更新组织成顺序追加的日志式存储 |
| 内存映射文件 vs 普通 read/write | mmap 后像访问内存一样访问文件，普通 I/O 通过系统调用读写 |

- [ ] 为什么链接结构文件增长方便但随机访问差。
- [ ] 为什么多级目录优于单级目录和二级目录。

### 24.3 【易错】考前专门过一遍

- [ ] “两级索引”不是 `2 * 每块指针数`，而是 `每块指针数^2`。
- [ ] 索引文件最大长度通常只算数据块容量，不把索引块当成文件数据。
- [ ] 目录项不一定保存完整 FCB；UNIX 中通常保存文件名和 inode 号。
- [ ] `unlink` 删除的是名字，不一定立刻删除文件数据。
- [ ] 链接计数 `i_nlink` 不是打开计数。
- [ ] 删除文件不等于删除该文件所在目录。
- [ ] `seek` 只改变偏移，不直接读写数据。
- [ ] 文件保护、保密、安全分别对应权限、加密、备份/防破坏。
- [ ] 成组链接中 `s_nfree == 1` 和 `s_nfree == 100` 是两个特殊边界。

## 25. 本章一句话总结

文件系统的主线是：

```text
用户用路径名找到目录项，
目录项找到 FCB/inode，
FCB/inode 找到磁盘数据块，
打开文件表让后续读写不用反复查路径，
链接计数决定文件名字删掉后实体是否还能保留，
空闲空间管理决定新文件如何获得磁盘块。
```

期末最值得抓住的硬考点是：

```text
索引长度计算、FCB/inode/目录项区别、打开文件表关系、
link/unlink 流程、删除文件判断、UNIX 成组链接。
```
