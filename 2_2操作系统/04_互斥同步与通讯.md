## 从 0 理解

并发的问题在于：两个进程看起来都按自己的代码顺序执行，但它们之间的交叉顺序不可预测。如果它们共享变量，结果就可能取决于“谁刚好先执行一步”。

临界资源是一次只允许一个进程访问的资源，访问它的代码段叫临界区。互斥不是让程序慢下来，而是让共享资源的访问变得可预测。

同步和互斥的区别：

- 互斥：不能同时做。例如两个进程不能同时写同一共享变量。
- 同步：必须按某种顺序做。例如缓冲区空时消费者必须等生产者先放入产品。

## 核心机制/算法/公式

### 互斥三原则

| 原则 | 含义 |
|---|---|
| 互斥 | 同一时刻最多一个进程进入相关临界区 |
| 进展 | 临界区空闲时，应允许某个申请者进入 |
| 有限等待 | 申请者不能无限期等待 |

### 信号量语义

P 操作：申请资源。若资源不足，则进程等待。

V 操作：释放资源。若有等待进程，则唤醒一个。

如果信号量初值为 3，当前值为 -2，表示已有 3 个资源被用完，还有 2 个进程正在等待。

### PV 解题模板

1. 找资源：缓冲区空位、产品数量、座位、盘子、设备等。
2. 找互斥：哪些共享变量或共享容器一次只能一个进程改。
3. 找同步顺序：谁必须等谁。
4. 定义信号量初值和含义。
5. 写进程伪代码，检查每条路径是否 P/V 配对。

### 生产者消费者模板

```c
semaphore empty = N;
semaphore full = 0;
semaphore mutex = 1;

producer() {
  while (true) {
    produce();
    P(empty);
    P(mutex);
    put();
    V(mutex);
    V(full);
  }
}

consumer() {
  while (true) {
    P(full);
    P(mutex);
    get();
    V(mutex);
    V(empty);
    consume();
  }
}
```

### 读者写者

读读可以并发，读写和写写必须互斥。常见写法用 `readcount` 记录读者数，第一个读者 P 写锁，最后一个读者 V 写锁。

### test-and-set 有限等待改进思路

普通 `while(test_and_set(&lock)) continue;` 可能让某个进程反复抢不到锁，不满足有限等待。改进需要引入等待数组和轮转选择，让释放锁的进程按顺序把机会交给某个等待者，而不是让所有进程自由竞争。

## 易错点与区分表

| 易错点 | 正确理解 |
|---|---|
| 信号量负值 | 绝对值表示等待进程数 |
| P/V 顺序 | 先同步资源，再进互斥区，避免占着 mutex 等资源 |
| 管程 signal | 不完全等同于信号量 V，和管程语义、紧急队列有关 |
| 忙等 | 进程占 CPU 循环等待；阻塞等待则让出 CPU |
| 临界区内申请 I/O | 若未释放相关临界资源，其他进程不能进入相关临界区 |

## 题型与解析

### 作业题：Hyman 算法反例

题目要求证明 Hyman 互斥算法错误。关键抓住：可以构造交错执行，让 P0 在 `turn:=id` 前后被切换，随后 P1 进入临界区，最终 P0 和 P1 都进入临界区，破坏互斥性。

答题时不需要背完整代码，写出反例思路即可：

1. 令 `turn=1`。
2. P0 设置 `blocked[0]=true`，推进到将要执行 `turn=0` 的位置。
3. 切换到 P1，P1 因 `turn=1` 可进入临界区。
4. 切回 P0，P0 设置 `turn=0` 后也进入临界区。
5. 两者同时在临界区，故算法错误。

### 作业题：单缓冲、多消费者每条消息都要读一次

核心思想：缓冲区只有一条消息，所有 n 个消费者都读过之后，生产者才能放下一条。需要：

- `empty=1`：生产者是否可放。
- `full[i]=0`：第 i 个消费者是否可读当前消息。
- `mutex=1`：保护已读计数。
- `count=0`：当前消息已被多少消费者读取。

生产者放入消息后，对每个消费者 `V(full[i])`。消费者 i 读完后递增 `count`，最后一个读者 `V(empty)`。

### 真题题型：爸爸、儿子、女儿水果问题

这类题不要先写代码，先列约束：

- 盘子容量 1：`plate_empty` 与具体水果信号量。
- 桌边 2 个座位：`seat=2`。
- 儿子只吃苹果，女儿只吃橘子。
- 没水果时孩子叫醒爸爸，爸爸按叫醒者需求放水果。
- 需要分析饥饿：若调度长期偏向一个孩子，另一个孩子可能长期得不到想要水果；若要求公平，需要额外队列或轮转策略。
