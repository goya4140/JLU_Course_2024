## 1. 并发进程：为什么程序一并发就容易出错

### 1.1 从顺序程序开始理解

顺序程序的直觉很简单：程序一步一步执行，前一步的结果给后一步使用。

比如：

```text
S1: a = x + y
S2: b = a - z
S3: c = a + b
S4: d = c + 5
```

`S2` 依赖 `S1` 的结果，`S3` 又依赖前面的结果。只要输入相同，程序每次运行的结果应该相同。

课件把顺序程序的特性概括为：

| 特性 | 含义 | 直觉 |
|---|---|---|
| 连续性 | 指令逐条执行 | 程序像一条线 |
| 封闭性 | 不受其他程序和外界因素影响 | 自己的变量不被别人偷偷改 |
| 可再现性 | 结果与推进速度无关 | 同样输入，多次运行结果一致 |

考试判断：

- “顺序程序具有可再现性”一般是对的。
- “并发程序天然具有可再现性”是错的。
- “可再现性和运行速度无关”是课件定义里的重点。

### 1.2 【重点】前趋图

前趋图是一个有向无环图。图中每个结点可以表示语句、计算步骤或进程；有向边表示“谁必须先完成”。

课内定义：

```text
Pi -> Pj 表示 Pj 启动之前 Pi 必须已经完成。
```

没有前趋的结点叫初始结点，没有后继的结点叫终止结点。

直觉上，前趋图就是把“哪些步骤必须排队，哪些步骤可以并发”画出来。

考试常见判断：

- 有环就不是合法前趋图，因为依赖关系会互相等待。
- 没有路径约束的两个结点可以并发执行。
- 前趋图描述的是偏序，不是唯一的全序。

### 1.3 【必会】并发程序的特性

并发不是简单地“同时运行”。在单 CPU 上，并发通常是多个进程交替占用 CPU；在多 CPU 上，可能真的同时执行。但无论哪一种，只要它们访问共享数据，就可能因为执行交叉不同而得到不同结果。

课件给出并发程序的三个特性：

| 特性 | 含义 | 为什么重要 |
|---|---|---|
| 间断性 | 程序交叉执行，走走停停 | 进程随时可能被中断 |
| 非封闭性 | 一个进程的运行环境可能被其他进程改变 | 共享变量可能被别人改 |
| 不可再现性 | 多次执行可能对应不同交叉，结果不同 | 这是同步互斥要解决的问题 |

生活化理解：

```text
两个人同时改同一张表格。
每个人都觉得自己按步骤改得没错，
但如果 A 刚读完旧值，B 先写入新值，A 再用旧值覆盖，
最后结果就错了。
```

### 1.4 【必会】与时间有关的错误

与时间有关的错误，就是程序逻辑本身看起来没问题，但结果依赖于进程交叉时机。

典型例子是共享变量 `N`：

```text
进程 P:
  A1: N = 0
  A2: N = N + 1
  A3: goto A2

进程 Q:
  B1: print(N)
  B2: N = 0
  B3: goto B1
```

如果 Q 在打印后、清零前被中断，P 又对 `N` 加了几次，随后 Q 清零，就会把 P 的累计结果抹掉。

考试判断方式：

- 只要多个进程读写同一共享变量，并且至少有一个写操作，就要警惕与时间有关的错误。
- 错误不一定每次出现，它取决于交叉时机。
- “多次运行结果不同”不是随机数导致的，而是并发交叉导致的。

### 1.5 【重点】Bernstein 条件

课件给出程序并发执行仍保持可再现性的条件。设：

```text
R(pi) = 程序 pi 读的变量集合
W(pi) = 程序 pi 写的变量集合
```

两个程序 `p1`、`p2` 可以安全并发的条件是：

```text
R(p1) ∩ W(p2)  ∪  R(p2) ∩ W(p1)  ∪  W(p1) ∩ W(p2) = 空集
```

把公式翻译成人话：

```text
别人写的东西，我不能读；
我写的东西，别人不能读；
我写的东西，别人也不能写。
```

也就是：

- 读-读不冲突。
- 读-写冲突。
- 写-读冲突。
- 写-写冲突。

【易错】不要看到“共享变量”就认为一定不能并发。多个进程只读同一个变量，通常不冲突；真正危险的是写操作和其他读/写交叉。

## 2. 临界资源、临界区与进程互斥

### 2.1 【必会】共享变量与临界区

共享变量是多个进程都需要访问的变量。

临界区域，也常叫临界区，是访问共享变量的程序段。

课件强调：临界区要和它对应的共享变量联系起来理解。也就是说，所谓“临界”，不是某段代码天然危险，而是它访问了某组需要保护的共享变量。

例子：

```c
// shared x
if (x > 0) {
  x = x - 1;
}
```

如果多个进程同时执行这段代码，就可能都看到 `x > 0`，然后重复扣减或重复借出资源。

【题型】习题课模拟选择：

```text
所谓临界区是指什么？
```

答案应选“一段程序”，不是缓冲区、数据区或同步机制。

考试判断：

- 临界资源：一次只允许一个进程访问的资源。
- 临界区：访问临界资源或共享变量的代码段。
- 互斥：多个进程不能同时进入关于同一组共享变量的临界区。

### 2.2 【必会】进程互斥的二层含义

课件对进程互斥的定义有两个层次：

1. 任何时刻最多只能有一个进程处于同一组共享变量的相同临界区。
2. 任何时刻最多只能有一个进程处于同一组共享变量的不同临界区。

第二点容易漏。比如：

```text
CR1: 修改账户余额
CR2: 统计账户余额
```

这两个临界区代码不同，但如果它们访问的是同一组共享变量，就不能同时进入。

应试口诀：

```text
互斥不是按“代码段名字”判断，
而是按“是否访问同一组共享变量”判断。
```

### 2.3 【必会】临界区四段结构

进程互斥算法通常写成：

```text
repeat
  entry section
  critical section
  exit section
  remainder section
until false
```

| 部分 | 作用 |
|---|---|
| entry section | 申请进入临界区 |
| critical section | 真正访问共享变量 |
| exit section | 离开临界区，释放资格 |
| remainder section | 与共享变量无关的其余代码 |

考试判断：

- `P(mutex)` 常放在 entry section。
- `V(mutex)` 常放在 exit section。
- 临界区不是 P/V 本身，而是被 P/V 保护的共享变量访问代码。

### 2.4 【必会】互斥正确性的三原则

习题课明确要求掌握三原则：

| 原则 | 英文 | 含义 | 错误表现 |
|---|---|---|---|
| 互斥 | mutual exclusion | 同一组共享变量的临界区一次只允许一个进程进入 | 两个进程同时进临界区 |
| 进展 | progress | 临界区空闲时，应允许某个申请者进入 | 明明没人用，却没人能进 |
| 有限等待 | bounded waiting | 申请者等待有限个进程进入并离开后，应获得机会 | 某进程长期被插队，可能饿死 |

【必会】评价互斥算法时，不是只看能不能互斥，还要看进展和有限等待。

比如一个算法强制 P0、P1 轮流进入临界区。它可能满足互斥，但如果轮到 P1，而 P1 根本不想进入，P0 却被挡在外面，就不满足进展。

## 3. 进程互斥的软件实现

### 3.1 软件实现的特点

软件实现指完全用普通程序和共享变量实现互斥，不依赖特殊硬件指令。

课件总结：

- 可用于单 CPU 和多 CPU 环境。
- 不需要特殊硬件支持。
- 通常存在忙式等待问题。

忙式等待就是进程没有让出 CPU，而是在循环里一直检查条件：

```c
while (flag[other]) {
  // 什么也不做，但一直占 CPU
}
```

【易错】忙等的进程可能处于运行态或就绪态，不是等待态。等待态通常表示进程阻塞并让出 CPU，等待事件唤醒。

### 3.2 尝试 1：严格轮转

思想：

```c
while (turn == other) ;
critical section;
turn = other;
```

这个算法的问题是：两个进程必须交替进入临界区。

如果 `turn` 指向 P1，但 P1 不想进入临界区，P0 即使想进也进不去。临界区空闲，却不能放行 P0。

结论：

```text
满足互斥，但不满足进展。
```

### 3.3 尝试 2：先检查对方，再声明自己

一种错误思路：

```c
while (flag[other]) ;
flag[self] = true;
critical section;
flag[self] = false;
```

如果两个进程一开始都看到对方 `flag=false`，随后都把自己的 `flag` 设为 true，就可能一起进入临界区。

结论：

```text
不满足互斥。
```

### 3.4 尝试 3：先声明自己，再检查对方

另一种错误思路：

```c
flag[self] = true;
while (flag[other]) ;
critical section;
flag[self] = false;
```

如果两个进程几乎同时把自己的 `flag` 设为 true，那么它们都会看到对方想进入，于是都在 while 处忙等。

结论：

```text
满足互斥的倾向，但不满足进展，可能双方都进不去。
```

### 3.5 【重点】Peterson 算法

Peterson 算法用于两个进程互斥。它把“我想进”和“如果冲突让给谁”结合起来。

典型结构：

```c
// P0
flag[0] = true;
turn = 1;
while (flag[1] && turn == 1) ;
critical section;
flag[0] = false;
```

```c
// P1
flag[1] = true;
turn = 0;
while (flag[0] && turn == 0) ;
critical section;
flag[1] = false;
```

直觉解释：

```text
flag 表示“我想进”；
turn 表示“如果我们都想进，我让对方先进”。
```

如果只有一个进程想进，它可以直接进入。  
如果两个都想进，最后一次写 `turn` 的进程会把机会让给对方，因此只能有一个进。

考试一般不要求完整证明，但要知道：

- Peterson 是软件互斥算法。
- 用于两个进程。
- 依赖 `flag[]` 和 `turn`。
- 目标是满足互斥、进展和有限等待。

### 3.6 【重点】Lamport 面包店算法

面包店算法用于多个进程互斥。

直觉：

```text
想进入临界区的进程先取号；
号码小的先进；
号码相同时，进程编号小的先进。
```

课件中比较关系：

```text
(a, b) < (c, d) 当且仅当 a < c 或者 a == c 且 b < d
```

其中 `a/c` 是号码，`b/d` 是进程编号。

为什么要有 `choosing[]`？

因为取号不是瞬间完成的。可能 P1 正在取号但还没写入，P2 看到旧值也取了同样的号。如果没有 `choosing[]` 辅助判断，就可能破坏互斥。

考试定位：

- 面包店算法是软件算法。
- 面向 n 个进程。
- 用取号和编号解决竞争。
- 可以满足互斥、进展、有限等待。

### 3.7 【必会】Hyman 算法反例

作业三和习题课都明确考过 Hyman 算法。题目给出如下结构：

```pascal
blocked[id] := true;
while turn <> id do
begin
  while blocked[1-id] do
    {nothing}
  turn := id
end;
<Critical section>
blocked[id] := false;
```

考试不是要你背代码，而是要你能构造交叉执行，说明算法错误。

【题型】反例思路：

初始：

```text
blocked[0] = false
blocked[1] = false
turn = 0
```

构造一种交叉：

1. P1 运行，执行 `blocked[1] = true`。
2. 因为 `turn != 1`，P1 进入外层 while。
3. P1 检查 `blocked[0]`，发现为 false，于是准备执行 `turn = 1`，但此时切换。
4. P0 运行，执行 `blocked[0] = true`。
5. P0 发现 `turn == 0`，直接跳过 while，进入临界区。
6. 切回 P1，P1 执行刚才尚未执行的 `turn = 1`。
7. P1 发现 `turn == 1`，也进入临界区。

结果：

```text
P0 和 P1 同时进入临界区，破坏互斥性。
```

【易错】反例的关键不在于 `turn` 初值一定是什么，而在于抓住“检查条件后、赋值前发生切换”这种交叉。考试写反例时要把每一步写清楚，不要只说“可能同时进入”。

## 4. 进程互斥的硬件实现

### 4.1 为什么需要硬件原子指令

软件算法可以实现互斥，但复杂、难证明，而且忙等明显。硬件提供原子指令后，可以更直接地构造锁。

原子指令的意思是：

```text
这条指令执行期间不会被中断或交叉。
```

互斥最怕的就是“我刚检查完，还没修改，就被切换”。原子指令把检查和修改合成一个不可分割动作。

### 4.2 【必会】test_and_set

课件给出测试并设置指令：

```c
int test_and_set(int *target) {
  int temp;
  temp = *target;
  *target = 1;
  return temp;
}
```

用它实现互斥：

```c
int lock = 0;

do {
  while (test_and_set(&lock))
    continue;

  critical section;

  lock = 0;

  remainder section;
} while (1);
```

直觉：

- `lock=0` 表示锁空闲。
- 第一个执行 `test_and_set` 的进程会读到 0，并把锁设为 1，于是进入临界区。
- 后续进程读到 1，只能继续忙等。

课件结论：

```text
满足互斥性、进展性，不满足有限等待性。
```

### 4.3 【必会】为什么 test_and_set 不满足有限等待

真题简答直接考：

```text
说明何种状况下该算法不满足有限等待特性。
```

普通 `test_and_set` 的释放方式是：

```c
lock = 0;
```

锁一释放，所有等待进程一起竞争。哪个进程下一次先执行到 `test_and_set`，哪个就拿到锁。

这会导致：

```text
某个等待很久的进程不一定优先获得锁；
它可能每次都慢一步，被其他进程反复抢先。
```

所以它可能长期等待，甚至饥饿，不满足有限等待。

答题模板：

```text
当多个进程忙等同一把锁时，临界区释放后所有等待者自由竞争。
若某进程每次竞争都失败，而其他进程反复成功进入临界区，
则该进程等待的进入次数没有上界，因此不满足 bounded waiting。
```

### 4.4 【题型】test_and_set 有限等待改进

课件给出的改进思想是增加等待数组，并在退出临界区时按循环次序指定一个后继进程。

伪代码可写为：

```c
boolean waiting[n] = {false};
int lock = 0;

// Pi entry
waiting[i] = true;
key = true;
while (waiting[i] && key)
  key = test_and_set(&lock);
waiting[i] = false;

// critical section

// Pi exit
j = (i + 1) % n;
while (j != i && !waiting[j])
  j = (j + 1) % n;

if (j == i)
  lock = 0;
else
  waiting[j] = false;
```

直觉：

```text
普通 test_and_set 是“大家抢”；
改进版是“离开者按循环顺序点名下一个等待者”。
```

为什么满足有限等待？

- 每个进程最多等待循环中排在自己前面的有限个进程。
- 释放锁时不是让所有人重新自由竞争，而是把机会交给某个等待者。

考试写法不必逐字背代码，但必须写出两个关键：

1. 增加 `waiting[i]` 表示进程是否等待。
2. 退出时按循环顺序寻找下一个等待者，使等待次数有上界。

### 4.5 【重点】swap 指令

课件给出交换指令：

```c
void swap(int *a, int *b) {
  int temp = *a;
  *a = *b;
  *b = temp;
}
```

用它实现互斥：

```c
int lock = 0;
int key;

do {
  key = 1;
  do {
    swap(&lock, &key);
  } while (key == 1);

  critical section;

  lock = 0;

  remainder section;
} while (1);
```

直觉：

进程拿自己的 `key=1` 去和全局 `lock` 交换。

- 如果原来 `lock=0`，交换后 `key=0`，进程进入临界区。
- 如果原来 `lock=1`，交换后 `key=1`，继续等待。

考试定位：`swap` 和 `test_and_set` 都是硬件原子指令，都可用于实现互斥，都可能产生忙等。

### 4.6 【必会】关中断与开中断

最简单的硬件互斥方法：

```text
关中断
  critical section
开中断
```

直觉：

在单 CPU 系统中，进程切换通常依赖中断。如果关中断，当前 CPU 不会被普通中断打断，就能连续执行完临界区。

但课件强调：

```text
开关中断只在单 CPU 系统中有效，并且影响并发性。
```

为什么多 CPU 不行？

因为关掉一个 CPU 的中断，并不能阻止另一个 CPU 上的进程同时访问共享变量。

考试判断：

- 单 CPU、很短的内核临界区，可以用关中断。
- 多处理机环境不能只靠关中断实现全局互斥。
- 关中断会影响系统响应和并发性，因此临界区应很短。

### 4.7 【重点】多处理机与自旋锁

在多 CPU 环境下，即使 `test_and_set` 或 `swap` 是一条机器指令，也要保证它对共享内存的读写在总线上不可交叉。课件提到需要先锁总线。

自旋锁的直觉：

```text
拿不到锁时不睡眠，而是在 CPU 上循环等待。
```

适用场景：

- 临界区很短。
- 进入阻塞/唤醒的开销反而更大。
- 内核或多处理机环境中常见。

不适用场景：

- 等待时间很长。
- 单 CPU 用户进程长时间忙等，会浪费 CPU。

## 5. 进程同步：从互斥到顺序约束

### 5.1 【必会】互斥与同步的区别

互斥解决的是：

```text
不能同时做。
```

同步解决的是：

```text
必须按某种顺序做。
```

例子：

| 场景 | 类型 | 原因 |
|---|---|---|
| 两个进程不能同时修改同一个计数器 | 互斥 | 保护共享变量 |
| 消费者必须等生产者放入产品后才能取 | 同步 | 有先后顺序 |
| 司机必须等售票员关门后才能开车 | 同步 | 事件顺序 |
| 多个读者可同时读，写者独占 | 互斥 + 同步 | 读写规则不同 |

应试口诀：

```text
互斥看“同一时刻能不能一起”；
同步看“谁必须等谁先发生”。
```

### 5.2 同步机制概览

课件列出典型同步机制：

- 信号量与 P/V 操作。
- 管程。
- 会合。
- 条件临界区。
- 路径表达式。
- 事件。

期末最核心的是信号量与 P/V。管程和高级通信也会考选择题或概念题。

## 6. 信号量与 P/V 操作

### 6.1 【必会】信号量的定义

课件定义信号量为：

```c
typedef struct {
  int value;
  PCBpointer queue;
} semaphore;
```

它有两个组成：

| 成员 | 含义 |
|---|---|
| `value` | 资源数量或同步状态 |
| `queue` | 等待该信号量的进程队列 |

信号量不是普通整型变量。课件规定：

1. 必须置一次初值，且只能置一次初值。
2. 初值必须大于等于 0。
3. 除初始化外，只能对它执行 P 操作和 V 操作。
4. 其他直接读写都是非法的。

【易错】考试题里可以问 `mutex=-1` 的含义，但程序里不能随便写 `mutex = -1` 来控制同步。

### 6.2 【必会】P 操作

P 操作表示申请资源，或者等待某事件发生。

课件伪代码：

```c
void P(semaphore *s) {
  s->value--;
  if (s->value < 0)
    asleep(s->queue);
}
```

其中 `asleep` 做两件事：

1. 把当前进程 PCB 放入 `s->queue` 队尾，状态改为等待。
2. 转处理机调度程序，让出 CPU。

直觉：

```text
P = 先把资源数减 1。
如果减完还 >= 0，说明申请成功；
如果减完 < 0，说明资源不够，当前进程进入等待队列。
```

### 6.3 【必会】V 操作

V 操作表示释放资源，或者通知某事件已经发生。

课件伪代码：

```c
void V(semaphore *s) {
  s->value++;
  if (s->value <= 0)
    wakeup(s->queue);
}
```

其中 `wakeup` 把等待队列队首 PCB 移出，放入就绪队列。

直觉：

```text
V = 先把资源数加 1。
如果加完 <= 0，说明之前有人在等，
于是唤醒一个等待进程。
```

【易错】被 V 唤醒的进程进入就绪队列，不是立刻运行。它还要等处理机调度。

### 6.4 【必会】信号量 value 的考试含义

课件结论：

| `s->value` 状态 | 含义 |
|---|---|
| `value >= 0` | 等待队列为空，`value` 表示可用资源数 |
| `value < 0` | `|value|` 表示等待队列长度 |

【题型】真题选择：

```text
信号量初值为 3，经过多次 P、V 操作后当前值为 -2，
表示等待该资源的进程数为多少？
```

答案：

```text
2
```

因为负值的绝对值表示等待进程数。

【题型】作业三选择：

```text
两个并发进程，互斥信号量 mutex 初值为 1，若 mutex = -1，则？
```

分析：

- `mutex` 初值为 1，表示临界区一次允许 1 个进程进入。
- 当前 `mutex=-1`，表示有 1 个进程在等待。
- 对两个进程来说，另一个进程必然已经进入临界区。

答案：

```text
有一个进程进入临界区，另一个进程等待进入。
```

应试口诀：

```text
信号量为负，看绝对值：几个进程在等。
互斥量初值为 1，等的人之外通常还有一个正在用。
```

### 6.5 【必会】信号量的三种典型用途

| 用途 | 初值 | 含义 | 典型写法 |
|---|---|---|---|
| 互斥 | 1 | 一次只允许一个进程进入 | `P(mutex)` / `V(mutex)` |
| 简单同步 | 0 | 某事件尚未发生，等待通知 | 先发生者 `V(s)`，后发生者 `P(s)` |
| 同种资源管理 | 资源个数 k | 还有 k 个可用资源 | 申请 `P(s)`，归还 `V(s)` |

例子：

```c
semaphore mutex = 1;  // 保护共享变量
semaphore full = 0;   // 当前产品数
semaphore empty = k;  // 当前空位数
```

【易错】同步信号量初值常为 0，因为一开始“被等待的事件”还没发生。

### 6.6 P/V 是原语

课件强调 P/V 操作是原语，即不可中断的一段代码。

为什么必须不可中断？

如果两个进程同时对同一个信号量执行 `s->value--`，而这个减法不是原子的，就会再次出现共享变量竞争。信号量本身就是用来解决并发问题的，如果它自己的操作不原子，就失去意义。

## 7. PV 解题总模板

### 7.1 【必会】从题目翻译到信号量

PV 大题最重要的不是背具体题，而是会拆题。

建议按这个顺序：

```text
1. 找资源
2. 找互斥
3. 找同步顺序
4. 定义信号量初值和含义
5. 写每个进程的循环伪代码
6. 检查 P/V 顺序、死锁、饥饿
```

### 7.2 找资源

资源是“数量有限、用完要等”的东西。

常见资源：

| 题目对象 | 信号量含义 |
|---|---|
| 缓冲区空位 | `empty = 缓冲区容量` |
| 缓冲区产品 | `full = 0` |
| 座位 | `seat = 座位数` |
| 盘子空 | `plate_empty = 1` |
| 苹果/橘子 | `apple = 0`, `orange = 0` |
| n 台同类设备 | `device = n` |

### 7.3 找互斥

只要多个进程会修改同一个共享变量或同一个共享容器，就要考虑互斥。

常见互斥对象：

- 缓冲区数组。
- 输入/输出指针 `in/out`。
- 计数变量 `count/read_count`。
- 消息队列。
- 共享状态 `state[]`。

写法：

```c
P(mutex);
// 修改共享变量或共享容器
V(mutex);
```

### 7.4 找同步顺序

同步顺序是“谁必须等谁”。

常见句式：

| 题目表达 | 翻译 |
|---|---|
| 消费者必须等生产者放入产品 | 消费者先 `P(full)`，生产者放后 `V(full)` |
| 生产者必须等缓冲区有空位 | 生产者先 `P(empty)`，消费者取后 `V(empty)` |
| 司机必须等售票员关门 | 司机 `P(door_closed)`，售票员关门后 `V(door_closed)` |
| 所有消费者都读过后生产者才能继续 | 最后一个消费者 `V(empty)` |

### 7.5 【易错】P/V 顺序

生产者-消费者里常见正确顺序：

```c
P(empty);
P(mutex);
put();
V(mutex);
V(full);
```

消费者：

```c
P(full);
P(mutex);
get();
V(mutex);
V(empty);
```

为什么不是先 `P(mutex)` 再 `P(empty/full)`？

如果生产者先拿到 `mutex`，再发现缓冲区满而等待 `empty`，它会占着互斥锁睡眠。消费者想进入缓冲区取产品、释放空位，却拿不到 `mutex`。于是形成死锁。

应试口诀：

```text
先 P 资源，再 P 互斥；
先 V 互斥，再 V 对方需要的资源。
```

这不是绝对语法规则，但对大多数缓冲区题非常安全。

## 8. 经典同步互斥问题

### 8.1 【必会】生产者-消费者问题

问题：

```text
生产者把产品放入容量为 k 的缓冲区；
消费者从缓冲区取产品；
缓冲区满时生产者等待；
缓冲区空时消费者等待；
访问缓冲区需要互斥。
```

信号量：

```c
semaphore empty = k;  // 空缓冲区数量
semaphore full = 0;   // 产品数量
semaphore mutex = 1;  // 保护缓冲区和 in/out
```

生产者：

```c
producer() {
  while (true) {
    item = produce();

    P(empty);
    P(mutex);
    put(item);
    V(mutex);
    V(full);
  }
}
```

消费者：

```c
consumer() {
  while (true) {
    P(full);
    P(mutex);
    item = get();
    V(mutex);
    V(empty);

    consume(item);
  }
}
```

考试判断：

- `empty` 初值是缓冲区容量。
- `full` 初值是 0。
- `mutex` 初值是 1。
- `P(empty/full)` 要放在 `P(mutex)` 前面。

### 8.2 【重点】提高生产者-消费者并发性

课件提到，如果生产者和消费者操作的是缓冲区不同分量，可以用两个互斥信号量提高并发：

```c
semaphore mutex_in = 1;   // 保护 in 和 B[in]
semaphore mutex_out = 1;  // 保护 out 和 B[out]
```

生产者只保护 `in`，消费者只保护 `out`。这样在缓冲区既非空也非满时，生产者和消费者可能同时操作不同位置。

考试一般不会要求完整写这个优化，但要理解：

```text
互斥粒度越细，并发性越高；
但设计也更复杂。
```

### 8.3 【必会】读者-写者问题

问题：

```text
多个读者可以同时读；
读者和写者不能同时访问；
多个写者也不能同时写。
```

规则：

| 组合 | 是否允许同时 |
|---|---|
| 读者-读者 | 允许 |
| 读者-写者 | 不允许 |
| 写者-写者 | 不允许 |

基本读者优先写法：

```c
int read_count = 0;
semaphore mutex = 1;   // 保护 read_count
semaphore rw = 1;      // 控制读写互斥

reader() {
  while (true) {
    P(mutex);
    read_count++;
    if (read_count == 1)
      P(rw);
    V(mutex);

    read();

    P(mutex);
    read_count--;
    if (read_count == 0)
      V(rw);
    V(mutex);
  }
}

writer() {
  while (true) {
    P(rw);
    write();
    V(rw);
  }
}
```

直觉：

- 第一个读者负责锁住写者。
- 最后一个读者负责释放写者。
- 中间来的读者可以一起读。

【易错】这个解法可能导致写者饥饿。因为如果读者源源不断，`read_count` 可能一直不归 0，写者一直拿不到 `rw`。

### 8.4 【重点】写者优先读者-写者

写者优先的思想：

```text
一旦有写者等待，新来的读者不要再进入；
等已有读者读完后，让写者先写。
```

这常用于解释管程中的读者-写者题。考试如果不要求完整 PV 代码，至少要会说：

- 读者优先可能导致写者饥饿。
- 写者优先需要记录等待/正在写的写者数量。
- 新读者在有写者等待时应阻塞。

### 8.5 【重点】哲学家进餐问题

问题：

```text
5 个哲学家围坐，5 把叉子；
每个哲学家吃饭需要左右两把叉子。
```

直接写法：

```c
P(fork[i]);
P(fork[(i + 1) % 5]);
eat();
V(fork[i]);
V(fork[(i + 1) % 5]);
```

可能死锁：

```text
每个人都拿起左叉，同时等待右叉。
```

课件解决思路之一是引入状态和每个哲学家的等待信号量：

```text
thinking -> hungry -> eating
```

只有当左右邻居都不在 eating 时，哲学家才能进入 eating。

考试定位：

- 哲学家问题体现“不同种组合资源”。
- 要注意死锁和饥饿。
- 若只考选择/简答，知道“同时拿左右两把叉子”或“状态测试”可避免死锁。

### 8.6 【重点】吸烟者问题与 SP/SV

习题课提到吸烟者问题和 SP/SV。这个知识点通常不是期末大题主角，但它表达了一个思想：

```text
有些问题需要同时申请多个资源。
```

普通 P 操作一次只申请一个资源，若进程拿到一部分资源后等待另一部分，就可能形成死锁。SP/SV 或 AND 型信号量强调“同时满足条件才一起申请”，避免部分占有。

考试边界：

- 知道它与“同时申请多个资源”有关。
- 不必把 SP/SV 作为第 4 章主线展开成复杂证明。

## 9. 作业与真题 PV 变形题

### 9.1 【题型】多生产者、n 消费者、每条消息都要读一次

题目：

```text
多个生产者向 1 个缓冲区不断发消息；
n 个消费者从该缓冲区取消息；
缓冲区只能存放 1 条消息；
对每条放入缓冲区的消息，所有消费者必须都接收 1 次，
生产者才可以继续放下一条消息。
```

这题的难点不是“生产者消费者”，而是：

```text
一条消息不是被一个消费者取走，
而是必须被所有 n 个消费者各读一次。
```

#### 9.1.1 建模

资源与同步关系：

| 约束 | 信号量/变量 |
|---|---|
| 缓冲区一次只能放 1 条消息 | `empty = 1` |
| 多个生产者放消息要互斥 | `p_mutex = 1` 或由 `empty` 间接限制 |
| 每个消费者对每条消息读一次 | `can_read[i] = 0` |
| 统计已经读过的消费者数量 | `read_count = 0` |
| 保护 `read_count` | `count_mutex = 1` |
| 保护缓冲区读写 | 可用 `buf_mutex = 1`，也可根据读操作是否破坏消息简化 |

这里推荐写成清晰版本：

```c
semaphore empty = 1;              // 缓冲区是否可写
semaphore can_read[n] = {0};      // 第 i 个消费者是否可读当前消息
semaphore count_mutex = 1;        // 保护 read_count
semaphore buf_mutex = 1;          // 保护缓冲区写入/读取动作
int read_count = 0;
message buffer;
```

#### 9.1.2 生产者伪代码

```c
producer() {
  while (true) {
    message m = produce();

    P(empty);

    P(buf_mutex);
    buffer = m;
    read_count = 0;
    V(buf_mutex);

    for (int i = 0; i < n; i++)
      V(can_read[i]);
  }
}
```

#### 9.1.3 消费者 i 伪代码

```c
consumer(int i) {
  while (true) {
    P(can_read[i]);

    P(buf_mutex);
    message m = buffer;
    V(buf_mutex);

    consume(m);

    P(count_mutex);
    read_count++;
    if (read_count == n)
      V(empty);
    V(count_mutex);
  }
}
```

#### 9.1.4 为什么正确

互斥：

- `buf_mutex` 保护缓冲区访问。
- `count_mutex` 保护 `read_count`。

同步：

- 生产者必须先 `P(empty)`，保证上一条消息已被所有消费者读完。
- 生产者放入消息后，对每个 `can_read[i]` 执行 `V`，允许每个消费者读一次。
- 最后一个读完的消费者执行 `V(empty)`，允许生产者放下一条。

【易错】不能只用一个 `full` 信号量让消费者竞争。那样一条消息只会被某一个消费者取走，不满足“所有消费者都接收一次”。

### 9.2 【题型】水果盘、座位、沙发 PV 真题

题目约束：

```text
爸爸、儿子、女儿在就餐区域；
只有一个盘子，盘中最多放一个水果；
爸爸放苹果或橘子；
儿子只吃苹果，女儿只吃橘子；
桌边有 2 个座位和 1 个沙发；
放水果或取水果的人必须先坐到空闲座位；
爸爸空闲时在沙发上睡觉；
孩子发现没有水果，需要叫醒爸爸；
爸爸按叫醒者需求放水果；
孩子取走水果后盘子变空；
要求避免死锁，并分析饥饿。
```

这题是典型“资源 + 条件同步”混合题。

#### 9.2.1 先拆资源

| 对象 | 信号量 | 初值 | 含义 |
|---|---|---|---|
| 座位 | `seat` | 2 | 可用座位数 |
| 盘子空 | `plate_empty` | 1 | 盘子是否为空 |
| 苹果 | `apple` | 0 | 盘中是否有苹果 |
| 橘子 | `orange` | 0 | 盘中是否有橘子 |
| 总请求 | `request_sem` | 0 | 未处理的孩子请求数 |
| 请求互斥 | `req_mutex` | 1 | 保护请求状态，避免两个孩子同时叫醒造成混乱 |

为了让爸爸知道该放什么，不能只用一个简单的 `request` 变量。因为儿子和女儿可能几乎同时请求，如果后来的请求覆盖前面的请求，就会丢失请求。

更稳妥的写法是记录两类请求的数量：

```c
int apple_req_count = 0;
int orange_req_count = 0;
```

用 `req_mutex` 保护它们，用 `request_sem` 表示“当前共有多少个未处理请求”。

#### 9.2.2 儿子进程

考虑到盘子容量为 1，可以把孩子“发现没水果后叫醒爸爸”的动作抽象为发出请求，然后等待对应水果信号量。儿子请求苹果时，不直接覆盖一个共享变量，而是增加苹果请求计数：

```c
son() {
  while (true) {
    P(req_mutex);
    apple_req_count++;
    V(request_sem);
    V(req_mutex);

    P(apple);

    P(seat);
    take_apple();
    V(plate_empty);
    V(seat);

    eat_apple();
  }
}
```

这份写法适合“孩子想吃时若没有水果就叫醒爸爸”的抽象版本。若题目要求先检查盘子，应补充盘子状态变量并保护检查动作。

#### 9.2.3 女儿进程

```c
daughter() {
  while (true) {
    P(req_mutex);
    orange_req_count++;
    V(request_sem);
    V(req_mutex);

    P(orange);

    P(seat);
    take_orange();
    V(plate_empty);
    V(seat);

    eat_orange();
  }
}
```

#### 9.2.4 爸爸进程

爸爸要等待孩子请求，然后先坐到座位，再等盘子空，再放对应水果。

可以写成两个请求信号量的选择式伪代码。考试中如果不允许 `select`，更推荐用“总请求信号量 + 两类请求计数”的写法。

推荐考试写法：

```c
semaphore request_sem = 0;
semaphore req_mutex = 1;
semaphore seat = 2;
semaphore plate_empty = 1;
semaphore apple = 0;
semaphore orange = 0;
int apple_req_count = 0;
int orange_req_count = 0;
```

孩子请求：

```c
// son
P(req_mutex);
apple_req_count++;
V(request_sem);
V(req_mutex);
P(apple);
P(seat);
take_apple();
V(plate_empty);
V(seat);
eat_apple();
```

```c
// daughter
P(req_mutex);
orange_req_count++;
V(request_sem);
V(req_mutex);
P(orange);
P(seat);
take_orange();
V(plate_empty);
V(seat);
eat_orange();
```

爸爸：

```c
father() {
  while (true) {
    P(request_sem);      // 在沙发上睡觉，等待叫醒

    P(req_mutex);
    int r;
    if (apple_req_count > 0) {
      apple_req_count--;
      r = APPLE;
    } else {
      orange_req_count--;
      r = ORANGE;
    }
    V(req_mutex);

    P(plate_empty);
    P(seat);
    if (r == APPLE) {
      put_apple();
      V(apple);
    } else if (r == ORANGE) {
      put_orange();
      V(orange);
    }
    V(seat);
  }
}
```

如果担心苹果请求长期压过橘子请求，可以把爸爸选择 `r` 的策略改成 FIFO 请求队列或两类请求轮转。考试题若要求“无饥饿”，这一点要写出来；若只要求“正确协作并分析是否饥饿”，说明上述偏向选择可能带来饥饿，再提出公平队列即可。

#### 9.2.5 这题的关键检查

1. 盘子容量为 1  
   用 `plate_empty=1` 控制，爸爸放水果前 `P(plate_empty)`，孩子取走后 `V(plate_empty)`。

2. 座位数量为 2  
   放或取水果前 `P(seat)`，动作完成后 `V(seat)`。

3. 爸爸睡觉与叫醒  
   用 `request_sem=0` 表示没有请求时爸爸等待。

4. 孩子只吃对应水果  
   儿子等 `apple`，女儿等 `orange`。

5. 避免死锁  
   不要让孩子占着座位等待爸爸放水果，否则可能两个孩子占满座位，爸爸没有座位放水果。

【易错】这题最容易写成：

```c
孩子先 P(seat)，发现没水果后占着座位 P(apple/orange)
```

如果儿子和女儿都占着两个座位等待，爸爸无法坐下放水果，就会死锁。

因此原则是：

```text
等待别人生产长期事件时，不要占着会被对方需要的资源。
```

#### 9.2.6 饥饿分析

是否可能饥饿，取决于请求处理是否公平。

如果爸爸总是优先处理苹果请求，且儿子请求源源不断，那么女儿的橘子请求可能长期得不到处理，就可能饥饿。

若要避免饥饿，需要增加公平机制，例如：

- 请求队列，按 FIFO 处理孩子请求。
- 儿子/女儿请求分别计数，爸爸按某种公平策略选择。
- 在同类请求连续服务一定次数后切换。

考试回答可以写：

```text
在未引入公平队列或轮转策略时，若调度长期偏向某一孩子，
另一孩子可能长期得不到所需水果，存在饥饿可能。
若请求按 FIFO 排队并逐个处理，则可避免饥饿。
```

## 10. 管程与条件变量

### 10.1 【重点】为什么引入管程

信号量和 PV 操作很灵活，但也很分散。

一个共享变量可能在很多进程里被 P/V 保护：

```text
P(mutex)
修改共享变量
V(mutex)
```

如果某个地方忘了 P，或者 P/V 顺序写错，错误很隐蔽。

管程的思想是：

```text
把共享变量和所有访问这些变量的操作集中封装在一个模块里。
外部进程不能直接访问共享变量，只能调用管程提供的过程。
```

课件对比：

| 机制 | 优点 | 缺点 |
|---|---|---|
| PV 操作 | 高效、灵活 | 分散，可读性差，正确性不易保证 |
| 管程 | 集中、可读性好、正确性易保证、易修改 | 不够灵活，效率略低 |

### 10.2 【必会】管程语义

管程有三个核心语义：

1. 管程内共享变量在外部不可见。
2. 外部只能通过管程过程访问共享变量。
3. 管程互斥进入：每次只允许一个进程在管程中执行。

【题型】真题选择：

```text
以下关于管程的叙述，错误的是？
C. 管程中 signal 操作的作用和信号量机制中的 V 操作相同
```

该项错误。

为什么？

信号量 `V` 是释放资源或唤醒等待队列中的一个进程，通常会改变信号量计数。  
管程 `signal(c)` 是对条件变量队列进行唤醒，它不表示资源数 +1，而且具体语义与 Hoare/Java/Hansen 管程有关。

应试口诀：

```text
管程 signal 是“条件变量唤醒”；
信号量 V 是“资源/事件释放通知”；
两者名字像，语义不完全相同。
```

### 10.3 【必会】Hoare 管程中的 wait/signal

课件给出 Hoare 管程语义：

`wait(c)`：

- 如果紧急等待队列非空，唤醒第一个等待者；否则释放管程互斥权。
- 执行 `wait(c)` 的进程进入条件变量 `c` 的等待队列尾部。

`signal(c)`：

- 如果 `c` 队列为空，相当于空操作，当前进程继续。
- 否则唤醒 `c` 队列第一个等待者，当前进程进入紧急等待队列。

Hoare 管程的唤醒语义可记为：

```text
Signal and urgent wait
```

也就是唤醒者让出管程，被唤醒者立即继续。

【题型】习题课模拟选择：

```text
在 Hoare 管程中，进程 P 正在使用管程，当 P 执行 signal 操作时，其含义是？
```

答案应是：

```text
Signal and urgent wait
```

### 10.4 Java 管程语义

课件也讲到 Java 类似管程的机制：

- `synchronized` 方法使用对象锁。
- `wait()` 释放锁，线程进入 wait set。
- `notify()` 从 wait set 中任选一个线程移到 entry set。
- `notifyAll()` 把 wait set 中所有线程移到 entry set。

Java 的唤醒语义更接近：

```text
Signal and continue
```

被唤醒线程不是立刻运行，而是进入入口竞争，所以通常要用 `while` 重新检查条件：

```java
while (count == 0) {
  wait();
}
```

【易错】不要用 `if` 代替 `while`。因为线程被唤醒后，条件可能已经不成立。

### 10.5 管程实现生产者-消费者

管程写法把缓冲区和操作封装起来：

```text
monitor bounded_buffer
  shared buffer, count, in, out
  condition not_full, not_empty

  put(item):
    if count == n then wait(not_full)
    insert item
    signal(not_empty)

  get():
    if count == 0 then wait(not_empty)
    remove item
    signal(not_full)
```

理解重点：

- 互斥进入由管程自动保证。
- 条件不满足时用 `wait` 放弃管程互斥权。
- 条件可能满足时用 `signal` 唤醒等待者。

### 10.6 管程实现读者-写者：为什么 start_read 要 signal(rq)

习题课问过：

```text
管程实现读者写者问题时，start_read() 中 signal(rq) 的作用是什么？
```

背景是读者-写者管程中，多个读者可以一起读。若一个读者被允许开始读，它应继续唤醒其他等待读者，让一批读者都能进入。

如果没有 `signal(rq)`，可能出现：

```text
等待写者的第一个读者被最后一个写者唤醒，
但 rq 中其他读者还要等下一次唤醒，
而且每次可能只唤醒一个。
```

因此 `start_read()` 中的 `signal(rq)` 有“读者接力唤醒读者”的作用，提高并发性，符合读读可并发。

## 11. 会合与进程高级通信

### 11.1 【重点】会合 Rendezvous

会合用于两个并发执行流在某个点握手同步。课件背景是分布式系统：共享变量、信号量、管程都偏向同一存储空间，而分布式环境中进程可能不共享内存。

会合的基本思想：

```text
调用者发起调用；
被调用任务执行 accept；
双方都到达后握手；
会合期间由被调用者代调用者执行一段代码；
会合结束后调用者继续。
```

Ada 中相关语句：

- 调用语句。
- 接受语句 `accept`。
- 选择语句 `select`。

考试定位：

- 会合是一种同步机制。
- 适合分布式或任务间同步。
- 本章期末重点仍是 PV 和管程，会合通常考概念或阅读。

### 11.2 【必会】进程通信的概念

课件定义：

```text
进程通信：进程之间的相互作用。
```

通信分为：

| 类型 | 含义 |
|---|---|
| 低级通信 | 进程互斥、进程同步，传递信息量小 |
| 高级通信 | 传递大宗信息 |

高级通信分类：

- 共享内存 vs 消息传递。
- 直接通信 vs 间接通信。
- 对称 vs 非对称。
- 缓冲 vs 非缓冲。

### 11.3 【重点】共享内存模式

共享内存模式：

```text
操作系统提供一块公共内存；
多个进程通过读写这块公共内存交换信息；
操作系统还要提供互斥同步机制。
```

优点：

- 数据复制少，效率高。

缺点：

- 需要额外同步互斥。
- 不适合完全不共享存储的环境。

### 11.4 【重点】消息传递模式

消息传递模式通过 `send/receive` 传递消息。

直接方式：

```text
进程 -> 进程
```

间接方式：

```text
进程 -> 信箱 -> 进程
```

对称直接通信：

```c
send(R, message);
receive(S, message);
```

发送者和接收者都要命名对方。

非对称直接通信：

```c
send(R, message);
receive(pid, message);
```

发送者命名接收者，接收者不指定具体发送者，而是获得发送者 pid。

### 11.5 【重点】有缓冲直接通信

有缓冲直接通信中，消息先进入接收进程的消息队列。

典型信号量：

```c
semaphore Sm = 0;        // 接收进程消息队列中的消息数
semaphore m_mutex = 1;   // 消息队列互斥
semaphore Sb = k;        // 缓冲池空缓冲数
semaphore b_mutex = 1;   // 缓冲池互斥
```

发送：

```text
根据 R 找到接收者；
申请空缓冲；
把消息复制到缓冲；
把缓冲挂入接收者消息队列；
V(Sm) 通知接收者有消息。
```

接收：

```text
P(Sm) 等待消息；
从消息队列取出缓冲；
把消息复制到接收变量；
归还空缓冲。
```

### 11.6 【必会】无缓冲直接通信

真题选择考过无缓冲直接通信。

课件结构：

发送过程 `send(R, M)`：

```text
根据 R 找到消息接收者；
V(S_m)，若接收进程等待则唤醒；
P(S_w)，等待消息传送完毕。
```

接收过程 `receive(pid, N)`：

```text
P(S_m)，等待消息到达；
消息由发送进程空间复制到接收进程空间；
V(S_w)，唤醒发送进程。
```

关键点：

- 无缓冲意味着没有中间缓冲区保存消息。
- 发送和接收都必须发生。
- 发送者需要等待接收者执行 receive 并完成复制。
- `S_m`、`S_w` 是接收进程 PCB 中的信号量。

【题型】真题选择：

```text
<根据 R 找到消息接收者> 是什么？
```

正确理解：

```text
找到接收消息进程的 PCB。
```

不是找到接收消息变量 `N` 的地址。`N` 是接收进程空间中的消息接收位置，但 OS 首先要根据接收进程名/标识找到接收进程 PCB。

优缺点：

| 优点 | 缺点 |
|---|---|
| 节省缓冲空间 | 并发性差，发送进程要等待接收进程完成复制 |

### 11.7 【重点】间接通信：信箱

信箱属于操作系统空间，进程通过信箱发送/接收消息。

信箱结构可以理解为一个循环缓冲区：

```c
typedef struct {
  int in, out;
  semaphore s1;     // 空位数，初值 k
  semaphore s2;     // 消息数，初值 0
  semaphore mutex;  // 互斥，初值 1
  message letter[k];
} mailbox;
```

发送：

```c
send_mb(mailbox *mb, message m) {
  P(mb->s1);
  P(mb->mutex);
  mb->letter[mb->in] = m;
  mb->in = (mb->in + 1) % k;
  V(mb->mutex);
  V(mb->s2);
}
```

接收：

```c
receive_mb(mailbox *mb, message *n) {
  P(mb->s2);
  P(mb->mutex);
  *n = mb->letter[mb->out];
  mb->out = (mb->out + 1) % k;
  V(mb->mutex);
  V(mb->s1);
}
```

这其实就是生产者-消费者模型在通信中的应用。

### 11.8 UNIX 管道与信号

课件讲到 UNIX 高级通信机制：

管道：

```text
pipe 是一个无名文件，有一个读描述符和一个写描述符。
父进程创建 pipe 后，子进程可以继承描述符。
```

特点：

- 基于文件系统实现。
- 与文件有统一接口。
- 常用于有亲缘关系的进程通信。

信号：

```text
一种异步通知机制。
进程可预置信号处理程序；
收到信号后，在适当时机执行处理或终止/忽略。
```

考试定位：知道管道和信号属于进程高级通信机制即可，不需要展开 UNIX 内核细节。

## 12. 易混概念总表

| 概念 | 容易误解 | 正确理解 |
|---|---|---|
| 并发 vs 并行 | 并发一定是同时执行 | 并发强调逻辑上同时推进，并行强调物理上同时执行 |
| 顺序程序可再现 | 并发程序也可再现 | 并发程序可能因交叉不同而不可再现 |
| 读共享变量 | 共享就一定冲突 | 多个只读通常不冲突，写才危险 |
| 临界资源 | 一段代码 | 一次只允许一个进程访问的资源 |
| 临界区 | 共享数据区 | 访问临界资源/共享变量的一段程序 |
| 互斥 | 所有临界区都互斥 | 互斥是相对同一组共享变量而言 |
| 进展 | 谁等待久谁先进 | 临界区空闲时应能放行某个申请者 |
| 有限等待 | 最终可能会进 | 等待的进入次数要有上界 |
| 忙等 | 等待态 | 忙等通常仍占 CPU 循环检查 |
| P 操作 | 释放资源 | P 是申请资源，申请不到则等待 |
| V 操作 | 申请资源 | V 是释放资源或通知事件 |
| 信号量为负 | 系统出错 | 负值绝对值表示等待队列长度 |
| `mutex=0` | 没人进入临界区 | 对初值 1 的互斥量，通常表示已有进程进入，无等待者 |
| `mutex=-1` | 两个进程进入临界区 | 表示一个进程等待；若只有两个进程，另一个在临界区 |
| 同步信号量初值 | 一般设 1 | 简单同步常设 0，表示事件尚未发生 |
| P/V 顺序 | 先拿 mutex 更安全 | 缓冲区题通常先 P 资源，再 P mutex，避免占锁等待 |
| 管程 signal | 等同于信号量 V | 管程 signal 是条件变量唤醒，语义不同 |
| Hoare signal | 唤醒者继续执行 | Hoare 是 signal and urgent wait |
| Java notify | 被唤醒者立即运行 | Java 中被唤醒者进 entry set，还要重新竞争锁 |
| 有缓冲通信 | 发送者必须等接收者 | 有缓冲时消息可先进队列 |
| 无缓冲通信 | 需要消息缓冲池 | 无缓冲不需要中间缓冲，但发送者等待接收完成 |
| 临界区内申请 I/O | 其他进程都不能抢 CPU | 可以抢 CPU，但不能进入相关临界区 |

## 13. 课内题型归纳与应试模板

### 13.1 【题型】判断信号量当前值

题型：

```text
信号量初值为 k，当前值为 -m，问等待进程数。
```

模板：

```text
若 value < 0，则等待队列长度为 |value|。
```

扩展判断：

```text
若这是互斥信号量且初值为 1，
value = -m 通常表示 1 个进程在临界区，m 个进程等待。
```

### 13.2 【题型】评价互斥算法

答题顺序：

```text
1. 是否可能两个进程同时进入临界区？若可能，违反互斥。
2. 临界区空闲时，是否可能没人进？若可能，违反进展。
3. 某个等待进程是否可能长期被插队？若可能，违反有限等待。
```

Hyman 题优先写互斥反例；`test_and_set` 题优先写有限等待缺陷。

### 13.3 【题型】写 PV 大题

标准答题格式：

```text
信号量定义：
  semaphore xxx = 初值；// 含义

共享变量：
  ...

进程 A：
  while (true) {
    ...
  }

进程 B：
  while (true) {
    ...
  }

正确性说明：
  互斥如何保证；
  同步顺序如何保证；
  是否死锁；
  是否饥饿。
```

注意：

- 题目要求“给出信号量变量的定义、初值及含义”时，必须写注释。
- 不要只写代码。
- 大题最后用两三句话说明正确性，会很稳。

### 13.4 【题型】临界资源 + I/O 等待

真题选择：

```text
一个正在访问临界资源的进程由于申请 I/O 而等待，它会怎样？
```

正确理解：

```text
它等待 I/O 后会让出处理机，因此其他进程可以抢占/使用 CPU；
但如果它没有释放相关临界资源，其他进程不能进入与该资源相关的临界区。
```

所以答案应是：

```text
允许其他进程抢占处理机，但不能进入与该进程相关的临界区。
```

【易错】不是“不允许任何进程抢占处理机”。等待 I/O 的进程本来就不占 CPU。

### 13.5 【题型】管程选择题

常见判断：

- 管程是同步工具：对。
- 管程每次只允许一个进程进入：对。
- 管程中 `signal` 和信号量 `V` 完全相同：错。
- Hoare 管程 `signal` 是 `Signal and urgent wait`：对。
- Java `notify` 后被唤醒线程立刻运行：错。

### 13.6 【题型】直接通信选择题

无缓冲直接通信：

```text
send(R, M):
  根据 R 找到接收进程 PCB
  V(S_m)
  P(S_w)

receive(pid, N):
  P(S_m)
  copy message
  V(S_w)
```

判断重点：

- `S_m` 和 `S_w` 是接收进程 PCB 中的信号量。
- 根据 `R` 找到的是接收进程 PCB。
- 无缓冲节省空间，但并发性差。

## 15. 本章一句话总结

第 4 章的核心不是“背几个同步问题”，而是掌握一种翻译能力：

```text
把并发约束翻译成：
哪些资源有限，
哪些操作必须互斥，
哪些事件必须先后发生，
然后用 P/V、管程或通信机制把这些约束写出来。
```

只要你能稳定完成这件事，第 4 章的选择题、简答题和 PV 大题都会变得可控。
