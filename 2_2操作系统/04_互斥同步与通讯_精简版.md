## 2. 并发错误、临界区与互斥原则

### 2.1 并发程序为什么会出错

【必背】并发程序的典型问题：多个进程执行顺序不确定，若它们访问同一共享变量，结果可能依赖具体交叉时序，所以出现“与时间有关的错误”和“不可再现性”。

考前只抓一句话：

```text
只要多个进程并发读写同一共享对象，并且没有互斥保护，就可能出现结果不确定。
```

#### 例题 2.1：并发错误判断

题目：

两个进程同时执行 `x = x + 1`，变量 `x` 初值为 0。若 `x = x + 1` 不是原子操作，最终结果一定是 2 吗？

答案：

不一定，可能是 1，也可能是 2。

解答过程：

`x = x + 1` 至少包含“读 x、加 1、写回 x”。若两个进程都先读到 0，再分别写回 1，最终结果就是 1。错误不在加法本身，而在共享变量的读改写没有被当作临界区保护。

一句话记法：

```text
并发错题看共享变量：读-改-写交叉，结果就可能和时间有关。
```

### 2.2 临界资源、临界区、临界区四段结构

| 概念 | 正确理解 | 常见错误 |
|---|---|---|
| 临界资源 | 一次只允许一个进程访问的资源 | 误认为是一段程序 |
| 临界区 | 访问临界资源或共享变量的程序段 | 误认为是共享数据区本身 |
| 互斥 | 同一组共享变量相关的临界区不能同时进入 | 误认为所有临界区之间都互斥 |

【必背】临界区四段结构：

```text
entry section      进入区：申请进入临界区
critical section   临界区：访问共享变量/临界资源
exit section       退出区：释放进入资格
remainder section  剩余区：与共享资源无关的代码
```

#### 例题 2.2：临界区是什么

题目：

所谓临界区是指什么？

答案：

临界区是访问临界资源或共享变量的一段程序。

解答过程：

题目如果问“临界资源”，回答资源；如果问“临界区”，回答程序段。临界区不是资源，也不是共享数据区，而是会访问这些对象的代码段。

易错排雷：

| 说法 | 判断 | 理由 |
|---|---|---|
| 临界区是一段程序 | 对 | 访问临界资源的代码 |
| 临界资源是一段程序 | 错 | 临界资源是资源 |
| 所有临界区之间都必须互斥 | 错 | 只有访问同一组共享对象的临界区才需要互斥 |

### 2.3 互斥算法三原则

【必背】评价互斥算法时看三件事：

| 原则 | 含义 | 反例长相 |
|---|---|---|
| 互斥 | 同一时刻最多一个进程进入相关临界区 | 两个进程同时进临界区 |
| 进展/空闲让进 | 临界区空闲且有人想进时，应允许某个申请者进入 | 明明没人用，却没人能进 |
| 有限等待 | 一个进程申请后，在有限次他人进入后应轮到它 | 一直被插队，可能饥饿 |

【会判】互斥不是唯一标准。一个算法只要违反任意一条，就不是合格的互斥算法。

#### 例题 2.3：轮流进入算法错在哪里

题目：

两个进程用变量 `turn` 严格轮流进入临界区。若 `turn=1`，但 P1 不想进入临界区，P0 想进入却必须等待。该算法违反哪条原则？

答案：

违反进展/空闲让进。

解答过程：

临界区此时空闲，且 P0 想进入，但由于 `turn` 指向 P1，P0 被阻止。它没有造成两个进程同时进入，所以不违反互斥；问题是“空闲却不让想进的人进”。

一句话记法：

```text
没人用却不让进，错在进展；有人一直插队，错在有限等待。
```

### 2.4 临界区内申请 I/O 的选择题

【易错】进程在临界区内因为申请 I/O 而阻塞时，其他进程可以抢占处理机，但不能进入与该临界资源相关的临界区。

#### 例题 2.4：临界区 + I/O 等待

题目：

一个正在访问临界资源的进程由于申请 I/O 操作而进入等待状态，此时其他进程能否抢占处理机？能否进入该临界区？

答案：

可以抢占处理机；不能进入与该临界资源相关的临界区。

解答过程：

等待 I/O 的进程不再占用 CPU，所以调度程序可以让其他进程运行。但如果该进程没有释放临界资源，对应互斥关系仍然存在，其他进程不能进入同一临界资源相关的临界区。

一句话记法：

```text
阻塞释放 CPU，不等于释放临界资源。
```

## 3. 软件互斥与硬件互斥

### 3.1 软件互斥算法定位

| 算法 | 考试定位 |
|---|---|
| 严格轮流 | 知道可能违反进展 |
| 单纯 flag | 知道可能同时进入或互相等待 |
| Peterson | 两进程互斥经典正确思路：意愿 + turn |
| Lamport 面包店 | 多进程互斥：取号，按号和进程号排序 |
| Hyman | 作业明确考反例，必须会构造 |

精简版不要求你背所有软件算法代码，重点是会用“三原则”找错。

### 3.2 Hyman 算法反例

【会写】Hyman 题不是让你泛泛说“它错”，而是要给出一个具体交叉执行，使 P0 和 P1 同时进入临界区。

#### 例题 3.1：作业三 Hyman 算法

题目：

Consider the following program:

```pascal
var blocked: array[0..1] of boolean;
    turn: 0..1;

procedure P(id: integer);
begin
  repeat
    blocked[id] := true;
    while turn <> id do
      begin
        while blocked[1-id] do
          {nothing}
        turn := id
      end;
    <Critical section>
    blocked[id] := false;
    <Remainder>
  until false
end;

begin
  blocked[0] := false;
  blocked[1] := false;
  turn := 0;
  parbegin
    P(0); P(1)
  parend;
end.
```

This is a software solution to the mutual exclusion problem proposed by Hyman. Find a counter example to demonstrate that this solution is incorrect.

答案：

可以构造一种交叉执行，使 P0 和 P1 同时进入临界区，因此该算法违反互斥性。

解答过程：

初始状态：

```text
blocked[0] = false
blocked[1] = false
turn = 0
```

一种反例执行序列：

| 步骤 | 执行进程 | 操作 | 结果 |
|---|---|---|---|
| 1 | P1 | `blocked[1] := true` | P1 表示自己想进入 |
| 2 | P1 | 判断 `turn <> 1` | 因为 `turn=0`，条件为真 |
| 3 | P1 | 判断 `blocked[0]` | 此时 `blocked[0]=false`，跳出内层等待 |
| 4 | 切换到 P0 | `blocked[0] := true` | P0 也表示自己想进入 |
| 5 | P0 | 判断 `turn <> 0` | 因为 `turn=0`，条件为假，P0 进入临界区 |
| 6 | 切回 P1 | 执行 `turn := 1` | P1 把 turn 改为 1 |
| 7 | P1 | 再判断 `turn <> 1` | 条件为假，P1 也进入临界区 |

结论：

P0 和 P1 同时处于 `<Critical section>`，所以算法不满足互斥。

应试写法：

```text
给出初值 -> 写交叉步骤 -> 指出 P0、P1 同时进临界区 -> 违反互斥性。
```

易错排雷：

不要只写“该算法可能死锁”或“可能饥饿”。作业这题最直接的反例是破坏互斥，也就是两个进程都进临界区。

### 3.3 test_and_set、swap 与忙等

【必背】`test_and_set` 是硬件原子指令：读出旧值，同时把目标设为 1/true。

典型写法：

```c
while (test_and_set(&lock))
  ;              // busy waiting

// critical section

lock = 0;
```

含义：

| `lock` 原值 | `test_and_set` 返回 | 新 `lock` | 进程结果 |
|---|---:|---:|---|
| 0 | 0 | 1 | 获得锁，进入临界区 |
| 1 | 1 | 1 | 继续忙等 |

【易错】`test_and_set` 能保证互斥，但普通写法不保证有限等待。

#### 例题 3.2：test_and_set 为什么不满足有限等待

题目：

用 `test_and_set` 实现互斥时，说明何种状况下该算法不满足有限等待特性。如何改进？

答案：

普通 `test_and_set` 在锁释放后让所有等待进程自由竞争，某个进程可能长期抢不到锁，因此不满足有限等待。改进思路是增加等待数组，在退出临界区时按循环顺序指定一个等待进程进入。

解答过程：

普通释放写法：

```c
lock = 0;
```

释放后，所有正在忙等的进程同时竞争。若 P1 每次都比 P2 先执行到 `test_and_set`，P1 可以反复进入临界区，而 P2 一直失败。P2 等待次数没有上界，违反有限等待。

改进核心：

```text
不是“谁抢到算谁的”，而是“退出者按顺序交给某个等待者”。
```

可以维护：

```c
boolean waiting[n];
boolean lock;
```

退出临界区时，从当前进程的下一个编号开始循环查找等待者，若找到等待进程 j，则令 `waiting[j] = false`，让 j 结束等待；若没有等待者，再把 `lock = false`。

一句话记法：

```text
普通 test_and_set：大家抢，可能饿死；
有限等待改进：排队交棒，等待有界。
```

### 3.4 关中断、自旋锁、swap 的选择题定位

| 方法 | 关键点 | 易错点 |
|---|---|---|
| 关中断 | 单 CPU、短内核临界区可用 | 多 CPU 上关一个 CPU 的中断不能阻止其他 CPU |
| 自旋锁 | 忙等，不让出 CPU | 临界区必须短，否则浪费处理机 |
| `swap` | 硬件原子交换，可实现互斥 | 和 `test_and_set` 一样可能忙等 |

#### 例题 3.3：关中断适用范围

题目：

为什么关中断不适合作为用户进程的一般互斥方法，也不适合作为多处理机系统的通用互斥方法？

答案：

关中断影响系统响应，且用户进程不应随意控制中断；在多处理机系统中，关闭一个 CPU 的中断不能阻止其他 CPU 上的进程访问共享内存。

解答过程：

关中断的本质是防止当前 CPU 被中断打断。如果只有一个 CPU，并且临界区很短，可以用于内核内部的短临界区。但多 CPU 环境中，其他 CPU 仍可能并发执行，所以需要总线锁、原子指令或更高层同步机制。

## 4. 信号量与 P/V 操作

### 4.1 信号量 value 的含义

【必背】信号量不只是一个整数，它通常包含：

```text
value：资源数或等待情况
queue：等待该信号量的进程队列
```

考试解释 `S.value` 时按表判断：

| 当前值 | 含义 |
|---:|---|
| `S > 0` | 还有 `S` 个可用资源 |
| `S = 0` | 没有可用资源，且按经典定义没有进程等待 |
| `S < 0` | 有 `|S|` 个进程在该信号量队列中等待 |

【易错】负数不是系统错误，负数的绝对值表示等待进程数。

#### 例题 4.1：作业三信号量取值

题目：

对于两个并发进程，设互斥信号量为 `mutex`，初值为 1，若 `mutex = -1`，则____。

A. 表示没有进程进入临界区  
B. 表示有一个进程进入临界区  
C. 表示有一个进程进入临界区，另一个进程等待进入  
D. 表示有两个进程进入临界区

答案：

C。

解答过程：

`mutex` 初值为 1，表示临界区最多允许 1 个进程进入。当前 `mutex=-1`，负数绝对值为 1，表示有 1 个进程正在等待该信号量。题目总共只有两个并发进程，所以另一个进程必然已经进入临界区。

选项级排雷：

| 选项 | 判断 | 原因 |
|---|---|---|
| A | 错 | `mutex=-1` 已经说明有人等待，不可能没人进入 |
| B | 不完整 | 只说进入者，漏掉等待者 |
| C | 对 | 一个在临界区，一个在等待 |
| D | 错 | 互斥信号量不会允许两个同时进入临界区 |

一句话记法：

```text
互斥量初值 1，value = -m：通常 1 个在里面，m 个在外面等。
```

### 4.2 P/V 操作语义

【必背】经典 P/V 定义：

```text
P(S):
  S.value--;
  if S.value < 0:
    当前进程阻塞，进入 S.queue

V(S):
  S.value++;
  if S.value <= 0:
    从 S.queue 唤醒一个等待进程
```

常用初值：

| 场景 | 信号量 | 初值 | 含义 |
|---|---|---:|---|
| 互斥访问一个对象 | `mutex` | 1 | 谁先 P 到谁进入 |
| 等待某事件发生 | `S` | 0 | 一开始事件未发生 |
| N 个空缓冲区 | `empty` | N | 可放入的位置数 |
| 已有产品数 | `full` | 0 | 一开始没有产品 |
| k 个同类资源 | `resource` | k | 同时最多 k 个进程使用 |

#### 例题 4.2：同步量为什么常设 0

题目：

若 P2 必须等待 P1 完成某个操作 A 后才能执行操作 B，应如何设置信号量？

答案：

设同步信号量 `S = 0`。P2 执行 B 前 `P(S)`，P1 执行完 A 后 `V(S)`。

解答过程：

`S=0` 表示事件 A 一开始还没发生。P2 先执行 `P(S)` 会阻塞；只有 P1 完成 A 后执行 `V(S)`，P2 才能继续。

模板：

```c
semaphore S = 0;

P1:
  A;
  V(S);

P2:
  P(S);
  B;
```

一句话记法：

```text
等事件：信号量初值 0；事件发生者 V，等待事件者 P。
```

### 4.3 PV 题通用建模步骤

【会写】所有 PV 大题先写这六步：

1. 找并发进程：哪些角色在同时运行。
2. 找共享资源：缓冲区、盘子、座位、筷子、文件等。
3. 找互斥：哪些对象一次只允许一个进程操作。
4. 找同步顺序：谁必须等谁先发生。
5. 定义信号量：写清初值和含义。
6. 写伪代码：检查 P/V 顺序，避免占着互斥锁睡眠。

【易错】缓冲区类题常用顺序：

```text
生产者：P(empty) -> P(mutex) -> 放入 -> V(mutex) -> V(full)
消费者：P(full)  -> P(mutex) -> 取出 -> V(mutex) -> V(empty)
```

不要写成：

```text
P(mutex) -> P(empty/full)
```

因为可能占着互斥锁等待资源，导致对方进不了缓冲区释放资源。

## 5. 生产者-消费者与基础 PV 模板

### 5.1 标准生产者-消费者

【会写】题目只要出现“有限缓冲区、生产者放、消费者取、满不能放、空不能取”，优先套这个模板。

信号量：

```c
semaphore empty = N;   // 空缓冲区数量
semaphore full = 0;    // 满缓冲区数量/产品数量
semaphore mutex = 1;   // 互斥访问缓冲区
```

伪代码：

```c
Producer:
while (1) {
  item = produce();
  P(empty);
  P(mutex);
  put(item);
  V(mutex);
  V(full);
}

Consumer:
while (1) {
  P(full);
  P(mutex);
  item = get();
  V(mutex);
  V(empty);
  consume(item);
}
```

#### 例题 5.1：为什么 P(empty) 要在 P(mutex) 前面

题目：

生产者-消费者问题中，生产者能否先执行 `P(mutex)`，再执行 `P(empty)`？

答案：

一般不能这样写，可能导致死锁。

解答过程：

假设缓冲区满，生产者先拿到 `mutex`，然后执行 `P(empty)` 阻塞。此时消费者需要进入缓冲区取产品并 `V(empty)`，但消费者执行 `P(mutex)` 时会被挡住，因为 `mutex` 被生产者占着。于是生产者等空位，消费者等锁，形成死锁。

一句话记法：

```text
先等资源数量，再进互斥区；不要拿着 mutex 睡觉。
```

### 5.2 “互斥”和“同步”的区分

| 问题 | 本质 | 信号量初值 |
|---|---|---|
| 多进程不能同时改同一个缓冲区 | 互斥 | 1 |
| 消费者必须等产品出现 | 同步 | `full=0` |
| 生产者必须等空位出现 | 同步/资源计数 | `empty=N` |
| P2 必须等 P1 的 A 完成 | 同步 | 0 |

#### 例题 5.2：互斥还是同步

题目：

“两个进程不能同时访问打印机”和“消费者必须等生产者放入产品后才能取走产品”分别属于互斥还是同步？

答案：

前者是互斥，后者是同步。

解答过程：

打印机一次只允许一个进程使用，重点是“能不能一起做”。消费者等待生产者，重点是“谁先谁后”。互斥处理共享资源排他访问，同步处理事件顺序。

一句话记法：

```text
不能一起：互斥；必须先后：同步。
```

## 6. 作业三：多生产者、n 个消费者、每条消息都读一次

这是第 4 章最值得反复看的作业题。它看起来像生产者-消费者，但不能直接套普通 `full`。

### 6.1 原题

题目：

设系统有多个生产者进程向 1 个缓冲区不断发消息，n 个消费者进程从该缓冲区取消息。该缓冲区大小只能存放 1 条消息，并且对每条放入缓冲区的消息，所有消费者必须都接收 1 次，生产者才可以继续向缓冲区发消息。

问题：用信号量和 P、V 操作完成生产者进程和消费者进程发、收消息的正确描述。

要求：

1. 给出信号量变量的定义、初值及其含义。
2. 实现对缓冲区及消息操作的互斥与同步。
3. 用类 C 语言伪代码描述。

### 6.2 考点拆解

【会写】这题的关键约束有三条：

| 约束 | 需要什么 |
|---|---|
| 缓冲区只能放 1 条消息 | `empty = 1` |
| 多个生产者不能同时写缓冲区 | 写缓冲互斥，或由 `empty` 保证一次只允许一个生产者 |
| 每条消息必须被 n 个消费者各读 1 次 | 每个消费者一个 `can_read[i]`，不能用一个 `full` 竞争 |
| 最后一个消费者读完，生产者才能放下一条 | `read_count` + `mutex`，最后一个 `V(empty)` |

【易错】不能只设一个 `full`。普通 `full` 会让消费者竞争消息，结果是一条消息只被一个消费者拿走，不满足“所有消费者都接收 1 次”。

### 6.3 信号量定义

设消费者编号为 `0..n-1`。

```c
semaphore empty = 1;             // 缓冲区为空，可放入新消息
semaphore can_read[n] = {0};     // can_read[i] 表示消费者 i 可以读取当前消息
semaphore mutex = 1;             // 保护 read_count

message buffer;                  // 大小为 1 的消息缓冲区
int read_count = 0;              // 当前消息已被多少个消费者读过
```

如果题目特别强调“缓冲区访问也必须互斥”，可以加：

```c
semaphore buf_mutex = 1;         // 保护对 buffer 的读写
```

多数考试写法中，生产者在 `empty=0` 时不会写新消息，消费者只是读取同一条消息，因此核心互斥是保护 `read_count`。为了稳妥，下面给出带 `buf_mutex` 的版本。

### 6.4 正确伪代码

生产者：

```c
Producer:
while (1) {
  msg = produce();

  P(empty);              // 等上一条消息被所有消费者读完
  P(buf_mutex);
  buffer = msg;          // 放入新消息
  V(buf_mutex);

  P(mutex);
  read_count = 0;        // 新消息尚未被任何消费者读
  V(mutex);

  for (int i = 0; i < n; i++) {
    V(can_read[i]);      // 通知每个消费者：你可以读一次
  }
}
```

消费者 i：

```c
Consumer(i):
while (1) {
  P(can_read[i]);        // 等待当前消息对自己可读

  P(buf_mutex);
  msg = buffer;          // 复制当前消息
  V(buf_mutex);

  consume(msg);

  P(mutex);
  read_count++;
  if (read_count == n) {
    V(empty);            // 最后一个读完者释放缓冲区
  }
  V(mutex);
}
```

### 6.5 解答过程

1. 生产者先 `P(empty)`，保证缓冲区里没有上一条未完成的消息。
2. 放入消息后，对每个消费者执行一次 `V(can_read[i])`，保证每个消费者都获得一次读取资格。
3. 消费者 i 只能在 `P(can_read[i])` 成功后读当前消息，因此不会重复读同一条，也不会漏读。
4. 每个消费者读完后累加 `read_count`。
5. 最后一个消费者使 `read_count == n`，执行 `V(empty)`，允许生产者放下一条消息。

一句话记法：

```text
一条消息给所有人看：不能让消费者抢 full，要给每个消费者一张入场券。
```

## 7. 读者-写者与哲学家进餐

### 7.1 读者-写者问题

【会写】读者-写者的核心约束：

| 同时访问关系 | 是否允许 |
|---|---|
| 读者 + 读者 | 允许 |
| 读者 + 写者 | 不允许 |
| 写者 + 写者 | 不允许 |

常用信号量：

```c
semaphore rw = 1;       // 控制对共享文件/数据的访问
semaphore mutex = 1;    // 保护 readcount
int readcount = 0;
```

读者优先基本模板：

```c
Reader:
while (1) {
  P(mutex);
  readcount++;
  if (readcount == 1) P(rw);   // 第一个读者挡住写者
  V(mutex);

  read();

  P(mutex);
  readcount--;
  if (readcount == 0) V(rw);   // 最后一个读者释放写者
  V(mutex);
}
```

写者：

```c
Writer:
while (1) {
  P(rw);
  write();
  V(rw);
}
```

#### 例题 7.1：第一个读者和最后一个读者做什么

题目：

读者-写者问题中，为什么只有第一个读者执行 `P(rw)`，最后一个读者执行 `V(rw)`？

答案：

因为多个读者可以并发读。第一个读者负责阻止写者进入，最后一个读者负责在所有读者离开后释放写者。

解答过程：

如果每个读者都 `P(rw)`，读者之间也会互斥，违背“读读并发”。因此读者群体作为整体占用 `rw`：第一个读者进入时锁住写者，最后一个读者离开时释放写者。

易错排雷：

读者优先算法可能导致写者饥饿：只要读者不断到来，写者可能长期拿不到 `rw`。

### 7.2 哲学家进餐问题

【重点】哲学家题主要考“为什么可能死锁”和“如何避免死锁”，不一定要求死背完整代码。

死锁发生方式：

```text
5 个哲学家都先拿左筷子
-> 每人再等右筷子
-> 每人都占有一个资源并等待另一个资源
-> 循环等待
```

防死锁常用方法：

| 方法 | 思路 |
|---|---|
| 至多允许 4 个哲学家同时拿筷子 | 保证至少有一个人能拿到两只 |
| 资源有序申请 | 所有人按筷子编号从小到大拿，破坏循环等待 |
| 奇偶不同顺序 | 奇数先左后右，偶数先右后左，破坏环形等待 |

#### 例题 7.2：哲学家为什么会死锁

题目：

若每个哲学家都先拿左边筷子，再拿右边筷子，为什么可能死锁？

答案：

因为可能出现每个哲学家都拿到左筷子，同时等待右筷子的情况，形成循环等待。

解答过程：

筷子是临界资源，每只筷子一次只能被一个哲学家持有。若所有人都占有一只筷子并等待下一只，就满足“占有并等待”和“循环等待”，没有人能继续吃饭，也没有人释放筷子。

一句话记法：

```text
每人拿一半，谁也吃不上。
```

## 8. 水果盘/座位/沙发类 PV 真题

这类题是第 4 章大题的典型形式：故事复杂，但本质仍是“资源数量 + 同步事件 + 互斥保护”。

### 8.1 真题型题目

题目：

桌上有一个盘子，盘中最多只能放一个水果。爸爸可以放苹果或橘子，儿子只吃苹果，女儿只吃橘子。桌旁有 2 个座位，放水果或取水果的人必须先坐到空闲座位。爸爸平时睡在沙发上，孩子发现没有自己需要的水果时叫醒爸爸，爸爸按需求放水果。用信号量和 P/V 操作描述爸爸、儿子、女儿的同步互斥关系，并说明是否可能出现死锁或饥饿。

### 8.2 建模表

| 对象/约束 | 信号量 | 初值 | 含义 |
|---|---|---:|---|
| 盘子容量为 1 | `plate_empty` | 1 | 盘子为空，可以放水果 |
| 苹果可取 | `apple` | 0 | 盘中有苹果 |
| 橘子可取 | `orange` | 0 | 盘中有橘子 |
| 两个座位 | `seat` | 2 | 可用座位数 |
| 儿子请求苹果 | `req_apple` | 0 | 有苹果请求 |
| 女儿请求橘子 | `req_orange` | 0 | 有橘子请求 |
| 请求计数互斥 | `mutex` | 1 | 保护请求计数 |

若不允许 `select` 等待两个请求，可以加请求总数：

```c
semaphore request = 0;  // 总请求数
int apple_req_count = 0;
int orange_req_count = 0;
```

### 8.3 推荐写法：避免孩子占着座位等待

【易错】孩子不能先占座位，再发现没有水果，然后占着座位等爸爸放水果。否则儿子和女儿可能占满两个座位等待，爸爸没有座位放水果，形成死锁。

正确思路：

```text
孩子没有水果 -> 先发请求并等待水果
水果出现后 -> 再坐下取水果
爸爸收到请求 -> 坐下放水果 -> 离座
```

儿子：

```c
Son:
while (1) {
  P(mutex);
  apple_req_count++;
  V(mutex);
  V(request);            // 叫醒爸爸：需要苹果

  P(apple);              // 等爸爸放苹果
  P(seat);               // 有水果后再坐下取
  take_apple();
  V(plate_empty);        // 盘子变空
  V(seat);

  eat_apple();
}
```

女儿：

```c
Daughter:
while (1) {
  P(mutex);
  orange_req_count++;
  V(mutex);
  V(request);            // 叫醒爸爸：需要橘子

  P(orange);
  P(seat);
  take_orange();
  V(plate_empty);
  V(seat);

  eat_orange();
}
```

爸爸：

```c
Father:
while (1) {
  P(request);            // 睡觉，直到有请求

  P(mutex);
  if (apple_req_count > 0) {
    apple_req_count--;
    V(mutex);

    P(plate_empty);
    P(seat);
    put_apple();
    V(seat);
    V(apple);
  } else if (orange_req_count > 0) {
    orange_req_count--;
    V(mutex);

    P(plate_empty);
    P(seat);
    put_orange();
    V(seat);
    V(orange);
  } else {
    V(mutex);
  }
}
```

### 8.4 解答过程

1. `plate_empty=1` 保证盘子最多一个水果。
2. `apple=0`、`orange=0` 表示开始没有可取水果。
3. `seat=2` 限制桌旁最多两人坐下。
4. 孩子用请求信号量叫醒爸爸，再等待对应水果。
5. 爸爸收到请求后，先等盘子空，再等座位，放对应水果。
6. 孩子取走水果后执行 `V(plate_empty)`，允许下一次放水果。

死锁分析：

错误写法中，如果孩子先 `P(seat)`，再等待水果，则两个孩子可能占满座位等待，爸爸无法 `P(seat)` 放水果，死锁。推荐写法中，孩子等到水果后才占座取水果，不占着座位等待爸爸，因此避免该死锁。

饥饿分析：

如果爸爸总是优先处理苹果请求，橘子请求可能长期得不到满足，女儿可能饥饿。若请求按公平队列或轮转策略处理，则可避免饥饿。

一句话记法：

```text
座位是短资源：坐下只做放/取，不要坐着等事件。
```

## 9. 管程与条件变量

### 9.1 管程必背结论

【必背】管程把共享变量和访问共享变量的过程封装在一起，并保证每次只有一个进程在管程中执行。

管程包含：

```text
共享变量
操作这些变量的过程
条件变量 condition
wait/signal 操作
```

管程的三个核心语义：

1. 管程内共享变量外部不可直接访问。
2. 外部只能调用管程提供的过程。
3. 管程互斥进入：一次只允许一个进程在管程内执行。

### 9.2 wait/signal 与 P/V 的区别

| 操作 | 作用对象 | 核心含义 |
|---|---|---|
| 信号量 `P(S)` | 信号量 S | 申请资源/等待事件 |
| 信号量 `V(S)` | 信号量 S | 释放资源/通知事件 |
| 管程 `wait(c)` | 条件变量 c | 条件不满足，释放管程并等待 |
| 管程 `signal(c)` | 条件变量 c | 唤醒等待在 c 上的进程 |

【易错】管程 `signal` 不等于信号量 `V`。`V` 会改变信号量计数；管程 `signal` 是对条件变量等待队列的唤醒，若无人等待，通常没有“资源累积”效果。

#### 例题 9.1：管程选择题

题目：

以下关于管程的叙述，错误的是？

A. 管程是一种同步机制  
B. 管程每次只允许一个进程在其中执行  
C. 管程中 `signal` 操作的作用和信号量机制中的 `V` 操作完全相同  
D. 管程可以把共享变量和访问共享变量的过程封装起来

答案：

C。

解答过程：

A 对，管程用于同步互斥。B 对，管程具有互斥进入特性。D 对，封装是管程的重要特点。C 错，因为管程 `signal` 是条件变量唤醒，不是信号量值加一，也不一定累积资源。

一句话记法：

```text
V 是资源/事件通知；signal 是条件队列唤醒。
```

### 9.3 Hoare 管程：Signal and urgent wait

【重点】Hoare 管程语义中，执行 `signal(c)` 后：

```text
若 c 上有等待者：
  signal 者进入紧急等待队列
  被 signal 的进程立即恢复并获得管程执行权
```

#### 例题 9.2：Hoare 管程 signal

题目：

在 Hoare 管程中，进程 P 正在管程内执行。当 P 执行 `signal(c)` 且条件变量 c 上有等待进程 Q 时，接下来谁继续在管程中执行？

答案：

Q 立即继续在管程中执行，P 进入紧急等待队列。

解答过程：

Hoare 管程强调 `Signal and urgent wait`。被唤醒者立即获得管程控制权，发出 signal 的进程让出管程并进入紧急队列，等待之后恢复。

易错排雷：

不要把它理解成“P 继续运行，Q 只是变为就绪”。那更接近某些 Mesa/Java 风格语义，不是 Hoare 管程的经典考点。

### 9.4 管程读者-写者：为什么读者要接力 signal

#### 例题 9.3：start_read 中 signal(rq) 的作用

题目：

用管程实现读者-写者问题时，`start_read()` 中为什么可能需要执行 `signal(rq)`？

答案：

为了让等待读者能够接力唤醒其他读者，使多个读者可以一起读，提高并发性。

解答过程：

读者-写者问题允许读读并发。若一个读者被允许开始读，但没有继续唤醒其他等待读者，就可能一次只放行一个读者，降低并发性。`signal(rq)` 的作用是读者接力唤醒读者，让一批读者一起进入读状态。

一句话记法：

```text
读读可并发，所以读者醒来后要继续叫醒读者。
```

## 10. 会合与进程高级通信

### 10.1 通信方式总表

| 分类 | 含义 | 例子/关键词 |
|---|---|---|
| 共享存储 | 进程通过共享内存交换数据 | 速度快，但需要同步互斥 |
| 消息传递 | 通过 send/receive 交换消息 | 适合分布式或不共享内存 |
| 直接通信 | 发送者直接指定接收进程 | `send(P, msg)` |
| 间接通信 | 通过邮箱/信箱交换 | mailbox |
| 有缓冲通信 | 消息先进入队列 | 发送者可不等接收者立即接收 |
| 无缓冲通信 | 没有中间缓冲，发送接收会合 | 发送者等待接收完成 |
| 管道 | 字节流通信，常见于父子进程 | pipe |
| 信号 | 异步通知事件 | signal |

【会判】有缓冲提高并发性但占空间；无缓冲节省空间但并发性差，需要发送者和接收者配合。

### 10.2 无缓冲直接通信

【必背】无缓冲直接通信中没有中间缓冲池，发送者要等接收者完成消息传送。

典型过程：

```text
Send(R, M):
  根据 R 找到接收进程 PCB
  V(S_m)       // 通知消息到达
  P(S_w)       // 等待消息传送完毕

Receive(N):
  P(S_m)       // 等待消息到达
  把消息复制到 N
  V(S_w)       // 通知发送者传送完成
```

其中：

```text
S_m：接收进程 PCB 中的“消息到达”信号量
S_w：接收进程 PCB 中的“发送者等待传送完成”信号量
```

#### 例题 10.1：无缓冲直接通信选择题

题目：

无缓冲直接通信中，`Send(R, M)` 根据接收者标识 R 找到的是什么？`S_m` 和 `S_w` 通常位于哪里？

答案：

找到接收进程的 PCB；`S_m` 和 `S_w` 通常位于接收进程 PCB 中。

解答过程：

R 是接收进程标识，不是接收变量 N 的地址，也不是一个消息缓冲池地址。无缓冲直接通信不设置中间消息缓冲区，因此发送与接收需要通过接收进程 PCB 中的同步信号量进行会合。

选项级排雷：

| 说法 | 判断 | 原因 |
|---|---|---|
| 根据 R 找接收进程 PCB | 对 | R 标识接收进程 |
| 根据 R 找消息变量 N 地址 | 错 | N 是 Receive 端用于接收的变量 |
| 无缓冲通信需要消息缓冲池 | 错 | 无缓冲没有中间缓冲 |
| 发送者执行后一定立即返回 | 错 | 发送者要等接收完成 |

一句话记法：

```text
无缓冲直接通信：找接收者 PCB，发送接收要会合。
```

### 10.3 直接通信 vs 间接通信

#### 例题 10.2：直接还是间接

题目：

进程 P 执行 `send(Q, msg)`，直接把消息发送给进程 Q；进程 R 执行 `receive(mailbox, msg)`，从某个信箱接收消息。两者分别属于什么通信方式？

答案：

前者是直接通信，后者是间接通信。

解答过程：

直接通信的特点是发送或接收操作中直接出现对方进程标识。间接通信通过邮箱、信箱、端口等中介对象交换消息，不直接指定对方进程。

## 11. 题型模板汇总

### 11.1 判断信号量当前值

题型：

```text
信号量初值为 k，当前 value = -m，问有多少进程等待？
```

模板答案：

```text
等待进程数 = m。
如果这是初值为 1 的互斥信号量，通常还意味着 1 个进程在临界区。
```

### 11.2 评价互斥算法

题型：

```text
给一段互斥算法，问是否正确。
```

答题步骤：

1. 查互斥：能否构造两个进程同时进入临界区。
2. 查进展：临界区空闲时，是否可能没人能进。
3. 查有限等待：某进程是否可能无限期被插队。

常见结论：

| 算法现象 | 违反 |
|---|---|
| 两个进程同时进入临界区 | 互斥 |
| 空闲时想进者进不去 | 进展 |
| 某进程一直抢不到 | 有限等待 |

### 11.3 写 PV 大题

标准答题格式：

```text
信号量定义：
  semaphore X = 初值; // 含义
  semaphore Y = 初值; // 含义

进程 A:
  ...

进程 B:
  ...

正确性说明：
  互斥如何保证；
  同步顺序如何保证；
  是否可能死锁/饥饿。
```

检查清单：

- 是否写了每个信号量的初值和含义。
- 互斥量是否初值 1。
- 同步量是否按事件初始状态设置，常见为 0。
- 是否先 P 资源数量，再 P 互斥量。
- 是否所有路径都能释放已经获得的资源。
- 是否存在“占座/占锁等待另一事件”的死锁风险。

### 11.4 管程选择题

必判结论：

| 说法 | 判断 |
|---|---|
| 管程是同步互斥机制 | 对 |
| 管程封装共享变量和操作过程 | 对 |
| 管程每次只允许一个进程在其中执行 | 对 |
| 管程 `signal` 与信号量 `V` 完全相同 | 错 |
| Hoare 管程 signal 后被唤醒者立即执行 | 对 |

### 11.5 通信选择题

必判结论：

| 说法 | 判断 |
|---|---|
| 直接通信要指定通信对方 | 对 |
| 间接通信通过邮箱/信箱 | 对 |
| 有缓冲通信中消息可以先进入队列 | 对 |
| 无缓冲通信需要发送者和接收者会合 | 对 |
| 无缓冲通信一定需要消息缓冲池 | 错 |
| 无缓冲直接通信中 `S_m/S_w` 在接收进程 PCB | 对 |

## 12. 易错判断总表

| 易错说法 | 正确判断 |
|---|---|
| 临界资源是一段程序 | 错，临界资源是一次只允许一个进程访问的资源 |
| 临界区是共享数据区 | 错，临界区是访问共享资源的程序段 |
| 所有临界区之间都互斥 | 错，互斥针对同一组共享对象 |
| 互斥算法只要保证一次一个进程进入就正确 | 错，还要满足进展和有限等待 |
| 严格轮流一定是好算法 | 错，可能违反进展 |
| Hyman 算法错误是因为一定死锁 | 不准确，作业重点是可构造同时进入临界区 |
| `test_and_set` 一定满足有限等待 | 错，普通写法可能让等待者一直抢不到 |
| 自旋锁适合长临界区 | 错，长时间忙等浪费 CPU |
| 信号量为负表示系统出错 | 错，负数绝对值表示等待进程数 |
| 互斥信号量初值通常为 0 | 错，互斥量通常初值 1 |
| 同步信号量初值通常为 1 | 错，等待事件发生常设 0 |
| PV 题先 P(mutex) 再 P(empty/full) 更安全 | 错，可能占锁等待导致死锁 |
| 多消费者都读同一条消息可以用一个 `full` | 错，一个 `full` 会让消费者竞争，不保证每人读一次 |
| 读者-写者中读者之间也必须互斥 | 错，读读可并发 |
| 管程 `signal` 等于信号量 `V` | 错，前者是条件变量唤醒，后者是资源/事件释放 |
| 无缓冲直接通信要找消息缓冲区 | 错，要找接收进程 PCB |
| 进程在临界区阻塞 I/O 后，其他进程不能抢 CPU | 错，可以抢 CPU，但不能进入相关临界区 |
| 孩子可以占着座位等待爸爸放水果 | 危险，可能导致爸爸无座位放水果而死锁 |

### 13.2 最容易救分的模板

生产者-消费者：

```c
P(empty); P(mutex); put(); V(mutex); V(full);
```

读者-写者：

```c
第一个读者 P(rw)，最后一个读者 V(rw)，写者 P(rw)/V(rw)。
```

```c
semaphore S = 0;
等待者: P(S);
```

```c
semaphore mutex = 1;
P(mutex);
// critical section
V(mutex);
```

```text
empty=1 + can_read[i]=0 + read_count + 最后一个消费者 V(empty)
```

### 13.3 本章最后压缩版

1. 临界区是程序段，临界资源是资源。
2. 互斥算法看三原则：互斥、进展、有限等待。
3. Hyman 会破坏互斥，要会写交叉执行。
4. `test_and_set` 原子、忙等、普通写法不保证有限等待。
5. 信号量为负，绝对值是等待进程数。
6. 互斥量初值 1，同步量常初值 0。
7. PV 大题先找资源数，再找互斥，再找同步顺序。
8. 缓冲区题先 P(empty/full)，再 P(mutex)。
9. 作业三多消费者题不能用一个 `full`，必须让每个消费者各读一次。
10. 管程 `signal` 不等于信号量 `V`。
11. 无缓冲直接通信找接收进程 PCB，没有中间缓冲池。
