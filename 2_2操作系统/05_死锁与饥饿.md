## 从 0 理解

死锁像两个人在窄桥两端相遇，双方都占着自己这一边，又都等对方后退，结果谁也动不了。饥饿则像排队时规则总偏向别人，你不是被别人“环形等待”卡死，而是长期轮不到。

死锁一定是无限等待，但无限等待不一定都是死锁；饥饿可能发生在调度、资源分配或同步策略中。

## 核心机制/算法/公式

### Coffman 四条件

| 条件 | 含义 | 常见破坏方法 |
|---|---|---|
| 互斥 | 资源一次只能给一个进程 | 对某些资源改为共享或虚拟化 |
| 不可剥夺 | 资源不能被强行抢走 | 允许资源剥夺或回滚 |
| 保持并申请 | 占有资源时继续申请新资源 | 一次性申请，或申请失败释放已有资源 |
| 循环等待 | 存在进程资源等待环 | 资源有序分配 |

采用有序分配策略预防死锁，主要破坏循环等待条件。

### 银行家算法

核心思想：不是“现在有资源就一定给”，而是“给完以后系统仍然能找到一个安全完成顺序才给”。

数据结构：

- `Available`：当前可用资源。
- `Max`：每个进程最大需求。
- `Allocation`：已分配资源。
- `Need = Max - Allocation`：还需要多少资源。

请求处理：

1. 检查 `Request[i] <= Need[i]`，否则请求超过声明最大需求，错误。
2. 检查 `Request[i] <= Available`，否则资源不足，等待。
3. 试探分配：
   - `Available -= Request[i]`
   - `Allocation[i] += Request[i]`
   - `Need[i] -= Request[i]`
4. 执行安全性检测。
5. 若安全则确认；否则回滚，让进程等待。

安全性检测：

1. `Work = Available`，所有 `Finish=false`。
2. 找一个 `Finish=false` 且 `Need[i] <= Work` 的进程。
3. 假设它完成，`Work += Allocation[i]`，`Finish[i]=true`。
4. 重复直到找不到。
5. 若所有 `Finish=true`，系统安全；否则不安全。

### 作业银行家算法例题

已知 `Available=(1,6,2,3)`，安全检测可得序列：

```text
P0 -> P3 -> P4 -> P1 -> P2
```

所以当前安全。

若 P2 请求 `(1,2,2,2)`：

- 请求不超过 `Need[2]=(2,3,5,6)`。
- 请求不超过 `Available=(1,6,2,3)`。
- 但试探分配后 `Available=(0,4,0,1)`，找不到可完成进程。

结论：不能分配，因为分配后系统进入不安全状态。

### 死锁检测算法

检测算法和安全性算法很像，但使用的是当前 `Request`，不是最大 `Need`。如果某进程当前请求可被 `Work` 满足，就假设它能完成并释放资源。最后 `Finish=false` 的进程就是可能参与死锁的进程。

## 易错点与区分表

| 易错点 | 正确理解 |
|---|---|
| 不安全一定死锁 | 不安全不等于已经死锁，只是存在进入死锁的风险 |
| 死锁一定包括所有进程 | 死锁进程是当前进程集合的一个子集 |
| 资源有序分配破坏保持申请 | 主要破坏循环等待 |
| 饥饿就是死锁 | 饥饿不一定处于等待状态，也不一定有等待环 |
| 银行家算法只看 Available | 必须试探分配后看是否存在安全序列 |

## 题型与解析

### 真题题型：死锁与饥饿异同

相同点：都可能导致进程长期不能向前推进，都与资源竞争或调度策略有关。

不同点：

- 死锁是一组进程相互等待对方占有的资源，有明确的等待环，可检测。
- 饥饿是某进程长期得不到资源或服务，可能是优先级、调度策略或同步策略不公平导致，不一定形成等待环。

### 作业题：组合资源死锁静态分析

资源 A-G，P1、P2、P3 的申请释放序列可画占有等待图。即使图中出现环，也要检查环是否可能同时成立：

- 如果一个环要求同一进程同时处于两个不同执行位置，则不可能。
- 如果一个环要求同一简单资源同时被两个进程占有，也不可能。

作业结论：给定三个进程并发执行时没有死锁可能。

### PV 设计题：桥/路径类资源

常用方法：

- 对每个独占位置设一个互斥信号量。
- 对可能形成环路的关键位置采用有序申请。
- 用总量信号量限制系统中同时进入的人数，避免所有关键资源被占满。
