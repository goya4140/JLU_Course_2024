# 05 死锁与饥饿（深度版）

> 本章目标：从“为什么进程会互相等死”开始，建立死锁、饥饿、安全状态、银行家算法和死锁检测的直觉，并能完成期末中的死锁大题。  
> 课内边界：本章深讲死锁与饥饿、Coffman 条件、资源分配图、死锁预防、避免、检测、恢复、银行家算法和静态死锁分析。PV 同步只在“无死锁设计题”中作为工具出现，具体 P/V 基础见第 4 章。

## 0. 本章定位与考试价值

第 4 章解决的是“多个进程怎样正确协作”。第 5 章进一步追问：

```text
如果多个进程都拿着一部分资源，
又都在等待别人手里的资源，
系统会不会永远推进不下去？
```

这就是死锁问题。

本章和第 4 章关系很紧：

- 第 4 章的互斥、同步、PV 操作会产生“占有资源”和“等待资源”。
- 第 5 章研究这些等待关系什么时候会变成死锁。
- 第 5 章的银行家算法、检测算法和资源分配图，是期末最标准的大题材料。

本章主线：

```text
进程竞争资源
    ↓
占有一些资源，同时申请另一些资源
    ↓
若形成互相等待，可能发生死锁
    ↓
死锁发生需要 Coffman 四个必要条件
    ↓
处理方式有四类：预防、避免、检测、恢复
    ↓
预防：静态破坏某个必要条件
    ↓
避免：动态检查系统是否仍安全，典型是银行家算法
    ↓
检测：允许死锁可能发生，之后用算法找出死锁
    ↓
恢复：终止进程、剥夺资源、回退
```

考试价值：

- 选择题常考：Coffman 条件、有序分配破坏哪个条件、死锁与安全状态关系、同类资源死锁最小进程数。
- 简答题常考：死锁与饥饿的异同。
- 大题常考：银行家算法、Need 矩阵、安全序列、请求能否分配。
- 作业题常考：死锁检测算法、参与死锁进程、资源请求序列、资源活动序列静态分析。
- PV 设计题常考：在资源路径中避免死锁、无饥饿、并行度高。

## 1. 死锁的直觉与定义

### 1.1 从两把资源开始

最小的死锁直觉：

```text
P1 已经拿到 r1，还想要 r2；
P2 已经拿到 r2，还想要 r1；
P1 不释放 r1，等 P2 释放 r2；
P2 不释放 r2，等 P1 释放 r1。
```

结果：

```text
P1 等 P2，P2 等 P1，谁都无法继续。
```

这不是简单的“慢”，而是系统状态已经形成闭环等待。如果没有外力介入，这组进程永远不会推进。

### 1.2 【必会】课内定义

课件定义：

```text
一组进程中的每一个进程，均无限期地等待此组进程中某个其他进程占有的，
因而永远无法得到的资源，这种现象称为进程死锁。
```

定义里有几个关键词：

| 关键词 | 含义 |
|---|---|
| 一组进程 | 死锁不是单个进程孤立现象 |
| 每一个进程都等待 | 参与死锁的进程都推进不了 |
| 等待组内其他进程占有的资源 | 等待关系在组内闭合 |
| 永远无法得到 | 不靠外部干预不会解除 |

考试判断：

- “一个进程自己等待自己”一般不是课内死锁定义的典型形式。
- “进程等待 I/O 一段时间”不一定是死锁。
- “多个进程互相等待对方释放资源”高度可疑。

### 1.3 死锁时刻

课件提到死锁时刻有两种理解：

1. 无限等待已经发生时。
2. 无限等待发生前，但系统已经注定会死锁时。

对考试来说，通常不用纠结“究竟从哪一瞬间开始算死锁”，重点是判断当前资源状态是否已经无法完全约简、是否存在安全序列、是否有参与死锁的进程集合。

### 1.4 【必会】死锁定义推出的结论

课件给出几个结论：

| 结论 | 为什么 |
|---|---|
| 参与死锁的进程至少有 2 个 | 死锁是组内互相等待 |
| 每个参与死锁的进程均等待资源 | 不等待的进程可以继续推进 |
| 参与死锁的进程中至少有 2 个占有资源 | 如果没人占有资源，等待链无法闭合 |
| 死锁进程是当前进程集合的一个子集 | 系统中可能只有部分进程死锁 |

【易错】系统中有死锁时，不是所有进程都一定死锁。可能只有 `{P1, P3, P5}` 形成死锁，其他进程仍能运行。

## 2. 死锁类型

### 2.1 【重点】竞争不同种资源引起的死锁

不同种资源死锁最直观。

例子：

```text
P1: 申请 A -> 申请 B -> 释放 A/B
P2: 申请 B -> 申请 A -> 释放 B/A
```

如果 P1 已拿到 A，P2 已拿到 B，之后 P1 等 B、P2 等 A，就会死锁。

交通路口也可以类比：每辆车占住一个路段，又要进入下一段，结果大家在环上互相挡住。

### 2.2 【重点】竞争同种资源引起的死锁

同种资源也会死锁。

比如系统有 4 台打印机，两个进程都需要多台打印机才能继续。若每个进程先拿一部分，再等剩余资源，就可能形成互相等待。

【题型】习题课模拟选择：

```text
某计算机系统中有 8 台打印机，由 K 个进程竞争使用，
每个进程最多需要 3 台打印机。该系统可能会发生死锁的 K 的最小值是？
```

解法见后文“同种组合资源死锁条件”。结论是：

```text
K = 4
```

### 2.3 进程通信引起的死锁

课件例子：

```text
P1: receive(P2, M1);
P2: receive(P3, M2);
P3: receive(P1, M3);
```

如果三个进程都先等待接收，而没有任何进程发送消息，就会形成通信等待环。

这说明死锁不只来自硬件资源，也可能来自通信事件。

### 2.4 其他原因引起的死锁

课件提到 “After you / after you”：

```text
你先。
不，你先。
不，你先。
```

双方都等待对方先动作，结果谁都不动作。这类问题不是典型资源死锁，但等待关系的直觉相同。

## 3. 【必会】Coffman 四个必要条件

### 3.1 四个条件

死锁发生必须同时满足四个必要条件，称 Coffman 条件：

| 条件 | 英文 | 含义 |
|---|---|---|
| 资源独占 | mutual exclusion | 某些资源一次只能被一个进程占用 |
| 不可抢占 | non-preemption | 资源只能由占有进程主动释放，不能被强行剥夺 |
| 保持申请 | hold while applying | 进程占有已有资源的同时，还继续申请新资源 |
| 循环等待 | circular wait | 存在进程等待环，P1 等 P2，P2 等 P3，最后又等 P1 |

应试口诀：

```text
独占、不可抢、占着再要、循环等。
```

### 3.2 必要条件不是充分条件

四个条件是必要条件：

```text
发生死锁 -> 四个条件都成立
```

但一般情况下，四个条件都成立不一定马上死锁。

课件特别强调：

```text
当每类资源只有一个实例时，Coffman 条件是充要条件。
```

也就是说：

- 单实例资源：循环等待通常就意味着死锁。
- 多实例资源：有环不一定死锁，因为某类资源可能还有其他实例可用。

### 3.3 【题型】有序分配破坏哪个条件

真题选择：

```text
在死锁的四个必要条件中，采用有序分配策略来预防死锁主要是通过破坏哪个条件？
```

答案：

```text
循环等待条件。
```

原因：

有序分配要求所有进程按资源编号递增顺序申请资源。这样等待关系只能沿编号增大的方向走，不可能形成一个回到起点的环。

## 4. 死锁处理总览

课件把死锁处理分为四类：

| 方法 | 时机 | 直觉 |
|---|---|---|
| 死锁预防 | 静态 | 事先规定资源申请规则，破坏某个必要条件 |
| 死锁避免 | 动态 | 每次分配前判断是否仍安全 |
| 死锁检测 | 事后 | 允许死锁可能发生，定期或按需检测 |
| 死锁恢复 | 事后 | 检测到死锁后解除它 |

区别：

```text
预防：一开始就不让危险模式出现。
避免：请求来了再判断这一步能不能走。
检测：先让系统跑，出事再查。
恢复：查到死锁后怎么救。
```

## 5. 资源分配图与死锁定理

### 5.1 【重点】资源分配图定义

资源分配图：

```text
G = (V, E)
V = P ∪ R
P = 进程结点集合
R = 资源类结点集合
```

边有两类：

| 边 | 方向 | 含义 |
|---|---|---|
| 申请边 | `Pi -> Rj` | 进程 Pi 正在申请资源 Rj |
| 分配边 | `Rj -> Pi` | Rj 的某个实例已经分配给 Pi |

图示直觉：

```text
进程 -> 资源：我想要它
资源 -> 进程：它已经给了我
```

### 5.2 有环一定死锁吗

不一定。

课件给了三种例子：

- 无环路，无死锁。
- 有环路，有死锁。
- 有环路，无死锁。

关键原因：

```text
如果某类资源有多个实例，即使图中有环，也可能有其他实例释放后打破等待。
```

考试判断：

- 单实例资源：有环通常就是死锁。
- 多实例资源：有环只是可能死锁，要进一步分析是否可约简。

### 5.3 【重点】资源分配图约简

课件约简算法：

1. 寻找一个非孤立且没有请求边的进程结点 `Pi`。如果没有，算法结束。
2. 去除 `Pi` 的所有分配边，使它成为孤立结点。
3. 寻找所有请求边都可满足的进程 `Pj`，将 `Pj` 的请求边改为分配边。
4. 回到步骤 1。

直觉：

```text
如果某个进程的请求能被满足，就假设它最终能运行完并释放资源。
释放资源后，再看其他进程能不能继续。
```

### 5.4 【必会】死锁定理

课件死锁定理：

```text
系统 S 为死锁状态的充分必要条件是：
S 的资源分配图不可完全约简。
```

翻译：

- 如果图可以完全约简，说明存在一种进程完成顺序，系统不会死锁。
- 如果图不能完全约简，剩下的非孤立结点就是被卡住的死锁相关部分。

## 6. 死锁预防

### 6.1 预防的思想

死锁预防是静态方法：

```text
对进程申请资源的活动施加限制，
让所有进程遵循这种限制，
从而保证死锁不发生。
```

优点：

- 简单。
- 系统运行时不必复杂判断。

缺点：

- 对进程限制强。
- 资源利用率可能低。
- 若进程不遵守约束仍可能出问题。

课件重点讲两种：

- 预先分配法。
- 有序分配法。

### 6.2 【重点】预先分配法

规则：

```text
进程运行前一次申请所需全部资源；
系统若能满足，就全部分配；
否则一个也不分配。
```

它破坏的是：

```text
保持申请条件。
```

因为进程不会“先占一部分，再申请另一部分”。它要么一开始全拿到，要么一个都不拿。

缺点：

- 资源利用率低。
- 进程很难提前准确知道全部资源需求。
- 某些资源可能很晚才用，却必须提前占有。

### 6.3 【必会】有序分配法

规则：

给每类资源编号：

```text
F: R -> N
```

进程只能按资源编号递增顺序申请资源。

课件形式：

```text
进程 Pi 可以申请资源 rj 中的实例
当且仅当 Pi 当前占有的任意资源 rl 都满足 F(rl) < F(rj)
```

直觉：

```text
只允许从小编号资源走向大编号资源，
等待边不可能绕成环。
```

它破坏的是：

```text
循环等待条件。
```

优点：

- 比预先分配资源利用率高。

缺点：

- 资源编号困难。
- 为了保持顺序，进程可能被迫提前申请暂时不用的资源。

### 6.4 【题型】有序分配证明无死锁

证明常用反证法：

假设发生死锁，有循环等待：

```text
P1 等待 r_k1，该资源被 P2 占有；
P2 等待 r_k2，该资源被 P3 占有；
...
Pn 等待 r_kn，该资源被 P1 占有。
```

根据有序申请：

```text
F(r_k1) < F(r_k2) < ... < F(r_kn) < F(r_k1)
```

这不可能成立，矛盾。故不会发生死锁。

## 7. 死锁避免与安全状态

### 7.1 【必会】安全状态

死锁避免不是简单拒绝所有危险请求，而是判断：

```text
如果现在把资源分出去，系统是否仍存在一种完成顺序？
```

课件定义：

```text
系统处于安全状态：
存在安全进程序列 <P1, P2, ..., Pn>，
使这些进程可以依次进行完。
```

安全序列的直觉：

```text
先让某个当前需求能满足的进程完成；
它释放资源后，再满足下一个；
最后所有进程都能完成。
```

### 7.2 【必会】安全、不安全、死锁的关系

关系：

```text
安全状态 -> 一定没有死锁
死锁状态 -> 一定是不安全状态
不安全状态 -> 不一定已经死锁
```

【题型】习题课模拟选择：

```text
死锁和安全状态的关系是？
```

正确判断：

```text
死锁状态一定是不安全状态。
```

【易错】不安全状态不是死锁状态。它只是“如果后续请求不巧，可能走向死锁”。

可以这样理解：

```text
安全 = 有保底完成路线；
不安全 = 没有保底路线，但未必已经卡死；
死锁 = 已经卡死。
```

## 8. 【必会】银行家算法

### 8.1 银行家算法的直觉

银行家算法由 Dijkstra 提出。

名字来自银行贷款：

```text
银行不会只看当前有没有钱，
还要看把钱借出去后，是否仍能保证所有客户最终完成。
```

操作系统中的资源分配也是这样：

- 请求量小于当前可用资源，不代表一定能分配。
- 分配后系统仍安全，才确认分配。
- 分配后系统不安全，就回滚，让进程等待。

### 8.2 【必会】数据结构

课件给出：

| 数据结构 | 含义 |
|---|---|
| `Available[m]` | 系统当前可用的每类资源数量 |
| `Claim[n][m]` 或 `Max[n][m]` | 每个进程声明的最大需求 |
| `Allocation[n][m]` | 当前已经分配给每个进程的资源 |
| `Need[n][m]` | 每个进程还需要的资源 |
| `Request[n][m]` | 当前请求 |
| `Work[m]` | 安全性检测中的临时可用资源 |
| `Finish[n]` | 安全性检测中进程是否可完成 |

核心公式：

```text
Need = Max - Allocation
```

如果题目直接给了 Need，就不用再算 Max。

### 8.3 【必会】安全性检测算法

步骤：

```text
Work = Available
Finish[i] = false

循环寻找某个 i：
  Finish[i] == false 且 Need[i] <= Work

若找到：
  Work = Work + Allocation[i]
  Finish[i] = true
  把 Pi 加入安全序列
  继续找

若找不到：
  如果所有 Finish[i] 都为 true，则安全
  否则不安全
```

考试写法：

```text
Work=(...)
Need[P0] <= Work，所以 P0 可完成，Work += Allocation[P0] = (...)
Need[P3] <= Work，所以 P3 可完成，Work += Allocation[P3] = (...)
...
得到安全序列 <...>，系统安全。
```

【易错】不是找 `Allocation <= Work`，而是找 `Need <= Work`。进程能不能完成，取决于“还差多少”能不能满足。

### 8.4 【必会】资源请求算法

当进程 `Pi` 请求 `Request[i]` 时：

1. 检查是否超过声明需求：

```text
Request[i] <= Need[i]
```

若不满足，说明请求超量，出错返回。

2. 检查当前是否有足够资源：

```text
Request[i] <= Available
```

若不满足，资源暂时不足，进程等待。

3. 假设分配：

```text
Available = Available - Request[i]
Allocation[i] = Allocation[i] + Request[i]
Need[i] = Need[i] - Request[i]
```

4. 执行安全性检测。

5. 如果安全，确认分配；如果不安全，取消分配：

```text
Available = Available + Request[i]
Allocation[i] = Allocation[i] - Request[i]
Need[i] = Need[i] + Request[i]
Pi 等待
```

【题型】习题课第 14 页就是这个流程图。考试大题照这个流程写，不容易丢步骤分。

### 8.5 【题型】作业四银行家算法例题

题目给出：

| 进程 | Allocation A B C D | Need A B C D |
|---|---|---|
| P0 | 0 0 3 2 | 0 0 1 2 |
| P1 | 1 0 0 0 | 1 7 5 0 |
| P2 | 1 3 5 4 | 2 3 5 6 |
| P3 | 0 3 3 2 | 0 6 5 2 |
| P4 | 0 0 1 4 | 0 6 5 6 |

```text
Available = (1, 6, 2, 3)
```

#### 8.5.1 当前是否安全

安全性检测：

```text
Work = (1, 6, 2, 3)
```

1. `Need[P0]=(0,0,1,2) <= Work`  
   P0 可完成：

```text
Work = Work + Allocation[P0]
     = (1,6,2,3) + (0,0,3,2)
     = (1,6,5,5)
```

2. `Need[P3]=(0,6,5,2) <= Work`  
   P3 可完成：

```text
Work = (1,6,5,5) + (0,3,3,2)
     = (1,9,8,7)
```

3. `Need[P4]=(0,6,5,6) <= Work`  
   P4 可完成：

```text
Work = (1,9,8,7) + (0,0,1,4)
     = (1,9,9,11)
```

4. `Need[P1]=(1,7,5,0) <= Work`  
   P1 可完成：

```text
Work = (1,9,9,11) + (1,0,0,0)
     = (2,9,9,11)
```

5. `Need[P2]=(2,3,5,6) <= Work`  
   P2 可完成：

```text
Work = (2,9,9,11) + (1,3,5,4)
     = (3,12,14,15)
```

所以系统安全，安全序列可以写：

```text
<P0, P3, P4, P1, P2>
```

#### 8.5.2 P2 请求 `(1,2,2,2)` 能否分配

先检查请求合法性：

```text
Request[2] = (1,2,2,2)
Need[2]    = (2,3,5,6)
Available  = (1,6,2,3)
```

满足：

```text
Request[2] <= Need[2]
Request[2] <= Available
```

所以可以假设分配。

假设分配后：

```text
Available'  = (1,6,2,3) - (1,2,2,2) = (0,4,0,1)
Allocation'[P2] = (1,3,5,4) + (1,2,2,2) = (2,5,7,6)
Need'[P2]       = (2,3,5,6) - (1,2,2,2) = (1,1,3,4)
```

此时运行安全性检测：

```text
Work = (0,4,0,1)
```

检查各进程 Need：

- P0 需要 `(0,0,1,2)`，C 和 D 不够。
- P1 需要 `(1,7,5,0)`，A/B/C 不够。
- P2 需要 `(1,1,3,4)`，A/C/D 不够。
- P3 需要 `(0,6,5,2)`，B/C/D 不够。
- P4 需要 `(0,6,5,6)`，B/C/D 不够。

找不到任何可完成进程。

结论：

```text
假设分配后系统不安全，必须取消分配，P2 等待。
```

【易错】请求合法且当前可满足，不代表可以真正分配；还必须通过安全性检测。

### 8.6 【题型】真题银行家算法大题

真题第 IV 题考法：

```text
给出 Allocation / Max 或 Need / Available；
问：
1. 哪些进程处于等待状态？资源总量是多少？
2. Need 矩阵怎样？
3. 系统是否安全？给出安全序列。
4. 某进程请求向量能否满足？说明理由。
```

答题模板：

1. 求 `Need`：

```text
Need = Max - Allocation
```

2. 求资源总量：

```text
Total[j] = Available[j] + sum_i Allocation[i][j]
```

3. 判断等待状态：

```text
若题目给出 Request 或某进程当前请求无法满足，则处于等待；
若只给银行家 Max/Allocation/Available，不能随便说 Need>0 就是等待。
```

4. 安全性检测：

```text
Work = Available
找 Need[i] <= Work 的进程
Work += Allocation[i]
直到全部完成或找不到
```

5. 请求判断：

```text
Request <= Need?
Request <= Available?
假设分配后是否安全？
```

【易错】`Need > 0` 表示未来还可能需要资源，不等于当前正在等待资源。

## 9. 死锁检测

### 9.1 避免与检测的区别

银行家算法属于死锁避免：

```text
每次分配前检查，不让系统进入不安全状态。
```

死锁检测则是：

```text
允许系统可能进入死锁；
之后用检测算法判断当前是否死锁。
```

检测算法不需要最大需求 `Max/Claim`，它需要的是当前请求 `Request`。

### 9.2 【必会】检测算法数据结构

课件给出：

| 数据结构 | 含义 |
|---|---|
| `Available[m]` | 当前可用资源 |
| `Allocation[n][m]` | 当前分配 |
| `Request[n][m]` | 当前未满足请求 |
| `Work[m]` | 临时可用资源 |
| `Finish[n]` | 进程能否完成 |

注意区别：

```text
银行家安全性检测用 Need <= Work；
死锁检测用 Request <= Work。
```

因为死锁检测只关心“当前卡住的请求能不能满足”，不是最大剩余需求。

### 9.3 【必会】检测算法流程

基本流程：

```text
Work = Available
Finish[i] = false

寻找某个 i：
  Finish[i] == false 且 Request[i] <= Work

若找到：
  Finish[i] = true
  Work = Work + Allocation[i]
  继续

若找不到：
  如果所有 Finish[i] 都为 true，则无死锁
  否则 Finish[i] == false 的进程参与死锁
```

课件补充：

如果只希望检测“占有资源的进程”：

```text
对 Allocation[i] == 0 的进程，初始化 Finish[i] = true
```

这样可以减少检测范围和开销。

### 9.4 【题型】作业四死锁检测例题

题目：

资源总量：

```text
A = 8, B = 6, C = 5
```

当前状态：

| 进程 | Allocation A B C | Request A B C |
|---|---|---|
| P1 | 1 0 0 | 0 0 0 |
| P2 | 3 2 1 | 0 0 0 |
| P3 | 0 1 2 | 2 0 2 |
| P4 | 0 0 0 | 0 0 0 |
| P5 | 2 1 0 | 0 3 1 |
| P6 | 0 0 1 | 0 0 0 |

```text
Available = (2,2,1)
```

#### 9.4.1 接收请求但未分配时

依次接收：

```text
Request[1] = (1,0,0)
Request[2] = (2,1,0)
Request[4] = (0,0,2)
```

如果只是接收请求，未分配资源，则 Request 变为：

| 进程 | Allocation A B C | Request A B C |
|---|---|---|
| P1 | 1 0 0 | 1 0 0 |
| P2 | 3 2 1 | 2 1 0 |
| P3 | 0 1 2 | 2 0 2 |
| P4 | 0 0 0 | 0 0 2 |
| P5 | 2 1 0 | 0 3 1 |
| P6 | 0 0 1 | 0 0 0 |

```text
Work = (2,2,1)
```

可找到安全/完成序列：

```text
<P4, P1, P2, P3, P5, P6>
```

所以没有死锁。

#### 9.4.2 将 1 个 A 分配给 P1 后

状态变为：

```text
P1 Allocation = (2,0,0)
P1 Request    = (0,0,0)
Available     = (1,2,1)
```

习题课给出的检测结果仍可找到完成序列：

```text
<P4, P1, P2, P3, P5, P6>
```

所以没有死锁。

#### 9.4.3 再接收 P1 请求 `(0,3,1)`

这里习题课区分两个状态：

1. 在“仅接收请求但未分配 A 给 P1”的状态下，再接收：

```text
Request[1] = (0,3,1)
```

可找到完成序列：

```text
<P4, P2, P3, P5, P6, P1>
```

系统没有进入死锁。

2. 在“已经将 1 个 A 分配给 P1”的状态下，再接收：

```text
Request[1] = (0,3,1)
```

运行检测算法找不到完成序列，系统进入死锁状态。

习题课要求找出参与死锁的进程。答题时要写：

```text
检测算法结束时 Finish=false 的进程，就是参与死锁的进程。
```

如果按表逐步推演，应把最终 `Finish=false` 的进程集合列出。由于题面/课件截图中有多步状态，考试时不要凭印象写集合，要按最终表运行检测算法。

【易错】死锁检测题看的是 `Request <= Work`，不是 `Need <= Work`。

### 9.5 死锁检测时刻

课件给出三类检测时机：

| 检测时刻 | 特点 |
|---|---|
| 等待时检测 | 发现早，恢复代价小，但开销大 |
| 定时检测 | 折中 |
| 资源利用率下降时检测 | 用系统异常迹象触发 |

考试理解：

- 检测越频繁，越早发现，开销越大。
- 检测越少，开销小，但死锁影响可能扩大。

## 10. 死锁恢复

检测到死锁后，系统要恢复。

课件给出三类：

### 10.1 重新启动

做法：

```text
重启系统或相关作业。
```

优点：

- 简单。

缺点：

- 代价大。
- 会影响未参与死锁的进程。

### 10.2 终止进程

可以终止环路上占有资源的进程。

两种策略：

1. 一次性全部终止。
2. 逐步终止，根据优先级或代价函数选择牺牲者。

选择牺牲进程时可能考虑：

- 进程优先级。
- 已运行时间。
- 还需多久完成。
- 占有资源多少。
- 回滚代价。

### 10.3 剥夺资源 + 回退

做法：

```text
从某些进程剥夺资源；
让这些进程回退到之前保存的快照；
把资源分给其他进程以解除死锁。
```

问题：

- 保存快照代价大。
- 回退后消除外部影响困难。
- 反复选择同一进程可能导致饥饿。

## 11. 鸵鸟算法

鸵鸟算法就是：

```text
视而不见。
```

工程师视角：

- 如果死锁发生频率很低。
- 死锁危害小于持续检测/预防开销。
- 处理死锁的常驻开销太高。

那么系统可能选择不专门处理死锁，等问题严重时由用户或管理员重启。

数学家视角：

```text
死锁是错误，必须处理，无论代价如何。
```

考试定位：

- 鸵鸟算法不是说死锁不存在。
- 它是一种工程取舍。

## 12. 饥饿与活锁

### 12.1 【必会】饥饿定义

课件定义：

```text
当等待时间给进程推进和响应带来明显影响时，称发生了进程饥饿。
```

如果饥饿到一定程度，即使任务最终完成也没有实际意义，称为饿死。

饥饿的关键：

```text
等待没有时间上界。
```

饥饿可以发生在：

- 排队等待。
- 忙式等待。

### 12.2 活锁

忙式等待条件下发生的饥饿称为活锁。

直觉：

```text
进程不是睡着不动，
而是一直在动作、一直重试，
但始终没有有效推进。
```

比如两个人在走廊相遇，每次都同时向同一侧让路，然后又同时向另一侧让路，始终过不去。

### 12.3 【必会】死锁与饥饿的相同点

真题简答考过：

```text
死锁与饿死之间有何相同点和不同点？
```

相同点：

- 都和资源竞争或调度策略有关。
- 都会导致进程长时间不能推进。
- 都可能使进程任务无法及时完成。

### 12.4 【必会】死锁与饥饿的不同点

| 对比 | 死锁 | 饥饿 |
|---|---|---|
| 状态 | 死锁进程处于等待状态 | 饥饿进程不一定处于等待状态，也可能忙等 |
| 等待对象 | 等待永远不会释放给自己的资源 | 等待可能释放，但一直分不到的资源 |
| 等待时间 | 无限期等待 | 没有时间上界，可能很久 |
| 循环等待 | 一定发生循环等待 | 不一定发生循环等待 |
| 进程数量 | 至少涉及两个进程 | 可能只有一个进程饥饿 |
| 可察觉性 | 通常可通过检测发现 | 不一定容易检测 |

答题模板：

```text
二者都表现为进程长期不能推进，通常与资源竞争有关。
不同的是，死锁是一组进程互相等待对方占有且不会释放的资源，
死锁进程处于等待状态，并且一定有循环等待，至少涉及两个进程；
饥饿是进程长期得不到所需资源或处理机，资源可能会释放但总是不分给它，
不一定处于等待态，也不一定有循环等待，甚至可能只有一个进程发生饥饿。
```

## 13. 同种组合资源死锁条件

### 13.1 【必会】课件结论

课件讨论同种组合资源：

```text
M：资源数量
N：使用该类资源的进程数量
Σ：所有进程所需要该类资源的总量
```

结论：

```text
当 Σ < M + N 时，一定没有死锁；
当 Σ >= M + N 时，至少存在一种交叉有死锁可能。
```

### 13.2 直觉解释

假设发生死锁，有 `n` 个进程参与死锁。

每个参与死锁的进程至少还差 1 个资源才能继续，因此：

```text
参与死锁的进程总需求 >= 它们已占有资源数 + n
```

未参与死锁的进程每个至少需要 1 个资源，因此可推出总需求下界：

```text
Σ >= M + N
```

所以如果：

```text
Σ < M + N
```

死锁不可能发生。

### 13.3 【题型】打印机题

习题课模拟：

```text
系统有 8 台打印机，由 K 个进程竞争使用；
每个进程最多需要 3 台打印机；
可能发生死锁的 K 的最小值是多少？
```

这里：

```text
M = 8
N = K
Σ = 3K
```

要有死锁可能：

```text
Σ >= M + N
3K >= 8 + K
2K >= 8
K >= 4
```

所以最小值：

```text
K = 4
```

构造直觉：

4 个进程每个最多要 3 台。若每个进程先占 2 台，则 8 台全部被占完；每个进程还差 1 台，于是互相等待，没有进程能完成。

## 14. 简单组合资源死锁的静态分析

### 14.1 适用条件

课件给出条件：

```text
已知各个进程有关资源的活动序列。
```

例如：

```text
P1: a b a' b' e f g e' f' g'
P2: b c b' c' d a d' a'
P3: c d c' d' e g f e' f' g'
```

其中：

- `a` 表示申请资源 A。
- `a'` 表示释放资源 A。

### 14.2 【必会】静态分析步骤

课件步骤：

1. 以每个进程“占有资源，申请资源”的状态为一个结点：

```text
(Pi : 请求资源 : 已占有资源集合)
```

2. 如果状态 `s1` 所申请资源为状态 `s2` 所占有，则由 `s1` 向 `s2` 画有向边。

3. 同一进程间不画边。

4. 找出所有环路。

5. 判断环路上的状态能否同时到达：

```text
若能同时到达，则有死锁可能；
若不能同时到达，则无死锁可能。
```

不能同时到达的常见原因：

- 环路中有同一进程的两个不同状态。
- 环路中有同一个被占资源同时出现在两个状态里。

### 14.3 【题型】作业四资源活动序列题

题目：

```text
资源 A~G；
P1: a b a' b' e f g e' f' g'
P2: b c b' c' d a d' a'
P3: c d c' d' e g f e' f' g'
```

分析得到关键状态：

```text
(P1 : b : a)
(P2 : c : b)
(P3 : d : c)
(P2 : a : d)
(P1 : f : e)
(P3 : g : e)
(P1 : g : e f)
(P3 : f : e g)
```

习题课结论：

存在两个环路，但都不能形成死锁：

1. 一个环路包含相同进程 P2。  
   同一进程不可能同时处于两个不同状态，因此该环路不可达。

2. 另一个环路包含相同被占资源 `e`。  
   同一个资源不能同时分给两个进程，因此该环路不可达。

结论：

```text
三个进程并发执行时，没有死锁可能。
```

【易错】看到环路不要立刻写“有死锁”。静态分析题还要判断环路状态能否同时到达。

## 15. 过河/道路类无死锁设计

### 15.1 题型定位

作业四有一类题：

```text
分析可能发生死锁的情况；
设计无死锁、无饿死、并行度高的解法；
用 PV 操作实现。
```

这类题横跨第 4 章和第 5 章：

- 第 4 章给 PV 工具。
- 第 5 章给死锁分析和预防策略。

### 15.2 【重点】先找死锁情形

过河/道路题常见死锁：

- 双向车辆/行人分别占住路径的一部分，互相等待对方让出下一段。
- 多个路段组成环，每个进程占有当前路段并申请下一个路段。

习题课给出的过河题示例中：

```text
一位西岸过河者踏上 1，一位东岸过河者踏上 2，可能形成死锁；
3/4 类似；
四位西岸过河者踏上 2,5,6,7，四位东岸过河者踏上 4,8,9,10，也可能形成死锁。
```

### 15.3 设计原则

常用策略：

1. 有序分配  
   对关键路段编号，强制所有进程按同一顺序申请，破坏循环等待。

2. 限制同时进入数量  
   比如设置 `Smax`，让系统中同时进入路径的人数不超过某个上限，避免全部关键资源被占满。

3. 方向批处理  
   同一方向可并发，另一方向等待；方向切换时一次唤醒另一方向等待者。

4. 公平队列或轮转  
   避免某一方向源源不断导致另一方向饥饿。

### 15.4 答题结构

考试可以按如下结构写：

```text
1. 死锁分析：
   描述哪几个资源/路段可能形成等待环。

2. 信号量定义：
   每个互斥路段一个 semaphore，初值 1；
   若限制总人数，定义 Smax，初值为上限；
   若要防饥饿，定义等待计数和队列/方向控制。

3. 进程伪代码：
   按有序分配或方向规则申请路段；
   经过后及时释放；
   离开时唤醒等待者。

4. 正确性说明：
   无死锁：破坏循环等待或限制资源占满；
   无饥饿：公平切换/队列；
   并行度高：同向或不冲突路段允许并行。
```

## 16. 死锁与银行家算法易错点总表

| 易错点 | 正确理解 |
|---|---|
| 有等待就是死锁 | 等待可能只是资源暂时不可用，不一定死锁 |
| 死锁一定涉及所有进程 | 死锁进程只是当前进程集合的子集 |
| 有环一定死锁 | 多实例资源有环不一定死锁 |
| 没有安全序列就是死锁 | 没有安全序列是不安全，不一定已经死锁 |
| 死锁状态可能安全 | 错，死锁状态一定不安全 |
| 不安全状态一定死锁 | 错，不安全只是可能走向死锁 |
| 有序分配破坏保持申请 | 错，主要破坏循环等待 |
| 预先分配破坏循环等待 | 错，主要破坏保持申请 |
| 银行家算法只看 `Request <= Available` | 还要假设分配并检查安全性 |
| `Need > 0` 表示进程正在等待 | Need 表示未来还需，不等于当前 Request |
| 银行家安全性检测看 `Allocation <= Work` | 应看 `Need <= Work` |
| 死锁检测算法看 `Need <= Work` | 应看当前 `Request <= Work` |
| 检测算法中 Finish=false 一定是所有系统进程 | 是检测结束后无法完成的进程，通常为死锁相关进程 |
| 饥饿一定处于等待态 | 饥饿也可能发生在忙式等待中 |
| 死锁和饥饿都一定有循环等待 | 死锁一定有循环等待，饥饿不一定 |

## 17. 课内题型归纳与模板

### 17.1 【题型】Coffman 条件选择题

常见问法：

```text
某策略破坏哪个死锁必要条件？
```

记忆：

| 策略 | 破坏条件 |
|---|---|
| 预先分配 | 保持申请 |
| 有序分配 | 循环等待 |
| 可抢占资源 | 不可抢占 |
| 把独占资源改成共享/虚拟化 | 资源独占 |

### 17.2 【题型】银行家算法大题

答题模板：

```text
1. Need = Max - Allocation。
2. Work = Available，Finish 全 false。
3. 找 Need[i] <= Work 的进程 Pi。
4. Work = Work + Allocation[i]，Finish[i]=true。
5. 重复，得到安全序列。
6. 若请求：
   先判 Request <= Need；
   再判 Request <= Available；
   假设分配；
   跑安全性检测；
   安全则分配，不安全则回滚。
```

### 17.3 【题型】死锁检测大题

答题模板：

```text
1. Work = Available。
2. Finish 初始化。
3. 找 Request[i] <= Work 的进程 Pi。
4. Work = Work + Allocation[i]，Finish[i]=true。
5. 重复。
6. 若最后存在 Finish=false，则这些进程无法完成，系统死锁。
```

### 17.4 【题型】死锁 vs 饥饿简答

答题模板见 12.4。关键词：

```text
相同：长期不能推进、资源竞争。
不同：等待状态、资源是否永远得不到、循环等待、涉及进程数、是否可检测。
```

### 17.5 【题型】同种资源最小进程数

模板：

```text
M = 资源总数
N = 进程数
每个进程最多需要 x 个
Σ = xN

有死锁可能需要：
Σ >= M + N
```

若问最小 N，解不等式。

### 17.6 【题型】静态资源序列分析

模板：

```text
1. 列出每个进程“申请某资源时已经占有什么”的状态。
2. 画等待边：请求资源被谁占有，就指向谁的状态。
3. 找环。
4. 判断环上状态能否同时到达。
5. 若环可达，有死锁可能；若所有环不可达，无死锁可能。
```

## 19. 本章一句话总结

第 5 章的核心是：

```text
把“谁占有什么、谁还在等什么”写清楚，
再判断等待关系是否仍有某种完成顺序。
```

如果存在完成顺序，就是安全或无死锁；如果不存在，就要根据场景判断是不安全、已经死锁，还是需要检测/恢复。银行家算法和死锁检测算法本质上都是在做这件事，只是一个看 `Need`，一个看 `Request`。
