## 3. 死锁定义与基本判断

### 3.1 知识点速记

【必背】死锁定义：

```text
一组进程中的每个进程，都无限期等待该组中其他进程占有且不会释放给它的资源，
导致这组进程都无法继续推进。
```

关键词：

| 关键词 | 判断含义 |
|---|---|
| 一组进程 | 不是单个进程孤立事件 |
| 每个进程都等待 | 参与死锁的进程都无法推进 |
| 等待组内其他进程占有的资源 | 等待关系闭合 |
| 无限期等待 | 不靠外部干预无法解除 |

【会判】死锁的基本结论：

1. 参与死锁的进程至少有 2 个。
2. 每个参与死锁的进程都在等待资源。
3. 参与死锁的进程中至少有 2 个已经占有资源。
4. 死锁进程只是当前进程集合的子集，不一定是全部进程。

### 3.2 对应例题：概念判断题

题目：

```text
判断下列说法是否正确：
A. 只要进程等待 I/O，就说明系统发生了死锁。
B. 系统中发生死锁时，所有进程都一定处于死锁状态。
C. 发生死锁时，参与死锁的进程至少有两个。
D. 发生死锁时，参与死锁的进程一定都在等待某种资源。
```

答案：

```text
A 错，B 错，C 对，D 对。
```

解答过程：

1. A 错：等待 I/O 可能只是正常阻塞，I/O 完成后可以继续。
2. B 错：死锁可能只发生在部分进程之间，其他进程仍然能运行。
3. C 对：课内死锁定义强调“一组进程”互相等待，至少两个进程。
4. D 对：参与死锁的进程推进不了，本质上就是在等资源或事件。

【一句话记法】

```text
等待不一定死锁；互相等死才是死锁。
```

## 4. Coffman 四个必要条件

### 4.1 知识点速记

【必背】死锁发生必须同时满足四个必要条件：

| 条件 | 含义 | 典型破坏方法 |
|---|---|---|
| 互斥 | 资源一次只能被一个进程使用 | 尽量共享化，但很多资源不能破坏 |
| 不可剥夺 | 资源只能由占有者主动释放 | 允许抢占/剥夺资源 |
| 保持申请 | 进程占有资源时继续申请新资源 | 一次性申请全部资源，或申请失败释放已有资源 |
| 循环等待 | 进程等待关系形成环 | 资源有序分配 |

【必背】口诀：

```text
互斥、不可抢、占着再要、循环等。
```

【易错】四条件是必要条件：

```text
发生死锁 -> 四条件同时成立
```

但考试中不要反过来说“只要四条件存在就一定已经死锁”。在多实例资源场景下，还要看具体状态是否真的卡住。

### 4.2 对应例题：有序分配破坏哪个条件

题目：

```text
在死锁的四个必要条件中，采用有序分配策略来预防死锁，
主要是通过破坏哪个条件？
```

答案：

```text
循环等待条件。
```

解答过程：

有序分配要求每类资源编号，进程只能按编号递增顺序申请资源。

如果进程已经持有低编号资源，只能继续申请更高编号资源，那么等待关系只能从小编号指向大编号：

```text
R1 -> R2 -> R3 -> ...
```

它不可能再绕回低编号资源，因此不可能形成循环等待。

【易错排雷】

| 错误说法 | 为什么错 |
|---|---|
| 有序分配破坏互斥 | 资源仍然可以互斥使用 |
| 有序分配破坏不可剥夺 | 资源仍然不能被强行抢走 |
| 有序分配破坏保持申请 | 进程仍可能占着一个资源再申请另一个 |
| 有序分配破坏循环等待 | 正确 |

【一句话记法】

```text
有序分配让等待边只能单向变大，所以破坏循环等待。
```

## 5. 死锁处理策略

### 5.1 知识点速记

【必背】死锁处理有四类：

| 策略 | 什么时候管 | 核心思想 | 典型方法 |
|---|---|---|---|
| 死锁预防 | 事前 | 静态破坏必要条件 | 一次性申请、有序分配 |
| 死锁避免 | 运行中 | 每次分配前判断是否安全 | 银行家算法 |
| 死锁检测 | 事后 | 允许可能发生，之后检查 | 死锁检测算法 |
| 死锁恢复 | 检测后 | 解除已经发生的死锁 | 撤销进程、剥夺资源、回滚 |

【会判】四者区别：

```text
预防：制定规则，不让危险等待模式出现。
避免：请求来了，先算一算这一步能不能走。
检测：先让系统跑，之后查是否死锁。
恢复：查出死锁后救场。
```

### 5.2 对应例题：策略判断

题目：

```text
判断下列说法：
A. 银行家算法属于死锁预防。
B. 有序资源分配属于死锁预防。
C. 死锁检测要求系统每次分配前都保持安全状态。
D. 检测到死锁后，可以通过撤销进程或剥夺资源恢复。
```

答案：

```text
A 错，B 对，C 错，D 对。
```

解答过程：

1. A 错：银行家算法是死锁避免，不是预防。它是在运行时动态判断安全性。
2. B 对：有序分配事先规定申请顺序，属于预防。
3. C 错：每次分配前保持安全状态是避免策略的思想，不是检测策略。
4. D 对：恢复方法包括终止进程、资源剥夺、回退等。

【一句话记法】

```text
银行家是避免；有序分配是预防；检测是出事后查。
```

## 6. 资源分配图与安全状态

### 6.1 资源分配图速记

资源分配图有两类结点：

| 结点 | 含义 |
|---|---|
| 进程结点 `Pi` | 一个进程 |
| 资源结点 `Rj` | 一类资源，可有一个或多个实例 |

两类边：

| 边 | 含义 |
|---|---|
| `Pi -> Rj` | 请求边：进程正在申请资源 |
| `Rj -> Pi` | 分配边：资源已经分配给进程 |

【会判】有环与死锁：

| 情况 | 判断 |
|---|---|
| 单实例资源有环 | 通常可判死锁 |
| 多实例资源有环 | 只说明可能死锁，不一定死锁 |
| 资源分配图可完全约简 | 无死锁 |
| 资源分配图不可完全约简 | 死锁状态 |

【必背】死锁定理：

```text
系统 S 为死锁状态的充分必要条件：
S 的资源分配图不可完全约简。
```

### 6.2 安全、不安全、死锁关系

【必背】三者关系：

```text
安全状态 -> 一定没有死锁
死锁状态 -> 一定是不安全状态
不安全状态 -> 不一定已经死锁
```

可以画成：

```text
死锁状态 ⊂ 不安全状态
安全状态 与 不安全状态相对
```

【易错】不安全状态只是“没有保底完成序列”，不代表当前已经互相等死。

### 6.3 对应例题：安全状态判断

题目：

```text
判断下列说法：
A. 安全状态一定没有死锁。
B. 不安全状态一定已经死锁。
C. 死锁状态一定是不安全状态。
D. 系统存在安全序列，说明系统当前是安全的。
```

答案：

```text
A 对，B 错，C 对，D 对。
```

解答过程：

1. A 对：安全状态存在某种完成顺序，所有进程都能完成。
2. B 错：不安全只是没有保证完成的路线，后续是否死锁还取决于请求顺序。
3. C 对：死锁已经无法完成，当然不存在安全序列。
4. D 对：安全序列就是安全状态的定义核心。

【一句话记法】

```text
安全是有路；不安全是没保底路；死锁是已经堵死。
```

## 7. 银行家算法

### 7.1 数据结构与公式

【会算】银行家算法需要这些量：

| 名称 | 含义 |
|---|---|
| `Available` | 当前系统每类资源还剩多少 |
| `Max` / `Claim` | 每个进程声明的最大需求 |
| `Allocation` | 当前已经分配给每个进程的资源 |
| `Need` | 每个进程还需要多少资源才能完成 |
| `Request` | 某进程当前提出的请求 |
| `Work` | 安全性检测中的临时可用资源 |
| `Finish` | 安全性检测中进程是否可以完成 |

核心公式：

```text
Need = Max - Allocation
```

如果题目直接给 `Need`，就不用再算 `Max`。

【易错】`Need > 0` 不等于进程正在等待。`Need` 表示未来还可能需要；`Request` 才是当前正在请求。

### 7.2 安全性检测模板

【会算】安全性检测步骤：

```text
Work = Available
Finish[i] = false

寻找某个进程 Pi：
  Finish[i] == false 且 Need[i] <= Work

若找到：
  Work = Work + Allocation[i]
  Finish[i] = true
  把 Pi 加入安全序列

重复上述过程。

若所有 Finish 都为 true：系统安全。
否则：系统不安全。
```

【易错】银行家算法安全性检测看：

```text
Need <= Work
```

不是：

```text
Allocation <= Work
```

### 7.3 资源请求算法模板

【会算】当进程 `Pi` 请求 `Request[i]`：

1. 先判断是否超过声明需求：

```text
Request[i] <= Need[i]
```

不满足：请求非法。

2. 再判断当前资源是否够：

```text
Request[i] <= Available
```

不满足：资源不足，进程等待。

3. 假设分配：

```text
Available = Available - Request[i]
Allocation[i] = Allocation[i] + Request[i]
Need[i] = Need[i] - Request[i]
```

4. 做安全性检测。

5. 若安全：确认分配。若不安全：回滚，进程等待。

### 7.4 对应例题：作业四第 1 题，当前状态是否安全

题目：

```text
在银行家算法中，出现如下资源分配情况。

|        | Allocation | Need      | Available |
|--------|------------|-----------|-----------|
|        | A B C D    | A B C D   | A B C D   |
| P0     | 0 0 3 2    | 0 0 1 2   | 1 6 2 3   |
| P1     | 1 0 0 0    | 1 7 5 0   |           |
| P2     | 1 3 5 4    | 2 3 5 6   |           |
| P3     | 0 3 3 2    | 0 6 5 2   |           |
| P4     | 0 0 1 4    | 0 6 5 6   |           |

（1）当前状态是否安全？
```

答案：

```text
当前状态安全。
安全序列可取 <P0, P3, P4, P1, P2>。
```

解答过程：

初始：

```text
Work = Available = (1, 6, 2, 3)
```

第 1 步，找 `Need <= Work`：

```text
P0 Need = (0,0,1,2) <= (1,6,2,3)
```

所以 P0 可完成，释放它已占有的资源：

```text
Work = Work + Allocation[P0]
     = (1,6,2,3) + (0,0,3,2)
     = (1,6,5,5)
```

第 2 步：

```text
P3 Need = (0,6,5,2) <= (1,6,5,5)
```

P3 完成：

```text
Work = (1,6,5,5) + (0,3,3,2)
     = (1,9,8,7)
```

第 3 步：

```text
P4 Need = (0,6,5,6) <= (1,9,8,7)
```

P4 完成：

```text
Work = (1,9,8,7) + (0,0,1,4)
     = (1,9,9,11)
```

第 4 步：

```text
P1 Need = (1,7,5,0) <= (1,9,9,11)
```

P1 完成：

```text
Work = (1,9,9,11) + (1,0,0,0)
     = (2,9,9,11)
```

第 5 步：

```text
P2 Need = (2,3,5,6) <= (2,9,9,11)
```

P2 完成：

```text
Work = (2,9,9,11) + (1,3,5,4)
     = (3,12,14,15)
```

所有进程都能完成，因此系统安全。

【易错排雷】

1. 安全序列不唯一，写出一条合法的即可。
2. 每一步比较的是 `Need` 和 `Work`。
3. 进程完成后释放的是 `Allocation`，不是 `Need`。

### 7.5 对应例题：作业四第 1 题，P2 请求能否分配

题目：

```text
如果进程 P2 提出请求 Request[2] = (1, 2, 2, 2)，
系统能否将资源分配给它？试说明原因。
```

答案：

```text
不能真正分配。
虽然 Request[2] <= Need[2] 且 Request[2] <= Available，
但假设分配后系统不安全，所以必须回滚，让 P2 等待。
```

解答过程：

先列数据：

```text
Request[2] = (1,2,2,2)
Need[2]    = (2,3,5,6)
Available  = (1,6,2,3)
```

合法性检查：

```text
Request[2] <= Need[2]       成立
Request[2] <= Available     成立
```

所以可以进入“假设分配”。

假设分配后：

```text
Available' = (1,6,2,3) - (1,2,2,2)
           = (0,4,0,1)

Allocation'[P2] = (1,3,5,4) + (1,2,2,2)
                = (2,5,7,6)

Need'[P2] = (2,3,5,6) - (1,2,2,2)
          = (1,1,3,4)
```

此时做安全性检测：

```text
Work = (0,4,0,1)
```

逐个检查：

| 进程 | Need | 是否 `Need <= Work` | 原因 |
|---|---:|---|---|
| P0 | 0 0 1 2 | 否 | C、D 不够 |
| P1 | 1 7 5 0 | 否 | A、B、C 不够 |
| P2 | 1 1 3 4 | 否 | A、C、D 不够 |
| P3 | 0 6 5 2 | 否 | B、C、D 不够 |
| P4 | 0 6 5 6 | 否 | B、C、D 不够 |

找不到任何一个能完成的进程，因此假设分配后系统不安全。

结论：

```text
不能分配给 P2，应取消假设分配，P2 等待。
```

【一句话记法】

```text
Request 够不够只是第一关；分出去后安全不安全才是最终判定。
```

## 8. 死锁检测算法

### 8.1 知识点速记

死锁检测与银行家算法不同：

| 对比 | 银行家算法 | 死锁检测 |
|---|---|---|
| 目的 | 避免进入不安全状态 | 判断当前是否已经死锁 |
| 使用矩阵 | `Need` | `Request` |
| 核心比较 | `Need <= Work` | `Request <= Work` |
| 结果 | 安全/不安全 | 无死锁/哪些进程死锁 |

【会算】死锁检测步骤：

```text
Work = Available
Finish[i] = false

寻找某个 Pi：
  Finish[i] == false 且 Request[i] <= Work

若找到：
  Finish[i] = true
  Work = Work + Allocation[i]

重复。

若最后所有 Finish = true：无死锁。
若最后有 Finish = false：这些进程参与死锁。
```

【易错】检测算法不是看最大需求，不需要 `Max`。

### 8.2 对应例题：作业四第 2 题，初始状态与请求序列

题目：

```text
某系统采用死锁检测手段以发现死锁。
资源类集合为 {A,B,C}，资源 A 共有 8 个实例，B 有 6 个，C 有 5 个。
进程集合为 {P1,P2,P3,P4,P5,P6}。

某时刻系统状态如下：

|        | Allocation | Request   | Available |
|--------|------------|-----------|-----------|
|        | A B C      | A B C     | A B C     |
| P1     | 1 0 0      | 0 0 0     | 2 2 1     |
| P2     | 3 2 1      | 0 0 0     |           |
| P3     | 0 1 2      | 2 0 2     |           |
| P4     | 0 0 0      | 0 0 0     |           |
| P5     | 2 1 0      | 0 3 1     |           |
| P6     | 0 0 1      | 0 0 0     |           |

（1）系统依次接收请求：
Request[1]=(1,0,0), Request[2]=(2,1,0), Request[4]=(0,0,2)。
给出系统状态变化的情况，并说明没有死锁。
```

答案：

```text
接收这些请求后，仍然可以找到完成序列，因此没有死锁。
一个可行完成序列为 <P4, P1, P2, P3, P5, P6>。
```

解答过程：

如果只是接收请求、尚未分配资源，则 `Allocation` 不变，`Available` 仍为：

```text
Work = Available = (2,2,1)
```

新的 `Request` 可理解为：

| 进程 | Allocation A B C | Request A B C |
|---|---:|---:|
| P1 | 1 0 0 | 1 0 0 |
| P2 | 3 2 1 | 2 1 0 |
| P3 | 0 1 2 | 2 0 2 |
| P4 | 0 0 0 | 0 0 2 |
| P5 | 2 1 0 | 0 3 1 |
| P6 | 0 0 1 | 0 0 0 |

检测过程可以这样写：

1. `P4 Request=(0,0,2)` 需要 C=2，但 `Work=(2,2,1)` 不够。先找别的进程。
2. `P1 Request=(1,0,0) <= (2,2,1)`，P1 可完成：

```text
Work = (2,2,1) + Allocation[P1](1,0,0)
     = (3,2,1)
```

3. `P2 Request=(2,1,0) <= (3,2,1)`，P2 可完成：

```text
Work = (3,2,1) + (3,2,1)
     = (6,4,2)
```

4. `P3 Request=(2,0,2) <= (6,4,2)`，P3 可完成：

```text
Work = (6,4,2) + (0,1,2)
     = (6,5,4)
```

5. `P5 Request=(0,3,1) <= (6,5,4)`，P5 可完成：

```text
Work = (6,5,4) + (2,1,0)
     = (8,6,4)
```

6. `P6 Request=(0,0,0)`，P6 可完成：

```text
Work = (8,6,4) + (0,0,1)
     = (8,6,5)
```

7. 此时 `P4 Request=(0,0,2) <= (8,6,5)`，P4 也可完成。

所以所有进程都能完成，无死锁。

【说明】深度版和习题课给出过 `<P4, P1, P2, P3, P5, P6>` 这样的序列。检测题的安全/完成序列不唯一，考试时只要你的每一步 `Request <= Work` 成立即可。

【易错排雷】

1. 这是死锁检测，不是银行家算法，不能用 `Need`。
2. 找到一个 `Request <= Work` 的进程后，释放的是它的 `Allocation`。
3. `Request=(0,0,0)` 的进程通常可以立即完成并释放资源。

### 8.3 对应例题：作业四第 2 题，后续请求导致死锁

题目：

```text
在（1）确定的状态下，系统接收 Request[1]=(0,3,1)，
说明此时易发生死锁，并找出参与死锁的进程。
```

答案：

```text
若系统已经把 1 个 A 分配给 P1 后，P1 又请求 (0,3,1)，
检测算法可能结束在 Finish=false。
按该状态推演，参与死锁的进程为 P1、P2、P3、P5。
```

解答过程：

按习题课中“已经将 1 个 A 分配给 P1”的状态理解：

```text
P1 Allocation = (2,0,0)
P1 Request    = (0,3,1)
Available     = (1,2,1)
```

其他关键请求保持为：

| 进程 | Allocation A B C | Request A B C |
|---|---:|---:|
| P1 | 2 0 0 | 0 3 1 |
| P2 | 3 2 1 | 2 1 0 |
| P3 | 0 1 2 | 2 0 2 |
| P4 | 0 0 0 | 0 0 2 |
| P5 | 2 1 0 | 0 3 1 |
| P6 | 0 0 1 | 0 0 0 |

初始：

```text
Work = (1,2,1)
```

先找能完成的进程：

1. `P6 Request=(0,0,0) <= Work`，P6 可完成：

```text
Work = (1,2,1) + (0,0,1)
     = (1,2,2)
```

2. `P4 Request=(0,0,2) <= Work`，P4 可完成，但 P4 没有占有资源：

```text
Work = (1,2,2) + (0,0,0)
     = (1,2,2)
```

此后检查剩余进程：

| 进程 | Request | 是否可满足 | 原因 |
|---|---:|---|---|
| P1 | 0 3 1 | 否 | B 不够 |
| P2 | 2 1 0 | 否 | A 不够 |
| P3 | 2 0 2 | 否 | A 不够 |
| P5 | 0 3 1 | 否 | B 不够 |

没有新的进程能完成，因此检测结束。

```text
Finish=false 的进程：P1、P2、P3、P5
```

这些进程参与死锁。

【易错排雷】

1. 死锁检测最后看 `Finish=false`，不是看谁的 `Request` 最大。
2. P4、P6 能完成，不属于最终死锁集合。
3. 如果题目状态有“请求已接收但未分配”和“已分配一个 A 给 P1”两种版本，要按题面最终表推演，不能凭记忆写答案。

【一句话记法】

```text
检测题最后剩下谁走不了，谁就是死锁集合。
```

## 9. 死锁与饥饿

### 9.1 知识点速记

【必背】饥饿：

```text
进程长期得不到所需资源或服务，等待时间没有上界，导致推进或响应受到明显影响。
```

饥饿可以发生在：

1. 排队等待中。
2. 忙式等待中。

忙式等待条件下发生的饥饿常称为活锁。

### 9.2 对应例题：真题简答，死锁与饿死异同

题目：

```text
死锁与饿死之间有何相同点和不同点？
```

答案模板：

```text
相同点：
二者都可能由资源竞争或调度/分配策略不合理引起；
都会导致进程长时间不能推进；
都可能使进程任务无法及时完成。

不同点：
死锁是一组进程互相等待对方占有且不会释放的资源；
死锁进程通常处于等待状态，一定存在循环等待，至少涉及两个进程；
饥饿是某进程长期得不到所需资源或处理机，资源可能会释放但总是不分给它；
饥饿进程不一定处于等待状态，也不一定存在循环等待，甚至可能只有一个进程饥饿。
```

解答过程：

这题不要只写“都会等待”。要从以下角度区分：

| 对比 | 死锁 | 饥饿 |
|---|---|---|
| 本质 | 互相等待导致僵局 | 长期得不到机会 |
| 等待对象 | 其他死锁进程占有的资源 | 可能会释放但总被别人拿走的资源/处理机 |
| 进程状态 | 通常处于等待状态 | 可能等待，也可能忙等 |
| 循环等待 | 一定有 | 不一定有 |
| 涉及进程数 | 至少两个 | 可能一个 |
| 检测 | 较容易用等待关系检测 | 不一定容易检测 |

【一句话记法】

```text
死锁是互相等死；饥饿是一直轮不到我。
```

## 10. 同种组合资源死锁条件

### 10.1 知识点速记

课内结论：

```text
M：该类资源总数
N：竞争该类资源的进程数
Σ：所有进程对该类资源的最大需求总和
```

如果：

```text
Σ < M + N
```

则一定不会发生死锁。

如果：

```text
Σ >= M + N
```

则至少存在某种资源分配方式可能发生死锁。

### 10.2 对应例题：打印机题

题目：

```text
某计算机系统中有 8 台打印机，由 K 个进程竞争使用，
每个进程最多需要 3 台打印机。
该系统可能会发生死锁的 K 的最小值是多少？
```

答案：

```text
K = 4。
```

解答过程：

这里：

```text
M = 8
N = K
Σ = 3K
```

要“可能死锁”，需要：

```text
Σ >= M + N
3K >= 8 + K
2K >= 8
K >= 4
```

因此最小值：

```text
K = 4
```

构造理解：

4 个进程每个最多需要 3 台打印机。如果每个进程先占有 2 台，则 8 台全部被占完；每个进程还差 1 台才能继续，于是都等不到剩余打印机，可能死锁。

【易错排雷】

1. 不是算 `8/3`，不是问最多能满足几个进程。
2. 题目问“可能发生死锁的最小 K”，要用死锁可能条件。
3. 结论是“可能”，不是说 K=4 时任何调度都会死锁。

## 11. 资源活动序列静态分析

### 11.1 知识点速记

适用题型：

```text
题目给出每个进程申请/释放资源的活动序列，
要求判断并发执行时是否有死锁可能。
```

分析步骤：

1. 找出每个进程中“已经占有某些资源，同时正在申请新资源”的状态。
2. 把每个状态写成：

```text
(Pi : 请求资源 : 已占有资源集合)
```

3. 如果状态 S1 请求的资源被状态 S2 占有，就画 `S1 -> S2`。
4. 找等待环。
5. 找到环后，还要判断环上的状态是否可能同时到达。

【易错】看到环不一定就有死锁可能，因为环上的状态可能不可能同时出现。

常见不可同时到达原因：

| 原因 | 解释 |
|---|---|
| 环中有同一进程的两个不同状态 | 一个进程不能同时处于两个执行位置 |
| 环中同一个独占资源被两个状态占有 | 一个资源实例不能同时分给两个进程 |

### 11.2 对应例题：作业四第 4 题

题目：

```text
设有 7 个简单资源 A~G，其申请命令为 a~g，释放命令为 a'~g'。
系统中有 P1、P2、P3 三个进程，其活动分别为：

P1: a b a' b' e f g e' f' g'
P2: b c b' c' d a d' a'
P3: c d c' d' e g f e' f' g'

分析它们并发执行时，是否有死锁可能性，并说明原因。
```

答案：

```text
没有死锁可能。
```

解答过程：

先列出关键“占有资源并继续申请资源”的状态：

```text
(P1 : b : a)
(P2 : c : b)
(P3 : d : c)
(P2 : a : d)
(P1 : f : e)
(P3 : g : e)
(P1 : g : e f)
(P3 : f : e g)
```

根据“请求资源被谁占有”画等待边，会发现可能存在环路。但作业/习题课结论是：这些环路都不能形成真实死锁。

原因主要有两个：

1. 某个环路包含同一进程 P2 的两个不同状态。  
   同一个进程不可能同时执行到两个不同位置，所以该环路不可达。

2. 另一个环路要求同一个资源 `e` 同时被不同状态占有。  
   简单资源是独占资源，一个资源不能同时分配给两个进程，所以该环路不可达。

因此虽然静态等待关系图中能看到环，但环路状态不能同时到达。

最终结论：

```text
三个进程并发执行时没有死锁可能。
```

【一句话记法】

```text
静态分析题：找环只是第一步，还要问这个环能不能真的同时发生。
```

## 12. 过河/道路类无死锁设计题

### 12.1 知识点速记

作业四第 3 题属于设计题：

```text
分析可能发生死锁的情况；
设计无死锁、无饿死、并行度高的解法；
用 PV 操作实现。
```

这类题结合第 4 章 PV 和第 5 章死锁思想。

答题思路：

1. 先说明可能死锁的情形。
2. 再说明破坏死锁条件的方法。
3. 最后写 PV 控制逻辑。

### 12.2 通用答题模板

【会写】分析死锁：

```text
若多个进程分别占有路径/资源的一部分，
又继续申请下一段资源，
可能形成循环等待，导致死锁。
```

【会写】无死锁设计：

可以从以下角度选一种或组合：

| 方法 | 破坏条件 |
|---|---|
| 规定统一方向/统一资源申请顺序 | 破坏循环等待 |
| 进入前一次性申请整条路径所需资源 | 破坏保持申请 |
| 限制系统中同时进入关键路径的人数 | 避免所有关键资源被占满 |
| 设置公平队列或方向切换 | 防止饥饿 |

【会写】PV 实现时要写：

1. 每段独占资源用互斥信号量，初值 1。
2. 若限制同时进入人数，设置计数信号量。
3. 若要求无饥饿，设置等待队列/方向优先级/轮换策略。
4. 获取资源顺序必须统一，释放顺序可反向释放。

【易错】只写“加锁”不够，要说明为什么不会死锁、为什么不会饿死、并行度如何保留。

## 13. 题型模板汇总

### 13.1 【题型】Coffman 条件判断

答题模板：

```text
死锁四个必要条件是：互斥、不可剥夺、保持申请、循环等待。
某策略通过____，破坏了____条件，因此可以预防死锁。
```

常见对应：

| 策略 | 破坏条件 |
|---|---|
| 资源有序分配 | 循环等待 |
| 一次性申请所有资源 | 保持申请 |
| 申请失败释放已有资源 | 保持申请 |
| 允许抢占资源 | 不可剥夺 |

### 13.2 【题型】银行家算法大题

固定步骤：

```text
1. 若给 Max，则先算 Need = Max - Allocation。
2. Work = Available，Finish 全 false。
3. 找 Need[i] <= Work 的进程 Pi。
4. Pi 完成后：Work = Work + Allocation[i]。
5. 重复直到全部完成或找不到。
6. 若全部完成，写安全序列；否则不安全。
7. 若有 Request：
   先判 Request <= Need；
   再判 Request <= Available；
   假设分配；
   跑安全性检测；
   安全则分配，不安全则回滚。
```

### 13.3 【题型】死锁检测大题

固定步骤：

```text
1. Work = Available。
2. Finish 初始化。
3. 找 Request[i] <= Work 的进程 Pi。
4. Pi 完成后：Work = Work + Allocation[i]。
5. 重复直到找不到。
6. 最后 Finish=false 的进程参与死锁。
```

### 13.4 【题型】死锁 vs 饥饿简答

固定写法：

```text
相同：都使进程长期不能推进，通常和资源竞争/调度策略有关。
不同：
死锁是进程组互相等待，至少两个进程，一定有循环等待，通常处于等待态；
饥饿是进程长期得不到资源或处理机，不一定有循环等待，不一定处于等待态，可能只有一个进程。
```

### 13.5 【题型】资源活动序列静态分析

固定步骤：

```text
1. 写出“请求资源 : 已占有资源”的状态结点。
2. 请求资源被谁占有，就画等待边。
3. 找环路。
4. 判断环路状态能否同时到达。
5. 若能同时到达，有死锁可能；若不能，无死锁可能。
```

## 14. 易错判断总表

| 说法 | 判断 | 理由 |
|---|---|---|
| 有等待就是死锁 | 错 | 正常阻塞不等于互相等待 |
| 死锁一定涉及系统所有进程 | 错 | 可能只是部分进程死锁 |
| 死锁四条件缺一不可 | 对 | Coffman 条件是必要条件 |
| 有序分配破坏保持申请 | 错 | 有序分配破坏循环等待 |
| 一次性申请所有资源破坏保持申请 | 对 | 不会占着一部分再申请另一部分 |
| 银行家算法属于死锁预防 | 错 | 属于死锁避免 |
| 安全状态一定没有死锁 | 对 | 存在完成序列 |
| 不安全状态一定死锁 | 错 | 只是可能死锁 |
| 死锁状态一定不安全 | 对 | 已经无完成序列 |
| 单实例资源分配图有环通常死锁 | 对 | 单实例环路无法靠其他实例打破 |
| 多实例资源分配图有环一定死锁 | 错 | 可能还有实例释放 |
| 银行家算法看 `Request <= Available` 就够了 | 错 | 还要假设分配后安全 |
| 银行家安全性检测看 `Allocation <= Work` | 错 | 看 `Need <= Work` |
| 死锁检测看 `Need <= Work` | 错 | 看当前 `Request <= Work` |
| `Need > 0` 表示进程正在等待 | 错 | Need 是未来需求，不是当前请求 |
| 检测结束 `Finish=false` 的进程参与死锁 | 对 | 它们无法在当前状态完成 |
| 饥饿就是死锁 | 错 | 饥饿不一定有循环等待 |
| 饥饿进程一定处于等待态 | 错 | 可能忙式等待 |
| 看到静态分析图有环就一定死锁 | 错 | 还要判断环是否可同时到达 |
