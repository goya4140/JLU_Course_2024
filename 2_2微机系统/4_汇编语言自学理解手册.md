# 第4章 汇编语言及其程序设计
## 从零建立理解——自学手册

> **这份资料的目标**：不是让你背住每条指令的格式，而是让你理解"汇编语言到底在干什么、为什么要长这个样子、各个部分之间怎么咬合在一起"。读完之后，你应该能把第2章学的寄存器、第2章学的物理地址转换，跟这一章的每一条指令自然地串起来，而不是把汇编语言当成一门孤立的新语言去死记硬背。

---

## 第一部分：先问一个问题——为什么不直接写机器码？

8086能识别的，本质上只有一种东西：一串01组成的二进制机器码。CPU读到这串二进制，按照固定的编码规则去解析出"做什么操作、操作哪个寄存器、操作哪个地址"，然后执行。

但人类写二进制太痛苦了。`10110000 01100001` 是什么意思？谁也记不住。于是产生了**汇编语言**：用有意义的英文助记符（如 `MOV AL, 61H`）代替这串二进制，让程序员能读懂、能写出。

但CPU不认识"MOV"这个单词，所以需要一个翻译官——**汇编程序（汇编器，如MASM）**，把人写的助记符翻译回机器码。

这就是为什么第4章一开始要讲清楚这几个概念的关系：

```
人写的：MOV AL, 61H                    （汇编语言源程序 .ASM）
            │
            │  汇编程序 MASM 翻译
            ↓
机器码：10110000 01100001            （目标文件 .OBJ）
            │
            │  连接程序 LINK 链接（处理多模块、库函数）
            ↓
可执行文件：xxx.EXE                    （CPU能直接运行）
```

记住这条流水线，因为第4章后面会反复出现"汇编时处理"和"运行时执行"这两个不同的阶段——伪指令属于前者，机器指令属于后者，这是整章最容易混淆、也最常考的一对概念。

带着这个视角，我们来看汇编语言这门"半人类半机器"的语言，到底是怎么设计出来的。

## 第二部分：一条指令的解剖——汇编语句的四个字段

每一条汇编语句，无论简单复杂，都遵循同一个模板：

```
[名字]   操作符   操作数  ；[注释]
```

这四个字段分别对应不同的角色，理解了角色分工，你就理解了为什么语法长这样：

**名字**：相当于给这一行起个"门牌号"，方便其他地方通过这个名字找到它。如果这一行是指令，名字叫**标号**（label）——就像给一个路口取名字，后面写`JMP 这个名字`就能跳到这里。如果这一行是数据定义，名字叫**变量**——就像给一个储物柜贴标签，后面用这个名字就能存取里面的东西。

**操作符**：这一行真正要做的"动作"，可以是CPU指令（MOV、ADD……）、伪指令（DB、SEGMENT……）或宏指令。

**操作数**：动作要施加的对象——是哪个寄存器？哪个内存单元？还是一个立即数？

**注释**：纯粹给人看的，CPU和汇编器都会忽略分号后面的内容。

这个四字段结构有几条命名规则（不能数字开头、前31个字符有效、不能是关键字），属于语法细节，不必死记，写多了自然就记住了。真正重要的是理解角色：**标号指向代码，变量指向数据**——这个区分会在后面"返回值运算符TYPE"那一节直接决定考点（标号的TYPE返回-1或-2，变量的TYPE返回字节数）。

## 第三部分：指令最核心的难题——怎么找到操作数？（寻址方式）

这是整个第4章的骨架，也是后续所有指令、所有题目共同依赖的基础设施。先建立直觉，再上公式。

### 3.1 一个朴素的问题

CPU执行一条指令，比如 `MOV AX, ？`，它需要知道"？"这个操作数到底在哪里。可能的情况无非这几种：

- 操作数直接写在指令里（比如 `MOV AX, 100`）——这叫**立即寻址**，操作数就在指令本身里，不用去任何地方找。
- 操作数在某个寄存器里（比如 `MOV AX, BX`）——这叫**寄存器寻址**，直接读寄存器即可，不需要访问内存，速度最快。
- 操作数在内存里——这种情况最复杂，因为内存有1MB（甚至更大），怎么算出具体在哪个字节，需要一套规则。这就是**存储器寻址**，第4章90%的篇幅在讲它。

### 3.2 存储器寻址的核心公式——有效地址 EA

回忆一下第2章学过的：**物理地址 = 段基址×16 + 偏移地址**。这个"偏移地址"，在第4章里有了一个正式的名字：**有效地址（Effective Address, EA）**。

有效地址不是凭空冒出来的，它是由指令里写的寻址表达式**计算**出来的，公式是：

$$EA = 基址 + 变址 \times 比例因子 + 位移量$$

这四个成分，每一个都有它的"专长"：

- **基址寄存器**（BX、BP）：通常指向一片数据区域的**起始位置**（比如数组首地址），类比你站在书架前，先确定"这是第几排书架"。
- **变址寄存器**（SI、DI）：通常表示在这片区域内**走了多远**（比如数组下标），类比你在这排书架上"数到第几本书"。
- **比例因子**（1/2/4/8，32位机才有）：处理数组元素本身占多个字节的情况（比如int数组每个元素4字节，下标×4才是真正的字节偏移）。
- **位移量**：指令里写死的一个常数偏移，比如 `[BX+4]` 里的4。

四个成分不是必须同时出现，最多保留每类一项，按"出现了哪些成分"就组合出了课件表4.4.1里列出的9种存储器寻址方式（直接寻址只有位移量、寄存器间接寻址只有基址或变址、基址变址寻址是基址+变址……）。**与其逐个背名字，不如学会反向判断**：拿到一条指令，看方括号里出现了什么成分，名字自然就推出来了。

### 3.3 一个极易踩的陷阱——默认段是谁？

算出EA只是拿到了"偏移地址"，要凑成完整的物理地址，还需要知道"配哪个段寄存器"。这里有一条默认规则，是整章选择题最爱挖的坑：

```
使用 BP 或 SP 寻址   →  默认配 SS（堆栈段）
使用 BX、SI、DI 寻址 →  默认配 DS（数据段）
取指令               →  默认配 CS（代码段）
串操作目的地址       →  默认配 ES（附加段）
```

为什么BP要单独配SS？因为BP（Base Pointer）这个寄存器的设计初衷就是**访问堆栈里的内容**（比如子程序里通过BP读取调用者传进来的参数），堆栈本身就在SS段里，所以这个默认关系不是巧合，而是寄存器的功能定位决定的。

记住这条规则之后，任何形如 `[BP+...]` 的寻址，物理地址公式自动变成 $16 \times (SS) + EA$，而不是想当然的DS——这正是真题第8题考的内容。

### 3.4 操作数确定了，CPU还要去哪里？——转移地址的寻址

上面讲的是"数据"在哪里，但JMP、CALL这类指令要找的不是数据，而是"**下一条要执行的指令**在哪里"，这叫**转移地址寻址**。

逻辑上分两个维度：

**维度一：段内还是段间？**
- 段内转移：目标和当前指令在同一个代码段，只需要改IP，不用动CS。
- 段间转移：目标在另一个代码段，IP和CS都要改。

**维度二：怎么算出目标地址？**
- 相对寻址：指令里给一个"相对偏移量"（带符号数），新IP = 旧IP + 偏移量。只能用于条件转移（JZ、JNZ这些），因为条件转移指令编码空间有限，只能塞下一个相对偏移。
- 直接寻址：指令里直接写死目标地址。
- 间接寻址：目标地址存在寄存器或内存里，需要先读出来。

把这两个维度交叉一下，就是课件里JMP指令的5种形式（段内相对/段内间接/段间直接/段间间接），本质上是同一套"相对 vs 直接 vs 间接"逻辑在段内、段间两种场景下的具体落地。

## 第四部分：堆栈——一种特殊的、有纪律的内存区域

讲完了普通的内存寻址，堆栈值得单独拎出来讲，因为它有自己的一套规矩，且贯穿了CALL/RET、PUSH/POP、中断处理全部内容。

### 4.1 堆栈是什么，为什么要有它

堆栈本质上就是一段普通内存，但被赋予了"先进后出"（像摞盘子，最后放的盘子最先拿走）的使用规则。它存在的意义是解决一个具体问题：**子程序调用完要返回原处，原处的地址需要被"记住"，记在哪里？**——记在堆栈里。同理，调用子程序前如果某些寄存器的值还有用，也需要"先存起来，用完再恢复"，这也用堆栈实现。

### 4.2 三条铁律

8086对堆栈做了三条强制规定，每一条都有具体的理由：

1. **堆栈向小地址方向增长**——这意味着每PUSH一次，栈顶指针SP的值会变小，不是变大。
2. **必须用SS段寄存器**——堆栈本质上是SS段里的一片连续内存，SP是这个段内的偏移量。
3. **PUSH先改指针再存数据，POP先取数据再改指针**——这个顺序细节决定了"边界情况"下指针指向哪里是确定的，不会有一刻指向"无效"位置。

### 4.3 PUSH的具体步骤（容易被问到执行细节）

以 `PUSH DI` 为例，按16位操作分解：

```
①SP ← SP-1     （先挪指针，腾出空间）
②存DI的高8位到 SP 指向的单元
③SP ← SP-1     （再挪一次）
④存DI的低8位到 SP 指向的单元
```

POP的顺序完全相反：先按当前SP取出低字节，SP+1；再取高字节，SP+1。

**这里有个值得记住的细节**：堆栈里存的字节顺序，和内存里DW定义数据的小端存储顺序是一致的——低地址存低字节。理解了这一点，PUSH/POP的字节顺序细节就不需要死记，跟着"小端"这条总原则推一遍就出来了。

## 第五部分：指令系统——按"做什么用"分类，而不是死记列表

8086的指令多达上百条，课件把它们分成10类。与其按字母表去硬背，不如理解每一类存在的**目的**，指令的具体语法会随着做题自然熟练。

### 5.1 数据传送类——"搬运工"

MOV、PUSH/POP、XCHG、LEA、LDS/LES、XLAT、IN/OUT，干的都是"把数据从一个地方挪到另一个地方"，区别只在"挪去哪里、怎么挪"：

- MOV是最通用的搬运，但有一条隐藏限制：**源和目的不能同时是内存操作数**（CPU一次只能处理一个内存地址），这正是练习题里"`ADD VAR12, VAR34`"被判错误的根本原因——ADD和MOV是同一条限制。
- LEA很特殊：它搬的不是数据本身，而是这个数据**所在的地址**。类比你问路时，别人不是把目的地"搬"给你，而是把地址告诉你——LEA干的就是"告诉你地址"这件事，这也是它和MOV的根本区别（`MOV AX, MEM` 是取值，`LEA AX, MEM` 是取地址）。
- XLAT是"查表翻译"，常用于把数字转ASCII，本质上是一种特殊形式的寄存器间接寻址（用AL的值作偏移量去查表）。

### 5.2 算术运算类——"计算器"

ADD/ADC/SUB/SBB/INC/DEC/CMP/MUL/DIV，对应小学数学的加减乘除，唯一要注意两点：

- ADC/SBB带了进位/借位（CF），是为了支持**多字节（超过16/32位）的大数运算**——比如两个64位数相加，先加低32位产生进位，再用ADC把这个进位加到高32位上，这就是带进位指令存在的意义。
- INC/DEC影响除CF外的所有标志位，**唯独不影响CF**——这是因为INC/DEC常被用在循环计数器上，如果它们会动CF，循环体里万一刚好有依赖CF的判断（比如紧跟着一条JC），就会被意外打乱，所以设计上特意保留CF不受影响。

### 5.3 BCD码调整类——为十进制运算"打补丁"

CPU天生只会做二进制加减法，但人类记账要用十进制（BCD码）。DAA/DAS/AAA/AAS这4条指令的作用，是在做完二进制ADD/SUB之后，**修正**结果让它符合十进制进位规则（逢9进1而不是逢15进1）。所以这几条指令永远紧跟在ADD/SUB后面用，绝不会单独出现。

### 5.4 逻辑运算与移位循环类——"位操作工具箱"

AND/OR/XOR/TEST/NOT 处理逻辑位运算，SHL/SHR/SAL/SAR/ROL/ROR/RCL/RCR 处理移位。这里有个常考的对比维度——逻辑移位 vs 算术移位：

```
SHR（逻辑右移）：最高位补0，不管原来是不是负数
0  → [Dn ... D0] →  CF
```

```
SAR（算术右移）：最高位补"符号位本身"，保持负数还是负数
[符号位复制] → [Dn ... D0] → CF
```

为什么要区分？因为逻辑右移用于**无符号数**（补0天经地义），算术右移用于**有符号数**（要保持负数的符号不变，否则一个负数右移几次就变成正数了，数值含义就错了）。SHL和SAL实际上是同一条指令的两个名字，没有区别。

### 5.5 控制转移类——"导航系统"

这一类最庞杂，但分成三条主线就清楚了：

**主线一：无条件跳转（JMP）**——见第三部分3.4，按"段内/段间"和"相对/直接/间接"组合。

**主线二：条件跳转**——本质上都是"读某个标志位，满足条件就跳"。这里有个常考的对照表，建议理解记忆而非死记：

```
判断"相等/不等"：JZ/JNZ           （直接看ZF）
判断"正负"：     JS/JNS            （直接看SF）
判断"溢出"：     JO/JNO            （直接看OF）
判断"无符号大小"：JA/JB/JAE/JBE    （看CF和ZF的组合）
判断"有符号大小"：JG/JL/JGE/JLE    （看SF异或OF，再结合ZF）
```

无符号比较只看CF，因为无符号数没有"符号"概念，进位/借位直接反映大小关系；有符号比较要看SF⊕OF，是因为单看SF会被溢出"骗"——比如最大正数加1变成最小负数，SF从0变1，但实际上数值是"变大"了，必须用OF修正这种误判。

**主线三：子程序调用与中断**——CALL/RET和INT/IRET是同一套思想在两个场景下的应用：都是"跳走之前先把返回地址存起来，回来时再取出来跳回去"。区别在于：

- CALL/RET只存IP（和CS），因为子程序执行完，调用现场的标志位状态通常不需要被破坏（也没人主动改它）。
- INT/IRET除了存IP、CS，还要多存一份**FLAGS**——因为中断是被动触发的，中断服务程序内部很可能会做运算（从而改变标志位），中断返回后，原程序还要继续依赖中断发生前的标志位状态，所以必须额外保护现场。

这就是为什么"RET和IRET的区别"是真题简答题——它考的不是死记"多弹一个FLAGS"，而是理解"为什么中断需要多保护一份现场，子程序不需要"。

## 第六部分：伪指令——汇编时的"施工图纸"

机器指令是运行时CPU真正执行的东西；伪指令则完全不同，它是写给**汇编程序**看的，告诉汇编程序"怎么组织这份源代码"，本身不产生任何机器码。理解了这个本质区别，伪指令的一堆名目就不再是孤立的语法规则，而是"施工图纸"上不同的标注符号。

### 6.1 段定义——划分施工区域

`SEGMENT...ENDS` 把源代码切成若干段（数据段、堆栈段、代码段），`ASSUME` 告诉汇编程序"哪个段寄存器对应哪个段"。这一步纯粹是给汇编程序看的说明书，CPU运行时根本不知道"段名"这回事，它只认段寄存器里实际的数值。

### 6.2 数据定义——预留仓库空间

DB/DW/DD分别预留1/2/4字节的空间，`?`表示"留空不初始化"，`DUP`表示"重复填充"。这部分需要特别小心的是**地址计数器`$`的递增规则**：每定义一项就把`$`往后推进相应字节数，所以同一段里后面定义的变量，偏移地址是前面所有变量字节数的累加——这也是为什么"求某变量的OFFSET"这类题目，必须从段首开始，把前面所有变量占的字节数老老实实数一遍。

### 6.3 返回值运算符——汇编时就能算出来的"元信息"

TYPE/LENGTH/SIZE/OFFSET，这几个运算符的值是**汇编阶段**就能确定的（不需要CPU运行），它们分别回答四个不同的问题：

- OFFSET：这个变量的地址偏移量是多少？
- TYPE：这个变量每个元素占几个字节？（DB→1，DW→2，DD→4）
- LENGTH：这个变量用DUP定义了几次？（没用DUP，固定返回1，**不是元素个数**，这是最容易踩的坑——`DB 'ABCD'`的LENGTH是1，不是4）
- SIZE：= TYPE × LENGTH，即这个变量总共占多少字节。

### 6.4 PROC/ENDP——给子程序"挂牌"

`过程名 PROC [属性]` 和 `过程名 ENDP` 配对使用，本质上只是给一段代码起一个名字、标出起止范围，方便CALL语句引用。属性NEAR/FAR决定了调用方式是段内还是段间——这一点和前面"转移地址寻址"的段内/段间逻辑完全一致，没有产生新概念，只是换了个使用场景。

## 第七部分：把碎片拼回完整的程序——EXE源程序结构

学完前六部分的"零件"，现在拼装成一台完整的"机器"——一份能跑起来的EXE程序。

一份标准EXE程序固定是三段结构：

```
DSEG SEGMENT 'DATA'      ← 数据段：变量、提示字符串
    ...
DSEG ENDS

SSEG SEGMENT STACK 'STACK'  ← 堆栈段：预留堆栈空间
    DW 100H DUP(?)
SSEG ENDS

CSEG SEGMENT 'CODE'       ← 代码段：实际执行的指令
    ASSUME CS:CSEG, DS:DSEG, SS:SSEG
START PROC FAR
    ...（程序体）
START ENDP
CSEG ENDS
    END START
```

有两处新手最容易漏掉，背后都有具体原因：

**为什么开头要 `PUSH DS / MOV AX,0 / PUSH AX`？**

因为这份程序的入口被定义成 `FAR` 过程，意味着程序最后会执行一条 `RET`（段间返回），段间RET要从栈顶弹出CS和IP两个值。如果不预先手动压栈一组"返回DOS的地址"，最后这条RET会弹出一堆垃圾值，程序直接跑飞。这两行代码本质上是在"伪装"成被CALL过一样，给RET一个合法的返回目标。

**为什么要手动 `MOV AX,DSEG / MOV DS,AX`？**

因为CPU启动时不会自动帮你把DS指向DSEG——ASSUME伪指令只是"告诉汇编程序"段对应关系，不会生成真正赋值DS的机器码。这行代码必须由程序员手动补上，否则DS里是个不确定的初始值，所有访问数据段的指令都会出错。

理解了这两点的"为什么"，EXE模板就不再是死记的八股文，而是逻辑自洽的一套结构。

## 第八部分：程序设计的四种骨架

第4章最后落到"怎么写出一个完整程序"，课件给出顺序、分支、循环、子程序四种结构，对应着写程序时遇到的四类基本需求：

**顺序结构**：什么都不判断，一条接一条执行——最简单，是默认状态。

**分支结构**：根据条件走不同的路。底层机制就是第五部分讲的"条件跳转指令"，通过CMP产生标志位，再用JZ/JG等指令决定走哪条路。

**循环结构**：重复做同一件事直到满足条件。最常用LOOP指令（CX自动减1并判断是否为0）。循环程序的五个组成部分（初始化、循环体、修改、控制、结束），本质上是把"重复"这件事拆解成"先设置好初值，每次跑一遍，跑完改一下计数器，判断够不够，够了就跳出"——这个套路熟悉之后，看到任何循环题都能按这个骨架去套。

**子程序结构**：把一段独立功能封装起来反复调用，依赖第三、四部分讲过的CALL/RET和堆栈机制实现"跳走-记住返回点-跳回"的完整闭环。

### 一个贯穿全章的例子——冒泡排序

冒泡排序之所以是课件反复出现的范例，是因为它**同时用到了本章几乎所有核心机制**：

```
外循环（DX控制遍数）—— 循环结构
  内循环（CX控制比较次数）—— 嵌套循环
    取数 MOV AL,[BX] —— 寄存器间接寻址
    比较 CMP AL,[BX+1] —— 寄存器相对寻址 + 标志位产生
    条件跳转 JNC —— 分支结构，依赖CF判断无符号大小
    交换 XCHG + MOV —— 数据传送指令
    INC BX —— 修改部分，地址递增
  LOOP 内循环判断 —— 循环控制
DEC DX / JNZ 外循环判断 —— 嵌套循环控制
```

如果你能独立把冒泡排序的逻辑讲清楚、自己默写出来，基本上意味着你已经把第4章的核心机制串成了一条线，而不是记住了一堆孤立的指令。

## 第十部分：常见的理解误区

**误区1：以为寻址方式名字要死记**

寻址方式的名字是"结果"，不是"原因"。先看方括号里有基址、变址、位移量中的哪几项，名字自动就能推出来，不需要死记9个名词对应9种组合。

**误区2：以为`[BX+DI]`这类寻址默认配DS**

只要基址寄存器（不是变址寄存器）出现的是BP，默认段就是SS。`[BP+DI]`配SS，`[BX+DI]`才配DS——区分点在BX还是BP，不在DI/SI。

**误区3：以为LENGTH返回的是元素个数**

LENGTH只在使用了DUP时才有意义（返回DUP的重复次数）；像 `DB 'ABCD'` 这种没用DUP的定义，LENGTH固定返回1，哪怕字符串有4个字符。

**误区4：以为INC/DEC和ADD/SUB一样会影响CF**

INC/DEC刻意被设计成不影响CF，这是为了不干扰循环计数场景里可能存在的CF判断逻辑。

**误区5：以为RET和IRET只是"多弹一个数"的区别**

本质区别在于触发场景——子程序调用是主动的，调用前后标志位状态通常不需要保护；中断是被动的，中断服务程序很可能改变标志位，必须额外保护和恢复FLAGS，这才是IRET要多弹一个FLAGS的根本原因。

**误区6：以为EXE程序开头的PUSH DS/0是可有可无的样板代码**

它是在为最后的段间RET伪造一个合法返回地址，没有这两行，程序执行到RET时栈顶不是有效的CS:IP，会直接跑飞。
