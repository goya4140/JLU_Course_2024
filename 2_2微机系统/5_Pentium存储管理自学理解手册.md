# 第5章 Pentium 微处理器保护模式存储管理
## 从零建立理解——自学手册

> **这份资料的目标**：不是让你背住GDT、LDT、段描述符这些名词的定义，而是让你理解"保护模式这套机制到底在解决什么问题、为什么非要设计成这个样子、各个部件之间怎么咬合在一起"。读完之后，你应该能在脑子里完整画出一条"虚拟地址→线性地址→物理地址"的流水线，并且对每一步"为什么必须这样走"都能说出理由，而不是把GDT/LDT/IDT/TSS当成几个孤立的英文缩写去死记硬背。

## 知识体系总览

在进入正文前，先看清整章的分层逻辑——每一层在回答一个不同层次的问题，层层递进：

```
§5 Pentium保护模式存储管理

├─ 第一层：动机与矛盾（回答"为什么要有这套机制"）
│   ├─ 矛盾一：程序所需地址空间 vs 主存实际容量 → 虚拟存储器
│   ├─ 矛盾二：多任务并存 vs 互不干扰 → 保护模式
│   └─ 对比：Cache解决"速度"矛盾，虚拟存储器解决"容量"矛盾（⚠️易混）
│
├─ 第二层：操作模式（回答"什么时候启用这套机制"）
│   ├─ 实地址模式：处理器整机级，段基址×16+偏移，不查表
│   ├─ 保护模式：本章主线，完整启用GDT/LDT/分页
│   └─ 虚拟8086模式：任务级（非处理器级），运行在保护模式内部（✅真题易混辨析）
│
├─ 第三层：地址与段的数据结构（回答"信息长什么样"）
│   ├─ 虚拟地址＝16位段选择符＋32位偏移量（共48位，2⁴⁶=64T空间）
│   ├─ 段选择符＝13位INDEX＋1位TI（0选GDT/1选LDT）＋2位RPL
│   └─ 段描述符＝8字节（32位基址＋20位界限＋P/DPL/S/TYPE/G/D-B/AVL）
│
├─ 第四层：分段寻址过程（回答"虚拟地址→线性地址怎么查"）
│   ├─ GDTR：32位基址+16位界限，全系统唯一，指向GDT
│   ├─ LDTR：16位选择符+Cache(32位基址/20位界限/12位属性)，每任务一份
│   ├─ 三步法：选择符INDEX×8+表基址→读出描述符→Cache基址+偏移＝线性地址
│   └─ LDT比GDT多一道前置步骤：先经GDT查出LDT本身位置（嵌套查表）
│
├─ 第五层：分页寻址过程（回答"线性地址→物理地址怎么查"）
│   ├─ 二级页表：页目录(≤1024项)→页表(≤1024项)，CR3指定页目录基址
│   ├─ 32位线性地址拆分：10位页目录索引+10位页表索引+12位页内偏移
│   └─ 段页式整合：分段求线性地址→分页求物理地址，不分页时线性地址即物理地址（✅真题画图）
│
└─ 第六层：保护机制（回答"权限怎么管控"，寄生于第三、四层结构之上）
    ├─ 特权级三角：DPL(段自带门槛)/CPL(当前代码权限)/RPL(请求声明权限)，0级最高
    ├─ TSS：任务的完整寄存器现场快照，不含代码数据，由TR寻址，登记在GDT中
    └─ 门描述符：调用门(改特权级)/任务门(切任务)/中断门/陷阱门(定位中断服务程序)
```

这六层不是互相独立的知识点堆砌，而是一条因果链：第一层的"动机"决定了第二层"必须有保护模式"；保护模式一旦启用，就必须有第三层这套数据结构来描述段；有了数据结构，第四、五层才能讲清楚"怎么用这些结构把地址翻译出来"（分段接分页）；而第六层的保护机制，本质上是寄生在第三、四层已经建立的描述符体系之上的——DPL字段就长在段描述符里，TSS的切换靠的也是GDT中的TSS描述符。换句话说，**第六层没有引入任何全新的硬件部件，它只是在重新使用前面已经造好的零件，去实现"权限管控"这个目标**。这也是为什么本章最后一个真题画图题（段页式综合地址转换）几乎能把全章串起来——它就是第四、五层知识的完整复现。

## 第一部分：先问一个问题——为什么不能让程序直接用物理地址？

设想最朴素的情况：一台只跑一个程序的电脑，程序直接拿到的就是物理内存地址，想读哪个字节就直接读哪个地址，简单粗暴，毫无问题。

但现实中的计算机要同时跑很多个任务（操作系统本身、你的程序、后台服务……），这时候朴素方案立刻暴露出两个致命问题：

**问题一：内存"不够用"还是"装不下"？**

每个程序在设计时，往往希望自己拥有一大段连续、独占的地址空间来组织数据——但物理内存是有限的、且要被所有任务共享。如果每个程序都要求"我需要被实际安装在内存条上的一段连续物理空间"，那同时运行的任务数量就被物理内存条容量死死卡住了。

**问题二：任务之间怎么互不干扰？**

如果所有程序都能直接读写物理地址，那一个任务的bug（比如指针越界）就可能直接踩到另一个任务、甚至操作系统自己的内存区域，整个系统会变得极不稳定、毫无安全性可言。

**虚拟存储器**就是用来解决问题一的：它让"程序员编程时使用的地址"（虚拟地址/逻辑地址）和"实际硬件上的地址"（物理地址）分离开，中间插入一层软硬件协同的翻译机制。程序可以认为自己拥有一大片连续的虚拟空间，而实际数据在物理内存和磁盘（辅存）之间按需调入调出，这个过程对程序员透明——你不需要操心你写的数据此刻物理上到底在哪里。

| 存储体系 | 虚拟存储器 | （对比）Cache存储器 |
|---|---|---|
| 解决的矛盾 | 程序需要的空间 vs 实际安装的主存容量 | CPU的运算速度 vs 主存的访问速度 |
| 存储层次 | 主存-辅存 | Cache-主存 |
| 信息传送单位 | 段、页（较大） | 块、行（较小且固定） |

值得辨析的是，Cache和虚拟存储器经常被放在一起问，但它们解决的根本不是同一类矛盾：**Cache解决"速度"问题（CPU等内存等得太久），虚拟存储器解决"容量"问题（程序想要的空间比物理内存大）**。两者都靠"硬件+软件协同、对使用者透明"这套思路实现，但层次完全不同，不要混为一谈。

**保护模式**则是用来解决问题二的：它给每个段都贴上"权限标签"（段描述符里的DPL），并给每个任务划出专属的私有地址空间（LDT），从硬件层面阻止低权限代码随意访问高权限数据，也防止任务之间互相踩踏。

> 这两个问题、两个解决方案，就是整章所有后续内容的出发点：**虚拟存储器决定了"地址要经过翻译"，保护模式决定了"翻译过程中要带着权限检查"**。

## 第二部分：三种操作模式——什么时候才会用上这套机制？

Pentium有三种操作模式：**实地址模式、保护模式、虚拟8086模式**。理解它们的关系，能帮你想清楚"段页式这套复杂机制究竟是不是任何时候都在跑"。

可以用一个类比理解：

- **实地址模式**好比"整台电脑被临时切换回了一台老式兼容机"——此时Pentium的段寻址方式和8086完全一样，物理地址 = 段基址×16 + 偏移地址，根本不走描述符表、不查GDT/LDT，简单粗暴。这是处理器**整机层面**的工作方式。
- **保护模式**是Pentium真正发挥实力的现代工作方式，本章后续所有的GDT、LDT、段描述符、分页机制，全部只在这个模式下才会启用。
- **虚拟8086模式**则更微妙——它不是处理器整体切换回老模式，而是在**保护模式内部**，给某一个具体任务套上一层"看起来像运行在8086上"的壳，让这个任务以为自己独占了一台老式8086，但操作系统本身、其他任务，依然好好地运行在现代化的保护模式之下。

这就是为什么这道题是真题考点、也最容易被问反：**虚拟8086方式是"一个程序的运行方式"（任务级概念），而实地址方式是"处理器的工作方式"（处理器整机级概念）**，二者所属的层级完全不同，不能并列地理解成"Pentium的三种平等模式随便切"。打个比方，实地址模式像是"整栋楼停电改用应急照明"，影响的是整栋楼；虚拟8086方式像是"在正常供电的楼里，给某个房间单独接了一盏老式煤油灯做怀旧装饰"，只影响这一个房间，楼里其他地方该怎么正常运转还怎么运转。

只有理解了这一点，后面整章讲的GDT、LDT、分段分页才有意义——它们都只在保护模式（以及保护模式内的虚拟8086任务所依托的底层框架）下才会被真正用到。

## 第三部分：虚拟地址长什么样——段选择符与段描述符

> 进入保护模式后，第一个要打破的旧观念是：地址不再是简单的"段基址:偏移量"，而是变成了"一张身份证（段选择符）+ 偏移量"，身份证本身不携带地址信息，要拿着它去查表才能找到真正的地址。

### 3.1 虚拟地址的两段式结构

Pentium保护模式下，存储器管理单元使用48位的存储器指针，对应64T（$2^{46}$）的地址空间。这48位拆成两部分：

```
        47 ······ 32  31 ······················ 0
虚拟地址 [  段选择符  ][        偏移量          ]
              16位              32位
```

注意这里的"段选择符"和8086里的"段寄存器值"看起来都是16位，但含义已经完全不同——8086里段寄存器值乘以16直接就是段基址的一部分，而Pentium保护模式下的16位段选择符，**本身不是地址，只是一把"查表用的钥匙"**。

### 3.2 段选择符内部的三个字段

这把"钥匙"内部还能再拆成三块：

```
15 ··············· 3   2    1  0
[    INDEX索引值    ][ TI ][RPL]
       13位            1位   2位
```

- **INDEX（13位）**：表中第几号位置，乘以8（因为每个描述符占8字节）就是表内偏移量；
- **TI（1位）**：去哪本表里查——TI=0查全局表GDT，TI=1查局部表LDT；
- **RPL（2位）**：声明"我这次访问请求的权限等级"，不参与定位地址，只参与第五部分讲的权限检查。

可以把这三个字段类比成图书馆的借书条：INDEX是"第几号书架第几本"，TI是"去总馆查还是去分馆查"，RPL是"你这次借阅声明的身份等级（比如普通读者还是VIP）"——借阅证号本身定位的是书的位置，身份等级则决定你有没有资格借这本书。

### 3.3 段描述符——"钥匙"打开之后看到的内容

段选择符查表查出来的，是一个8字节的**段描述符**，里面记录了这个段的全部"身份信息"：

```
字节0~1： 段界限低16位
字节2~4： 段基址低24位
字节5：   P存在位 | DPL特权级 | S分类 | TYPE类型
字节6：   G粒度 | D/B位 | 0(兼容位) | AVL | 段界限高4位
字节7：   段基址高8位
```

几个关键字段拆开理解：

- **段基址（32位，分散存放）**：这个段在4GB物理空间中真正的起始位置——这才是真正的"地址信息"，藏在描述符里，而不是藏在段选择符里；
- **段界限（20位）+ G粒度位**：决定段有多长。G=0时界限以字节为单位（最大1MB），G=1时以4KB为单位（最大4GB）——粒度位本质上是"给同一个20位数字换算单位"，让20位数字既能精确表示小段、又能表示到4GB的大段；
- **P存在位**：1表示这个段当前确实在物理内存里，0表示它被换出到了磁盘（辅存），CPU访问时如果发现P=0会触发异常，让操作系统先把它调回内存；
- **DPL特权级（2位）**：这个段允许被多高权限的代码访问，是第五部分权限机制的入口；
- **S分类位**：S=1是"程序段"（数据段/堆栈段/代码段），S=0是"系统段"（TSS、LDT、各种门）——这个区分会在第五部分反复用到。

> 一个直观的理解方式：**段选择符是"门牌号"，段描述符是"门牌号对应的整套房产信息（地址、面积、产权等级）"**。你拿着门牌号去物业（描述符表）查档案，查到的才是真正有用的信息。

## 第四部分：分段寻址的完整链路——GDT与LDT怎么查

> 有了"段选择符"和"段描述符"这两个角色，下一步就该讲清楚：CPU具体是怎么"拿着选择符，找到正确的那张描述符表，再从表里查出描述符"的。这是全章承上启下的核心环节。

### 4.1 两本"花名册"：GDT与LDT

保护模式下，段描述符并不是随便堆放在内存某处，而是按规则集中存放在两类表里：

- **GDT（全局描述符表）**：全系统**只有一份**，存放系统级共享的段描述符（比如操作系统内核用的段）；
- **LDT（局部描述符表）**：**每个任务各自一份**，存放该任务私有的段描述符。

这正好呼应了第一部分讲的"任务隔离"目标——每个任务有自己专属的LDT，意味着即使两个任务的段选择符INDEX字段数值相同，它们实际指向的也是完全不同的描述符（一个在公共GDT里查，一个在各自私有的LDT里查），天然就互不干扰。

### 4.2 谁来记住这两本表在哪——GDTR与LDTR

光有表还不够，CPU还得知道"这两本表具体存放在内存的哪个地址、有多大"，这就是**GDTR**和**LDTR**两个寄存器的作用：

| 寄存器 | 保存内容 | 特点 |
|---|---|---|
| GDTR | 32位基址 + 16位界限（共48位） | 全系统唯一，直接保存表的位置和大小 |
| LDTR | 16位选择符（可见部分）+ 32位基址/20位界限/12位属性（不可见Cache部分） | 每任务各自不同，但LDTR本身存的不是地址，而是一个"指向GDT中LDT描述符"的选择符 |

LDTR的设计有点绕，但其实很合理：**LDT本身也是一种"段"（系统段），它自己的位置信息（基址、界限）也要被描述、也要被纳入统一管理**。所以LDT描述符本身就登记在GDT里，LDTR要先拿着自己保存的选择符去GDT里查出"LDT描述符"，再把这个描述符里的基址等信息缓存进LDTR的"不可见部分"，这才算真正"激活"了当前任务的LDT。

可以这样理解GDT和LDT的关系：**GDT是"全市统一档案馆"，每个任务的LDT是"各自单位的内部档案室"，而内部档案室的门牌号信息，又登记在全市统一档案馆里**——所以要查一份"内部档案"，得先去市档案馆查到"内部档案室在哪儿"，再进去查具体档案，前后要查两次表。

### 4.3 完整寻址过程——三步法

不管是查GDT还是查LDT，核心步骤都是同一套"三步法"：

```
①段选择符×8 + 表基址（GDTR或LDTR提供）= 段描述符所在的内存地址
②从该地址读出8字节段描述符 → 把基址、界限、属性装入对应段寄存器的Cache部分
③该Cache中的基址 + 偏移量 = 32位线性地址
```

**关键细节**：①②两步只在"第一次"装入某个段寄存器（比如DS）时才需要真正访问内存查表；一旦表内容被读出并缓存进段寄存器的Cache部分，之后再用同一个段寄存器访问数据时，CPU直接用Cache里已经缓存的基址做第③步加法即可，不用每次都重新查表。这正是"Cache"这个词出现在段寄存器结构里的原因——它和我们熟悉的CPU高速缓存是同一个设计思想：**避免重复的慢速查表，把查过一次的结果缓存起来**。

如果走的是LDT（TI=1），流程会在①之前多出一道前置步骤："先用LDTR里的选择符去GDT中查出LDT本身的位置，再用这个位置作为表基址去做后面的三步法"——本质上还是同一套逻辑，只是多嵌套了一层。

## 第五部分：从线性地址到物理地址——分页机制

> 第四部分讲完，我们手里已经拿到了"32位线性地址"。如果到此为止，这个线性地址直接当物理地址用，故事就结束了（这正是"分段不分页"的工作方式）。但Pentium通常还会再走一层分页，把线性地址进一步翻译成物理地址，这一步要解决的是"段长度可变、调入调出不灵活"的问题。

### 5.1 为什么段不够，还要加一层页？

段的好处是"贴合程序的逻辑结构"——一个段就是一个有意义的模块（代码段、数据段……），便于按权限管理、按模块共享。但段的坏处是**长度可变**：一个段可能几KB，也可能几MB，当内存需要腾挪空间、把某个段换出换入时，长度不固定会导致严重的内存碎片问题，调度起来很不灵活。

**页**正好弥补这个缺陷：把整个线性地址空间切成大小完全固定的小块（每页4KB），不管段有多大，最终落到物理内存时都是按这个统一的4KB为单位来分配和置换的。这就好比，"段"是按照功能划分的"房间"（卧室、厨房、书房，大小各不相同），"页"则是不管房间多大、统一铺设的"标准地砖"——房间的划分负责"逻辑意义"，地砖的划分负责"物理管理的整齐划一"。

### 5.2 二级页表——为什么不能只用一级？

如果只用一级页表，要覆盖4GB地址空间、每页4KB，需要的页表项数量是 $4GB ÷ 4KB = 2^{20}$ 个，一张表会大到不现实。Pentium的解法是**二级页表**：

```
第一级：页目录（Page Directory）—— 最多1024项，每项指向一个第二级页表的基址
第二级：页表（Page Table）—— 最多1024项，每项指向一个真正的4KB物理页基址
```

$1024 \times 1024 = 2^{20}$，刚好覆盖完整4GB地址空间，但每一张表自身大小都被控制在 $1024 \times 4字节 = 4KB$ 以内（恰好一页装下一整张表），这正是二级结构的精妙之处——**用"目录指向表"的间接层，把一张巨表拆成了两张可管理的小表，而且每张小表自身又恰好填满一个标准页，结构上完全自洽**。

### 5.3 32位线性地址怎么拆

```
[10位页目录索引][10位页表索引][12位页内偏移]
 (A31~A22)       (A21~A12)      (A11~A0)
```

- 最高10位拿去页目录里查，找到对应的页表基址；
- 中间10位拿去刚找到的那张页表里查，找到对应的物理页基址；
- 最低12位（$2^{12}=4096=4KB$，刚好对应一页的大小）不参与查表，直接作为"页内偏移"加到物理页基址上。

```
线性地址
   |
   ├─ 页目录索引(10位) → CR3指定的页目录 → 查出页表基址
   |                                          |
   |                  页表索引(10位) →─────────┘ → 查出物理页基址
   |                                                  |
   └─ 页内偏移(12位) ───────────────────────────────→ + 物理页基址 = 物理地址
```

页目录本身放在哪？由**CR3寄存器**指定其基址（这个基址必须落在4KB边界上）。

## 第六部分：整章串联——段页式综合地址转换全过程

> 把第四部分（分段）和第五部分（分页）首尾接起来，就是本章最核心、也是真题反复考查的那张完整流程图。理解了这张图，前面所有零散的概念就都有了归宿。

```
虚拟地址（逻辑地址）
[16位段选择符: INDEX+TI+RPL]   [32位偏移量]
        |
        | INDEX×8，TI决定查GDT还是LDT
        v
  段描述符表（GDT或LDT）
        |
        | 取出该描述符的32位段基址
        v
  段基址 + 32位偏移量 = 32位线性地址  ←——— 分段阶段到此结束
        |
        | 拆成 [10位页目录索引][10位页表索引][12位页内偏移]
        v
  CR3指定的页目录 → 查出页表基址
        |
  该页表 → 查出物理页基址
        |
  物理页基址 + 12位页内偏移 = 32位物理地址  ←——— 分页阶段到此结束
```

如果系统配置成"只分段不分页"，那么走完上面第一段（求出32位线性地址）就直接结束，这个线性地址本身就当物理地址使用——这种配置的好处是不用查页目录/页表、转换速度快，代价是段的调入调出不够灵活（呼应第五部分开头讲的"为什么需要分页"）。

## 第七部分：保护机制——这一切是怎么被"保护"起来的

> 前面六部分讲的都是"怎么把地址翻译出来"，这一部分回到最初的动机——"怎么防止任务之间互相干扰、防止低权限代码碰高权限数据"。你会发现，保护机制并没有引入任何新的硬件部件，而是巧妙地复用了前面已经讲过的段描述符、GDT/LDT结构。

### 7.1 三个特权级角色：CPL、DPL、RPL

特权级一共4档（0~3，0最高3最低），三个相关概念分别站在不同的"立场"上：

- **DPL（描述符特权级）**：写死在段描述符里，相当于"这扇门自己设定的门禁等级"；
- **CPL（当前特权级）**：当前正在执行的代码所处的特权级，相当于"你此刻所在的楼层权限"；
- **RPL（请求特权级）**：藏在段选择符的低2位，相当于"你刷卡进门时，主动声明的身份等级"。

为什么需要RPL这个看起来多余的角色？因为有时候高权限代码会**代表**低权限调用方去访问数据，这时候系统希望"哪怕你本身权限很高，这次访问也按照调用方的权限走"，RPL就是用来传递这种"降级声明"的机制，防止高权限代码被滥用来绕过权限检查。

核心规则：**特权级为P的段中的数据，只能被特权级数值≤P（权限相等或更高）的代码访问**——数值越小权限越大，这是最容易考反的一点，做题前先在草稿纸上写清楚"0级最高"提醒自己。

### 7.2 TSS——任务的"完整存档"

多任务系统要支持"任务A运行一半，切换到任务B，过会儿再切回任务A接着跑"，靠的就是**TSS（任务状态段）**。TSS本质上是一份"存档文件"，把当前任务所有寄存器（通用寄存器、段寄存器、EIP、EFLAGS）、各特权级的堆栈指针、CR3（页目录基址）、LDT选择符全部记下来。切换任务时，处理器把当前所有寄存器内容写进当前任务的TSS（存档），再把目标任务TSS里的内容读出来装回寄存器（读档），任务就像被"无缝续上"一样继续运行。

**TSS本身不含代码和数据**，它只是纯粹的"状态快照"——这是它和数据段、代码段最根本的区别。每个任务都有自己专属的一个TSS，由TR寄存器指向当前任务的TSS（机制上和LDTR指向LDT几乎一样：TSS描述符也登记在GDT里，TR本身也是先存一个选择符，再去GDT查出TSS的位置）。

### 7.3 门描述符——任务切换/特权切换的"安检口"

如果说TSS是"存档"，那"门描述符"就是控制"什么时候、谁有资格触发存档读写"的安检关卡。门描述符共四类：

- **调用门**：允许低特权级代码安全地调用高特权级的子程序（穿过权限边界的合法通道）；
- **任务门**：用于任务切换（指向某个TSS，触发存档/读档动作）；
- **中断门、陷阱门**：用于确定中断/异常发生时该跳转到哪个服务程序入口。

门描述符的结构和普通段描述符不同：它不存"段基址"，而是存"目标段选择符 + 段内偏移量"——因为门指向的不是一整片连续物理空间，而是"另一个段里的一个具体入口点"，本质上是一层间接跳转，跟函数指针的概念有点像：它不直接是代码，而是"告诉你去哪里找代码"。

## 知识连接图——整章演化逻辑

```
[动机]虚拟存储器解决容量问题 + 保护模式解决任务隔离问题
        |
        v
[模式]三种操作模式中，只有"保护模式"（及其内部的虚拟8086任务）
       才会启用下面这套完整机制
        |
        v
[数据结构]虚拟地址=段选择符+偏移量 ──→ 段选择符查出 ──→ 8字节段描述符
        |                                                  |
        v                                                  |
[分段寻址]段选择符×8+GDTR/LDTR基址 → 查表 ──────────────────┘
        |                            (LDT需先经GDT查出LDT本身位置)
        v
   32位线性地址
        |
        v
[分页寻址]线性地址拆成页目录索引+页表索引+页内偏移
        |  CR3→页目录→页表→物理页基址+页内偏移
        v
   32位物理地址（段页式寻址全过程闭环）
        |
        v
[保护机制]复用段描述符里的DPL字段 → CPL/RPL/DPL三方权限检查
          复用GDT中登记的TSS/门描述符 → 支持任务切换与特权切换
```

整张图最值得记住的一点是：**前四层（动机→模式→数据结构→寻址）是一条单向的"翻译流水线"，第五层（保护机制）不是流水线的延伸，而是寄生在第三层数据结构之上的一套"附加检查"**——理解了这一点，你就不会把CPL/DPL/RPL当成和GDT/LDT平级的全新知识点去死记，而是能看出它们其实共享同一套底层结构。

## 本章要带走的五件事

1. **虚拟存储器解决"容量"矛盾，保护模式解决"隔离"矛盾**——两者动机不同，不要混为一谈，更不要和"Cache解决速度矛盾"搞混。
2. **段选择符不是地址，是查表用的钥匙**；真正的地址信息（段基址）藏在8字节的段描述符里，要先查表才能拿到。
3. **GDT全局唯一，LDT每任务一份**，而且LDT自己的位置信息也登记在GDT里——查LDT中的段，本质上要先后查两次表。
4. **分段把线性地址翻出来，分页再把线性地址翻成物理地址**，二者是接力关系而非并列关系；不分页时线性地址直接当物理地址用。
5. **特权级机制（CPL/DPL/RPL）和TSS/门描述符不是全新的硬件部件，而是复用了已经建好的段描述符和GDT体系**——保护机制是"长在"前面数据结构之上的，不是另起炉灶的新知识点。
