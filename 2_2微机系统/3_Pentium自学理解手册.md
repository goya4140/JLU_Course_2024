# 第3章 32位 Pentium 微处理器
## 从零建立理解——自学手册

> **这份资料的目标**：不是让你背住知识点，而是让你理解"Pentium比8086多出来的那些东西，为什么要有、它们之间怎么连起来"。读完之后，你应该能在脑子里建立一张从8086到Pentium的演化地图，理解每一个新概念的来龙去脉。

## 知识体系总览

在进入正文前，先看清整章的分层逻辑——每一层在回答一个不同层次的问题，层与层之间是递进关系，而不是并列罗列。

```
§3 32位 Pentium 微处理器

├─ 第一层：升级动机（回答"为什么要在8086基础上做这些改变"）
│   ├─ 局限1：速度——单流水线，一次只能执行一条指令
│   ├─ 局限2：空间——1MB寻址空间不够用
│   ├─ 局限3：保护——任何程序可访问任意内存，无法隔离多任务
│   └─ 局限4：浮点——无内置浮点单元，依赖外置8087协处理器
│
├─ 第二层：内部结构（回答"CPU内部怎么改，才能跑得更快"）
│   ├─ 超标量：U/V双流水线并行执行（→ 直接回应局限1）
│   │   └─ 分工：U管全部整数+浮点，V管简单整数+交换浮点
│   ├─ 分立Cache：指令Cache(8KB) + 数据Cache(8KB)分开
│   │   └─ 目的：取指和读数据可同时进行，不互相排队
│   ├─ BTB动态转移预测：用历史记录猜分支走向
│   │   └─ 目的：减少流水线因跳转指令被冲刷的次数
│   └─ 11个子部件协同：总线接口①→Cache⑥⑦→预取②→译码⑧→运算③⑤
│
├─ 第三层：寄存器演化（回答"程序员和操作系统看到的CPU内部是什么样"）
│   ├─ 基本结构寄存器：8086寄存器的32位扩展（EAX⊃AX⊃AH/AL）
│   │   ├─ 通用寄存器×8 + 专用寄存器EIP/EFLAGS + 段寄存器×6（新增FS/GS）
│   │   └─ 向下兼容设计：写AX不影响EAX高16位
│   ├─ 系统级寄存器：操作系统专用，管理内存保护机制
│   │   ├─ 系统地址寄存器×4：GDTR/IDTR/LDTR/TR（指向各类描述符表）
│   │   └─ 控制寄存器CR0~CR4：PE位是实地址↔保护方式的总开关
│   └─ 这一层是连接"内部结构"和"操作模式"的桥梁——
│       CR0.PE位直接决定第五层要讲的工作模式切换
│
├─ 第四层：外部接口（回答"CPU怎么和外部世界交换数据、控制Cache"）
│   ├─ 地址线：A31~A3 + #BE7~#BE0（字节允许，弥补A2~A0未引出）
│   ├─ 数据线：D63~D0（64位，是8086的8倍带宽）
│   ├─ 总线周期控制：D/#C + M/#IO + W/#R 三信号组合定义周期类型
│   ├─ Cache控制：#Cache（处于Cache周期）+ #KEN（外部允许缓存）
│   └─ 地址流水线：#NA（支持地址提前发出，提升总线利用率）
│
└─ 第五层：操作模式与内存保护（回答"局限2、3怎么最终被解决"）
    ├─ 实地址模式：兼容8086，1MB，CR0.PE=0
    ├─ 保护模式：4GB物理+64TB逻辑，CR0.PE=1
    │   └─ 核心机制：段寄存器从"直接存基址"变成"存段选择符"
    │       ├─ 段选择符 = 13位索引 + TI位（查GDT/LDT）+ RPL
    │       ├─ 段描述符（8字节）：基址+段限+属性(含DPL特权级)
    │       └─ GDT(全局唯一)/LDT(每任务一个)/IDT(全局唯一，中断用)
    └─ 虚拟8086模式：保护模式下运行旧8086程序，EFLAGS.VM=1
        └─ 易混辨析：虚拟8086是"程序"的运行方式，实地址是"处理器"的工作方式
```

**这五层是怎么串起来的**：第一层提出的四个局限是整章的总纲——后面每一层都是在回应其中某一个或几个局限，而不是凭空冒出的新功能。第二层（超标量/Cache/BTB）直接解决"速度"局限，发生在CPU内部，程序员通常感知不到。第三层（寄存器）是承上启下的关键——它既是第二层内部结构在编程层面的体现（通用寄存器扩展），又通过CR0.PE位和GDTR/LDTR/IDTR/TR这几个系统寄存器，为第五层的保护模式埋下伏笔。第四层（引脚）是平行于前面几层的一条独立线索，它解决的是"CPU如何与外部Cache、内存系统打交道"，不直接依赖第二、三层的内容，可以单独理解。第五层（操作模式）是全章的落脚点，它把第三层引入的系统寄存器（尤其是描述符表的概念）真正用起来，最终解决第一层提出的"空间不够"和"没有保护"这两个局限——读到这里你应该能感觉到，保护模式不是凭空多出来的新知识，而是前面几层零件（描述符、段寄存器新含义、CR0.PE位）拼装出来的产物。

## 第一部分：先问一个问题——8086有什么不够用的地方？

学第3章之前，你已经学过8086了。8086是一颗设计非常清晰的芯片：BIU负责搬运，EU负责计算，分段寻址解决16位寄存器访问1MB空间的矛盾。

但8086有几个根本性的局限：

**局限一：太慢了。** 8086的流水线很简单——BIU预取指令，EU执行指令，两者虽然并行，但本质上还是一条流水线，一次只能执行一条指令。

**局限二：寻址空间太小。** 1MB，这在1978年够用，但软件越来越大，很快就不够了。80286把物理空间扩展到16MB，80386进一步扩展到4GB。到了Pentium，物理地址线32位，寻址空间4GB。

**局限三：没有内存保护。** 8086里任何程序都可以访问任意内存地址，一个程序崩溃可能把整个系统的内存搞乱。多任务操作系统需要把不同程序的内存隔离开，8086做不到。

**局限四：浮点计算很弱。** 8086本身没有浮点运算单元，需要配合专门的数学协处理器8087才能做浮点计算，而且很慢。

Pentium的每一个新设计，都是在解决上面这四个问题之一。带着这个视角，你会发现Pentium的结构不是"堆砌的新功能"，而是"有来由的升级"。

## 第二部分：Pentium内部结构——从一条流水线到两条

### 2.1 超标量：同时执行两条指令

8086的EU一次只能执行一条指令。Pentium的解法叫做**超标量（Superscalar）**：在CPU内部放两条完整的执行流水线，叫做 **U流水线** 和 **V流水线**，让它们同时各执行一条指令。

```
          ┌────────────────────────────────┐
指令流  →  │  指令译码器（检查配对可行性）    │
          └──────────────┬─────────────────┘
                         │
          ┌──────────────┴─────────────────┐
          ↓                               ↓
   ┌─────────────┐                ┌─────────────┐
   │  U 流水线   │                │  V 流水线   │
   │（执行全部   │                │（执行简单   │
   │  整数指令  │                │  整数指令） │
   │  和浮点）  │                │             │
   └─────────────┘                └─────────────┘
```

超标量为2，意味着理想情况下吞吐量是8086的两倍。但有一个限制：**两条指令必须满足配对规则才能并行**（比如，两条指令不能都写同一个寄存器）。指令译码器负责检查每次取来的两条指令能不能配对。

**两条流水线的分工：**
- **U流水线**：执行**全部整数指令**和浮点指令，功能最全
- **V流水线**：只执行**简单整数指令**（以及交换类浮点指令），逻辑更简单

这个分工是有道理的：浮点运算电路复杂，如果两条流水线都做浮点，芯片面积会翻倍；让V流水线专门跑简单指令，是性能和面积的折中。

每条流水线有**5级**：取指（IF）→ 译码D1 → 译码D2 → 执行（EX）→ 写回（WB）。

### 2.2 分立Cache：指令和数据分开存

8086没有片内Cache，所有指令和数据都从外部内存取，速度受限于总线。

Pentium设计了**两个独立的片内Cache**：
- **代码Cache（指令Cache）**：8KB，只存指令
- **数据Cache**：8KB，只存数据

为什么要分开？因为CPU在同一个时钟周期内可能**同时需要**取指令（从指令Cache）和读取操作数（从数据Cache）。如果两者共用一个Cache，这两个操作就会竞争，必须排队。分开之后，两个操作可以并行——这和U/V双流水线的并行思路是一脉相承的。

每个Cache还配有一个 **TLB（转换旁视缓冲器，Translation Lookaside Buffer）**，这是地址转换的加速缓存，后面讲保护模式时会用到。

### 2.3 动态转移预测：用历史猜未来

流水线有一个天敌：**分支指令（JMP/JZ/LOOP等）**。

当流水线取了一条条件跳转指令，它不知道条件成不成立，也就不知道下一条指令从哪里取。如果等到条件判断出来再取指，前面的流水线就空转了（这叫"流水线冲刷"）。

Pentium的解法叫 **BTB（Branch Target Buffer，转移目标缓冲器）**：

```
BTB 记录历史：
"上次执行到这条跳转指令时，跳了还是没跳？跳去哪里了？"

下次再碰到同一条跳转指令时：
→ 查 BTB 历史记录 → 预测结果 → 按预测结果提前取指 → 继续执行

如果预测正确：✅ 更新 BTB，流水线没有中断
如果预测错误：❌ 清空流水线，重新取正确路径的指令（代价较大）
```

BTB 是动态的，靠实际运行历史来预测，而不是靠静态规则。对于循环（同一个跳转执行很多次）预测成功率很高。

### 2.4 把内部结构串起来

现在把Pentium内部11个子部件的关系理清楚：

```
外部内存
    │  ←→  总线接口部件①（管三总线：32位地址线 + 64位数据线 + 控制线）
    │
    ├──→  代码Cache⑥（8KB）
    │         │
    │         ↓
    │     预取缓冲器②（两个各32B，A缓冲器/B缓冲器交替）
    │         │
    │     ←  BTB④（查转移历史，影响预取地址）
    │         │
    │         ↓
    │     指令译码部件⑧（检查U/V配对，生成微指令）
    │         │
    │     控制ROM⑪（存微指令）
    │         │
    │     控制部件⑨（调度整个流水线）
    │         │
    │    ┌────┴────┐
    │    ↓         ↓
    │  整数运算部件③     浮点运算部件⑤
    │  （U/V各一个ALU）   （8级流水）
    │    │
    │  整数及浮点寄存器组⑩
    │
    └──→  数据Cache⑦（8KB）  ←→ 运算部件③⑤（读写操作数）
```

这张图的关键点：**①是CPU和外部世界的唯一通道**；⑥和⑦把内部的指令路径和数据路径分离；③是最核心的计算引擎，有U/V两份。

## 第三部分：寄存器——从16位到32位的演化

### 3.1 通用寄存器：加了个"E"前缀

8086有8个16位通用寄存器（AX、BX、CX、DX、SP、BP、SI、DI）。Pentium把它们扩展到32位，名字前面加了个"E"（Extended，扩展）：EAX、EBX、ECX、EDX、ESP、EBP、ESI、EDI。

**包含关系（以EAX为例）：**

```
EAX（32位）
╔════════════════════════════════╗
║         EAX（32位）            ║
║  ┌────────────┬───────────────┐║
║  │  高16位    │  AX（低16位） ║║
║  │（无名字）  │  ┌─────┬─────┐║║
║  │            │  │ AH  │ AL  │║║
║  │            │  │(8位)│(8位)│║║
║  │            │  └─────┴─────┘║║
║  └────────────┴───────────────┘║
╚════════════════════════════════╝
```

向下兼容的设计：写AX只修改EAX的低16位，高16位不变；写AL只修改最低8位。这保证了8086的程序在Pentium上无需修改就能运行。

**SP→ESP（堆栈指针）和BP→EBP（基址指针）同理扩展，但它们的低16位没有进一步分成高低8位（没有SPH/SPL这种写法）。**

### 3.2 专用寄存器：EIP 和 EFLAGS

- **EIP（指令指针）**：32位版的IP，指向下一条待取指令的段内偏移地址。CPU复位后，EIP=0000FFF0H（配合CS=F000H，首条指令在物理地址FFFF0H处）。每取一条指令，EIP自动增加该指令的字节数。

- **EFLAGS（标志寄存器）**：32位版的FLAGS，低16位与8086的FLAGS完全相同（CF/PF/AF/ZF/SF/OF/IF/DF/TF），高16位新增了几个保护模式标志：

  | 新增标志 | 含义 |
  |---|---|
  | VM（bit17） | VM=1 → CPU当前运行在虚拟8086模式下 |
  | RF（bit16） | 恢复标志，调试相关 |
  | NT（bit14） | 嵌套任务标志，保护模式任务切换相关 |
  | IOPL（bit13~12） | I/O特权级，控制程序是否能执行I/O指令 |
  | AC（bit18） | 对齐检查，AC=1时检测非对齐内存访问 |

### 3.3 段寄存器：从4个到6个

8086有4个段寄存器（CS/SS/DS/ES）。Pentium新增了**FS和GS**两个附加段寄存器，总共6个。

但更重要的变化是：在保护模式下，段寄存器的含义彻底变了。

**实地址模式下（与8086相同）**：段寄存器 × 16 = 段基址，程序员直接看到的是段基址。

**保护模式下**：段寄存器里存的不再是段基址，而是一个**"段选择符"**——一个索引号，用来在一张叫"描述符表"的表里查找这个段的真实信息（基址、长度、访问权限等）。

这个变化是实现内存保护的关键，第三部分的最后会详细讲。

### 3.4 系统级寄存器：操作系统专用

Pentium新增了一类程序员通常看不到的寄存器，专门给操作系统用，用来管理内存保护机制：

**四个系统地址寄存器：**

```
GDTR（48位）= [32位基地址 | 16位段限]
              ↑ 指向 GDT（全局描述符表）在内存中的位置

IDTR（48位）= [32位基地址 | 16位段限]
              ↑ 指向 IDT（中断描述符表）在内存中的位置

LDTR（16位选择符 + 64位描述符缓存）
              ↑ 指向当前任务的 LDT（局部描述符表）

TR  （16位选择符 + 64位描述符缓存）
              ↑ 指向当前任务的 TSS（任务状态段）
```

先记住这四个名字和它们指向什么，具体的寻址过程放到后面"保护模式"部分一起讲。

**五个控制寄存器（CR0~CR4）：**

最重要的是**CR0**，它的几个关键位控制着整个处理器的工作状态：

| CR0位 | 名称 | 含义 |
|---|---|---|
| PE（bit0） | 保护模式允许 | **PE=0 实地址方式，PE=1 保护方式** |
| PG（bit31）| 分页允许 | PG=1 开启分页机制 |
| CD（bit30）| 禁止Cache | CD=1 关闭片内Cache填充 |
| NW（bit29）| 不透明写 | NW=1 写操作只写Cache不写主存 |

记住PE位：**它就是实地址↔保护方式的开关**。操作系统启动时通过设置CR0.PE=1来切换到保护方式。

## 第四部分：外部引脚——CPU与外部世界的接口

引脚是CPU的"对外接口"，每根引脚的存在都有具体的功能需求。理解引脚的最好方法是问：**这根线传递的是什么信息？谁发给谁？**

### 4.1 地址线：A31~A3，加上 #BE7~#BE0

Pentium有**32位地址线（A31~A0）**，但实际引出的地址引脚只有**A31~A3**，没有A2、A1、A0。

**为什么地址线的最低三位不直接引出来？**

因为Pentium的数据总线是64位（8个字节），每次总线操作天然按8字节对齐。A2、A1、A0的信息被编码成了**8根字节允许信号#BE7~#BE0**（低有效），每根对应数据总线上的一个字节通道：

```
数据总线：  D63..D56  D55..D48  ...  D15..D8  D7..D0
对应信号：   #BE7      #BE6     ...   #BE1     #BE0
```

某根#BEx为0（有效）时，对应字节通道上的数据参与本次传输；为1时该字节不参与。这样既支持64位传输，也支持任意字节宽度的访问。

### 4.2 数据线：D63~D0

64位数据总线，这是Pentium比8086快的重要原因之一——一次能搬的数据是8086的8倍。

### 4.3 总线周期控制：三个信号组合定义当前传输类型

| 信号 | 全称 | 含义 |
|---|---|---|
| M/#IO | Memory / IO | 高电平=访问内存，低电平=访问I/O端口 |
| W/#R | Write / Read | 高电平=写操作，低电平=读操作 |
| D/#C | Data / Code | 高电平=传数据，低电平=取代码（指令） |

三根信号的组合完整描述了当前总线周期的性质：

```
D/#C  M/#IO  W/#R  → 总线周期类型
  0     1      0   → 指令预取（从内存取指令）
  1     1      0   → 数据读（从内存读数据）
  1     1      1   → 数据写（往内存写数据）
  1     0      0   → I/O读
  1     0      1   → I/O写
```

### 4.4 Cache控制：#Cache 和 #KEN

这两根引脚控制什么时候把外部内存的数据"缓存"到片内Cache：

- **#Cache（低有效）**：由CPU发出，表示"当前总线周期是一个可缓存的访问（Cache周期）"。
- **#KEN（低有效）**：由外部存储器系统发给CPU，表示"当前访问的这段内存地址是可以缓存的"。

**两者的配合逻辑：**

```
CPU读内存（#Cache有效）
        ↓
外部存储器系统判断：这个地址区域允许缓存吗？
        ├── 允许 → #KEN有效（低电平）→ CPU进入"突发读"模式
        │                              （连续读一整个Cache行，填入片内Cache）
        └── 不允许 → #KEN无效 → 普通读取，不填Cache
```

类比理解：你去图书馆借了一本书（#Cache），图书馆判断这本书是否允许复印（#KEN），允许的话你就复印一份带回家（填Cache），以后看自己的副本就行，不用再去图书馆。

### 4.5 地址流水线：#NA

**#NA（下一地址，Next Address，低有效）**：这根引脚支持**地址流水线（Address Pipelining）**——一种提高总线带宽的技术。

正常情况下，一个总线周期分"地址阶段"（发地址）和"数据阶段"（传数据），必须等当前周期的数据传输完成后，才发出下一个地址。

有了地址流水线：外部系统在**当前周期的数据还在传输时**，就可以驱动#NA有效，告诉CPU"你可以提前发下一个地址了"。CPU收到#NA有效信号后两个时钟周期，发出下一个总线周期的地址。这样地址和数据就重叠了，总线利用率提升。

## 第五部分：三种操作模式——从1MB到4GB的演进

### 5.1 实地址模式（Real-Address Mode）

这是Pentium上电复位后的**默认状态**。在这个模式下，Pentium完全模拟8086的行为：

- 地址计算方式：**物理地址 = 段寄存器 × 16 + 偏移地址**
- 访问空间：**1MB**（A31~A20全被屏蔽为0，由A20M#引脚控制）
- 没有内存保护，没有分页
- 但可以使用32位寄存器和Pentium新增的指令（这是和真正的8086的区别）

两个特殊保留区域：
```
0000:0000H ~ 0000:03FFH  →  中断向量区（256个中断，每个4字节 = 1024字节）
FFFF:0000H ~ FFFF:000FH  →  系统初始化区（Pentium复位后从这里开始执行）
```

**从实地址模式切换到保护模式**：操作系统将CR0的PE位（bit0）置1。

### 5.2 保护虚拟地址模式（Protected Mode）

这是现代操作系统（Windows、Linux）真正运行的模式。

**空间大小**：
- 物理空间：**4GB**（32位地址线）
- 逻辑空间（每个任务看到的虚拟地址空间）：**64TB**（16384个段，每段最大4GB）

**核心变化——段寄存器的含义变了：**

在实地址模式下，段寄存器直接存段基址。在保护模式下，段寄存器存的是**段选择符**——一个"查表索引"，用来在描述符表里查找段的真实信息。

这个机制的设计目的：**让操作系统能控制每个程序可以访问哪些内存**。操作系统管理描述符表，每个程序只能通过段选择符访问操作系统允许它访问的段，越界或非法访问会引发保护异常。

保护模式的完整寻址过程将在第5章详细展开（那章的核心就是这个）。这里先建立一个印象：**保护模式 = 实地址模式 + 描述符表机制 + （可选的）分页机制**。

### 5.3 虚拟8086模式（Virtual 8086 Mode）

这是保护模式的一个子集，用来解决一个实际问题：**如何在现代操作系统（保护模式）里运行老的8086程序？**

在虚拟8086模式下，程序感觉自己在实地址模式里运行（看到的是1MB地址空间，用的是老式的段×16+偏移的地址计算），但实际上操作系统通过分页机制把这1MB的地址空间映射到4GB物理内存的任何位置，同时用保护机制把不同的虚拟8086任务隔离开。

**最重要的概念辨析（✅真题考点）：**

```
实地址方式  ≠  虚拟8086方式

实地址方式：  整个"处理器"的工作模式（所有程序都在实地址方式下运行）
虚拟8086方式：某个"程序"的运行方式（只有这个任务在模拟8086，处理器仍在保护方式下）
```

进入虚拟8086方式：在保护方式下，将EFLAGS的**VM位（bit17）置1**。

## 第六部分：描述符表——内存保护的核心数据结构

> 这部分是理解保护模式的基础，第5章会展开，这里建立基本概念。

### 6.1 段描述符：一个段的完整"身份证"

在保护模式下，每个内存段不再只有"从哪里开始"一个属性，而有一套完整的信息，用**8字节**的数据结构记录，这就是**段描述符**：

```
段描述符（8字节 = 64位）
┌─────────────────┬─────────────────┬──────────────┐
│   段基地址       │    段长度        │   属性        │
│   （32位）       │   （20位）       │  （12位）     │
│                 │                 │              │
│  段从这里开始    │ 段有多长（可配置 │ 权限/类型/   │
│                 │  粒度：字节或4KB）│ 特权级等     │
└─────────────────┴─────────────────┴──────────────┘
```

属性字段里最重要的是**DPL（描述符特权级）**，0~3四个级别（0最高，即内核级；3最低，即用户程序级），控制谁可以访问这个段。

### 6.2 描述符表：段描述符的数组

所有段描述符像数组一样排在一起，就形成**描述符表**。有三种描述符表：

```
GDT（全局描述符表，Global Descriptor Table）
  ├── 整个系统只有一个
  ├── 操作系统使用的段 + 各任务共用的段
  └── 首地址和大小存在 GDTR 寄存器（48位：32位基地址 + 16位段限）

LDT（局部描述符表，Local Descriptor Table）
  ├── 每个任务各有一个
  ├── 该任务私有的段（只有这个任务能用）
  └── 当前任务的 LDT 信息存在 LDTR 寄存器

IDT（中断描述符表，Interrupt Descriptor Table）
  ├── 整个系统只有一个
  ├── 把每个中断号映射到中断服务程序的入口地址（保护模式的中断向量表）
  └── 首地址和大小存在 IDTR 寄存器（48位）
```

**GDT和LDT个数的规律（✅真题选择题考过）：**

> 一个系统有N个任务时：GDT = **1个**（全局唯一），LDT = **N个**（每个任务一个）

### 6.3 段选择符：用"索引"查描述符

现在把整个链条连起来：

```
程序里用到的"逻辑地址"：
┌──────────────────┐   ┌──────────────────────────────────────────┐
│ 段寄存器（16位）  │   │              偏移量（32位）               │
│                  │   │                                          │
│ 这就是"段选择符"  │   │ 在这个段内，距段基址多远                  │
└──────────────────┘   └──────────────────────────────────────────┘
         │
         │ 段选择符的结构：
         │  [15..3] = 13位索引（表示第几个描述符）
         │  [2]     = TI位（0=查GDT，1=查LDT）
         │  [1..0]  = RPL（请求特权级）
         │
         ↓
    按索引号×8，在GDT或LDT里找到对应的段描述符
         │
         ↓
    从描述符里取出32位段基地址
         │
         ↓
    段基地址 + 32位偏移量 = 32位线性地址
         │
    （如果开了分页，线性地址再经过分页机制→物理地址）
    （如果没开分页，线性地址直接就是物理地址）
```

## 第七部分：把所有概念连成一张地图

学到这里，可以画一张Pentium的整体知识地图，把所有概念的关系展示出来：

```
                        ┌─────────────────────────────────────────┐
                        │           Pentium 知识地图               │
                        └─────────────────────────────────────────┘

  【解决"太慢"的问题】              【解决"空间不够"的问题】
  超标量（U/V双流水线）  ←────→    32位地址线（4GB物理空间）
  分立Cache（指令/数据）             │
  BTB（动态转移预测）                 ↓
  8级浮点流水线                   需要一套机制来管理这4GB空间
                                      │
  【内部结构落地】                     ↓
  11个子部件                   【解决"没有保护"的问题】
  ↓                            描述符表（GDT/LDT/IDT）
  寄存器组（5类）    ────→     段描述符（8字节，记录段信息）
  ├── 基本结构（通用/EIP/EFLAGS/段）   │
  ├── 系统级（GDTR/LDTR/IDTR/TR）      ↓
  └── 控制（CR0 PE位→模式切换）   三种操作模式：
                                ├── 实地址（兼容8086，1MB）
  【外部接口】                   ├── 保护（全功能，4GB+64TB虚拟）
  引脚（A31~A3/#BE7~#BE0        └── 虚拟8086（保护模式内运行旧程序）
       /D/#C/M/#IO/W/#R
       /#Cache/#KEN/#NA）
```

这张图的核心逻辑是：**Pentium的每一个新增设计，都是在解决8086的某个具体局限**。超标量解决速度，Cache解决访存瓶颈，描述符表解决内存保护，保护模式和虚拟8086模式解决寻址空间和多任务隔离。
