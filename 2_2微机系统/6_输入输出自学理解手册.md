# 第6章 输入输出
## 从零建立理解——自学手册

> **这份资料的目标**：不是让你背住8259A、8237A各有几个寄存器、几种工作方式，而是让你理解"CPU和外设打交道这件事，为什么会从最原始的'瞎传'一路演化出查询、中断、DMA这几套机制，每一套又是为了解决上一套留下的什么麻烦"。读完之后，你应该能在脑子里画出一条从"接口"到"8259A"再到"8237A"的演化链条，并且对每一个新部件都能说出"它是为了解决谁的问题而存在的"，而不是把这些芯片当成几个孤立的英文缩写去死记硬背。

## 知识体系总览

在进入正文前，先看清整章的分层逻辑——每一层在回答一个不同层次的问题，二级展开的是该层的具体知识点：

```
§6 输入输出

├─ 第一层：根本矛盾与接口基础（回答"CPU和外设为什么必须靠接口打交道"）
│   ├─ 矛盾：外设与CPU在速度、信号类型、数据格式上三重不匹配
│   ├─ 接口 vs 端口：接口=整套软硬件机制；端口=接口内可寻址的具体寄存器
│   ├─ 接口内4个寄存器：数据输出 / 数据输入 / 状态 / 控制
│   └─ 接口四大功能：地址译码选择 / 数据缓冲锁存 / 格式电平转换 / 传送协调
│       └─ ⚠️易错：状态字、控制字走数据总线，地址总线只管"选谁"
│
├─ 第二层：五种传送方式（回答"CPU用什么节奏跟外设交换数据"）
│   ├─ 核心矛盾：CPU该不该死等外设？——五种方式是这道题的五级解法
│   ├─ 无条件传送：外设永远就绪，CPU直接IN/OUT（前提：随时可取送）
│   ├─ 程序查询：CPU不停轮询状态位，就绪才传——CPU被占满，效率低
│   ├─ 程序中断：外设就绪时主动发INTR，CPU暂停后去处理——CPU可并行
│   ├─ DMA：DMA控制器接管总线，外设↔内存直传，CPU完全不参与过程
│   └─ I/O处理机：专用处理机独立承担全部I/O，连管理都不需CPU介入
│       └─ ❌口径不一：题库对"分几种"表述不统一，作答建议五种都提
│
├─ 第三层：中断方式的芯片实现——8259A（回答"多源中断怎么仲裁、怎么落地"）
│   ├─ 内部寄存器分工
│   │   ├─ IRR：谁在排队（来了请求即锁存置1）
│   │   ├─ ISR：谁正在被服务（置1直到收到EOI命令才清0）
│   │   ├─ IMR：谁被屏蔽（OCW1写入，某位=1该路请求被禁止）
│   │   └─ PR（优先级分辨器）：按规则从IRR中裁出当前该服务谁
│   ├─ 中断类型码的形成
│   │   ├─ 高5位：ICW2由程序设定
│   │   ├─ 低3位：8259A按IRx引脚编号自动填充
│   │   └─ 用途：向CPU亮明身份→查中断向量表→找到ISR入口
│   ├─ 工作方式
│   │   ├─ 全嵌套：IR0固定最高，服务中只挡同级/低级，高级仍可插队
│   │   ├─ 循环优先级：服务过的自动/指定排到最低，动态均衡
│   │   ├─ 特殊全嵌套：级联专用，放开从片内部高优先级插队限制
│   │   └─ 多片级联：总中断源数 = [8-(N-1)] + (N-1)×8
│   ├─ 中断结束方式
│   │   ├─ 自动结束（AEOI）：第2个INTA脉冲后沿自动清ISR
│   │   ├─ 一般EOI：清当前ISR最高非零位，仅适用全嵌套
│   │   └─ 特殊EOI：带L2~L0编码指定清哪一位，适用任意优先级方式
│   └─ 初始化编程（ICW1→ICW2→[ICW3]→[ICW4]，顺序固定）
│       ├─ ICW1：是否级联、触发方式（边沿/电平）、是否需ICW4
│       ├─ ICW2：中断类型码高5位
│       ├─ ICW3：主片标明哪些IR接了从片；从片标明接在主片哪个IR
│       └─ ICW4：特殊全嵌套 / 缓冲方式 / AEOI / CPU类型
│
└─ 第四层：DMA方式的芯片实现——8237A（回答"外设和内存怎么绕开CPU直接对话"）
    ├─ 主从模态（8237A的双重身份）
    │   ├─ 从态：CPU是总线主控，8237A只是普通端口，被CPU读写配置
    │   └─ 主态：8237A接管总线控制权，自己发地址+读写信号，传完还回去
    │       └─ ⚠️ IOR/IOW双向：因为两种模态下信号方向相反
    ├─ 传送方式（借到总线后借多久）
    │   ├─ 单字节：传一个字节就释放总线，最公平但效率最低
    │   ├─ 数据块：一次传完整块才释放，效率最高但占用总线时间长
    │   ├─ 请求传送：类似数据块，但每传一字节就采样DREQ，无效则暂停
    │   └─ 级联：从片HRQ/HLDA接主片DREQ/DACK，扩展通道数
    ├─ 传送类型（数据往哪个方向流）
    │   ├─ DMA读：MEMR+IOW有效，存储器→I/O设备
    │   ├─ DMA写：IOR+MEMW有效，I/O设备→存储器
    │   └─ DMA校验：伪传输，控制线无效，只走流程自检芯片功能
    ├─ 启动方式：①外设DREQ信号 / ②软件编程内部启动
    └─ 结束方式：①字节计数器减至FFFFH（TC信号）/ ②外部EOP负脉冲
```

这四层不是四个互相独立的知识点堆砌，而是一条因果链：第一层的"接口"是地基，没有它CPU连外设的门都摸不到；第二层提出了同一道矛盾（CPU该不该死等）的五级解法，是本章真正的"出题层"；第三层和第四层**没有引入任何新矛盾**，它们只是把第二层里"中断"和"DMA"分别落地成具体芯片（8259A、8237A）。换句话说，**第一、二层是"提出问题"，第三、四层是"工程实现答案"**。这也是为什么8259A和8237A细节哪怕记混了，只要记得各自要解决的核心矛盾，也能推导回大半内容。

## 第一部分：先问一个问题——CPU为什么不能直接接外设？

设想最朴素的方案：CPU的引脚直接拉一根线接到键盘、接到打印机、接到硬盘……行不行？

完全不行，原因课件里写得很直白：外设种类繁多，工作原理、速度、信号类型、信息格式跟CPU完全不是一个语言体系。CPU内部跑的是几纳秒一个周期的并行数字信号；键盘可能几十毫秒才按一次键；打印机吐字符的速度比CPU慢几个数量级；有的外设给的是模拟电压不是数字量；有的外设走串行一根线传数据，CPU内部走的是并行总线。

这就好比让一个只会说普通话、语速飞快的人，直接和说各地方言、语速参差不齐的一群人对话——必须有个**翻译官**夹在中间，把双方的语言、节奏都做转换。这个翻译官，就是**接口**。

接口和"端口"是两个层次的概念，很容易混：**接口**是这整套软硬件翻译机制的统称；**端口**是接口电路里那几个"CPU能直接用指令读写"的具体寄存器。打个比方，接口像是一整间"前台办公室"，端口则是办公室里那几个贴着编号、CPU可以直接敲门取送文件的"窗口"。

从用户编程的角度看，一个标准接口本质上就是**4个窗口（寄存器）**：

```
                  ┌─────────────────────────┐
   数据总线 ◄────►│      数据输出寄存器      │──────► 数据 → I/O设备
                  ├─────────────────────────┤
                  │      数据输入寄存器      │◄────── 数据 ← I/O设备
   地址总线 ─────►│ 地址译码 │
                  │ 选择逻辑 ├─────────────────────────┤
                  │          │      状态寄存器          │◄────── 状态 ← I/O设备
   控制总线 ─────►│          │      控制寄存器          │──────► 控制 → I/O设备
                  └─────────────────────────┘
        CPU              I/O 接口                       I/O 设备
```

这4个窗口分别承担四项任务（接口的四大功能）：
- **地址译码或设备选择**——系统里设备很多，必须先"译码"出CPU这次到底想找谁；
- **数据缓冲和锁存**——数据总线是大家共用的，没被选中的设备必须"闭嘴"（呈高阻态），被选中的才能临时借用总线传数据；
- **信息格式与电平的转换**——CPU是并行数字信号，外设可能是串行、可能是模拟电压，接口负责"翻译"；
- **数据传送的协调**——这是承上启下的一条，它直接引出了下面整个第二部分的内容：CPU和外设到底按什么节奏交换数据？

❗常见误解：很多人下意识觉得"状态字、控制字"这种内容性信息应该走"控制总线"传输——其实不是。**地址总线只负责"选谁"，数据总线负责"传什么内容"（包括普通数据、状态字、控制字），控制总线只负责传"读/写时序信号本身"**。这是真题里专门考过的一个易错点。

## 第二部分：CPU和外设打交道的五种"性格"

接口解决了"翻译"的问题，但还有一个更核心的矛盾没解决：**外设的数据什么时候准备好，CPU怎么知道？** 围绕这一个矛盾，历史上演化出了五种解法，每一种都是对上一种缺陷的修正。可以用"等快递"来类比这五种方式：

**① 无条件传送方式**——假设快递员永远站在你家门口、随叫随到。CPU完全不查任何状态，直接用IN/OUT指令读写固定端口（课件举例端口地址80H），靠RD/WR信号区分是取件还是寄件。这种方式只在一种场景下成立：**外设可以随时提供或接收数据**（比如永远不缺纸的打印机、永远在线的开关量）。一旦这个假设不成立，无条件传送就会读到脏数据或者丢数据。

**② 查询传送方式**——既然外设不一定随时就绪，那就让CPU**不停去问**："你好了没？你好了没？"——读状态端口、测试状态位、没好就跳回去继续问（`JZ`循环）。这解决了"外设没准备好"的问题，但带来一个新麻烦：**CPU被这件事完全占满了，干不了别的**，查询期间CPU和外设只能串行工作，CPU的高速运算能力被白白浪费在"傻等"上。

**③ 程序中断方式**——既然CPU不该傻等，那就反过来：**外设准备好了，主动喊CPU**（发中断请求信号INTR），CPU平时正常干自己的事，只有外设喊它的瞬间才暂停手头工作、跑去处理、处理完再回来接着干。这解决了"CPU傻等"的问题，CPU和外设之间终于能并行工作了。但中断方式也有代价：每次中断CPU都要保护现场（压栈）、跑去执行中断服务程序、再恢复现场（出栈），这些"附加动作"本身要花时间，而且一般一次中断只传一两个字节，要是数据量很大（比如整段磁盘数据），中断的次数会多到附加开销反而成了瓶颈。

**④ DMA方式**——既然中断方式的瓶颈在于"每次还是要CPU去跑一趟中断服务程序"，那干脆**让CPU完全不参与传输过程**：专门造一个DMA控制器，让它直接接管总线，外设和内存之间点对点直传，CPU只在传输开始前和结束后露个面。这一步是质的飞跃：**不经过CPU、数据直接在存储器和外设间传输，速度最快**，特别适合外设速度快、一次传输数据量大的场合（比如磁盘读写）。

**⑤ I/O处理机方式**——DMA虽然不需要CPU参与传输过程，但DMA控制器本身的初始化、参数设置还是CPU在管。如果连"管理"这件事都不想让CPU操心，那就引入一个有自己指令系统、能独立执行程序的**I/O处理机**，把全部输入输出操作彻底承包出去，CPU和I/O处理机各自独立运行、互不打扰。

把这五种方式按"CPU参与程度"排成一条曲线，正好是从"完全不查"到"全程紧盯"再到"完全托管"的一个来回：

| 方式 | CPU参与程度 | 适用场合 | 解决了谁的问题 |
|---|---|---|---|
| 无条件传送 | 几乎为0（但有前提） | 外设永远就绪 | —（基线情形） |
| 程序查询 | 全程占用 | 慢速、数据量小、效率要求不高 | 外设不一定就绪 |
| 程序中断 | 仅响应瞬间占用 | 较慢、随机性强、需要并行 | 查询方式CPU被占满 |
| DMA | 几乎不参与传输 | 快速、大数据量 | 中断方式附加开销多、每次传太少 |
| I/O处理机 | 完全不参与管理 | 复杂系统、要求高度并行 | DMA仍需CPU管理配置 |

❌易错提醒：题库里对"数据传送方式到底分几种"存在两种不完全一致的口径——一种把"无条件传送"算作独立的第一种，凑够四种（无条件/查询/中断/DMA）；另一种则不提无条件传送，而是把"I/O处理机方式"算进去凑四种（查询/中断/DMA/I-O处理机）。两种口径其实并不矛盾，只是切入角度不同：前者是"按CPU要不要查状态"分类，后者是"按课件6.3节明确给出的四种基本控制方式"分类。**作答策略**：简答时五种都提一句最稳妥，重点详细展开中断方式与DMA方式的对比（这是题库和真题反复强调的部分）。

四种方式中，**中断**和**DMA**是分值最重的两块，分别由专用芯片8259A和8237A实现——下面两部分就是把第二部分这两条"高阶解法"做成具体硬件的过程。

## 第三部分：中断方式的专职管家——8259A

### 3.1 为什么需要一颗专门的"中断仲裁芯片"

第二部分讲到"外设主动喊CPU"，但现实中CPU要同时管理好几个外设（键盘、打印机、磁盘……），如果它们同时喊该怎么办？谁先被处理、谁等着、CPU怎么知道是谁在喊——这些"仲裁"工作如果全靠CPU自己一条条判断，CPU又会被拖回"傻忙"的老路。于是需要一颗专职的**中断仲裁芯片**，把"谁先被服务、怎么告诉CPU是谁"这件事统一管起来——这就是8259A可编程中断控制器。可以把它想象成一个总机前台：多路分机（中断源）同时打进来，前台负责排队、判断优先级、最后告诉老板（CPU）"是几号分机找您"。

8259A每片能管8级优先权中断源，通过级联可扩展到64级，并且能向CPU提供可编程的"身份标识码"（即中断类型码），还能单独屏蔽某一路、支持多种工作方式、可以和8086~Pentium直接连接不需要外加电路。

### 3.2 内部结构：四个寄存器的分工

总机前台要完成仲裁，内部需要记三本账：谁在排队、谁正在被接待、谁被屏蔽掉了不接。8259A正好对应三个寄存器，外加一个负责"排队规则"的裁决器：

```
   IR0~IR7 ──► IRR（中断请求寄存器）："谁在排队"
                  │  来了请求就把对应位锁存为1
                  ▼
               优先级分辨器 PR  ：按优先级规则挑出当前该服务谁
                  │
                  ▼
               ISR（中断服务寄存器）："谁正在被接待"
                  │  被选中服务的对应位置1，
                  │  直到收到中断结束命令才清0
                  ▼
               控制逻辑 ──► 向CPU发 INT=1

   IMR（中断屏蔽寄存器，即OCW1）："谁被前台拒之门外"
      某位=1，对应的IRx请求直接被屏蔽，连IRR都不会被处理
```

- **IRR**：所有IR输入线来的请求都先在这里排队等候、被锁存；
- **ISR**：记录"当前正被CPU服务"的那一级，在被服务期间一直保持置1，直到中断结束命令把它清掉；
- **IMR**：CPU可以随时设置屏蔽哪几路请求；
- **PR（优先级分辨器）**：决定IRR中谁能"晋级"到ISR、能不能向CPU申请中断。

理解了这四个寄存器的分工，后面"两个INTA响应周期具体做了什么"就很自然了：CPU回送两个INTA负脉冲——**第一个**负责"锁定服务对象"（ISR置1、IRR对应位清0），**第二个**负责"上报身份证号"（把类型码送上数据总线）。这两步分别对应上图里ISR的"置位"动作和下面3.3节要讲的"类型码"环节，本质上没有引入任何新概念，只是把IRR/ISR的状态变化和"亮明身份"这件事在时间上拆成了两步。

### 3.3 中断类型码是怎么"拼"出来的

CPU收到中断请求后，需要知道"具体是哪个中断"，才能去中断向量表里查到对应的服务程序入口——这个"身份证号"就是中断类型码，它不是凭空给的，而是**程序设定的高位 + 硬件自动填的低位拼出来的**：

$$中断类型码 = \text{ICW2高5位（程序写入）} \;+\; \text{IRx对应的低3位（硬件按引脚自动填充）}$$

举例：ICW2写入20H（即0010 0000B），如果中断源接在IR1引脚上，硬件会自动把低3位填成001B，拼出来的类型码就是0010 0001B = 21H。再结合第2章学过的"向量地址 = 4×类型码"，就能算出 4×21H = 84H，去内存0000H段、偏移84H处取出4字节，正是这条中断服务程序的入口地址。

这一步把"8259A仲裁出谁该被服务"和"CPU怎么找到对应的服务程序"两件事彻底打通了：ICW2是接口芯片这边设置的"门牌号前缀"，IRx编号是"具体房间号"，两者拼起来才是完整地址，最后还要靠第2章的中断向量表机制才能落地到具体代码。

### 3.4 工作方式：同一颗芯片怎么适应不同场景

8259A有10种工作方式，但考试集中考察的是"优先级怎么排"和"服务怎么结束"这两类问题，背后对应的都是现实场景里会遇到的具体麻烦：

**优先级怎么排——全嵌套 vs 循环优先级**：默认情况下8259A用**全嵌套方式**：IR0固定最高、IR7固定最低，正在服务某一级时，同级和更低级的请求会被挡住，但更高级的请求依然能插队。这符合大多数场景，但如果8路外设的重要性其实是动态变化的（比如这次轮到谁服务、谁就该排到最后），就需要**循环优先级方式**——自动循环（服务过的自动排到最低，下一个变最高）或特殊循环（编程指定谁是最低）。

**级联场景下还要不要嵌套——特殊全嵌套方式**：如果系统里接了多片8259A（一主多从），普通全嵌套规则放到从片身上会出问题：从片本身内部也有8级优先级，如果从片的一个中断正在被服务时，从片内部更高优先级的中断却因为"全嵌套规则"被挡住了，那就违背了"高优先级要能插队"的初衷。**特殊全嵌套方式**就是为这种级联场景定制的：放开"从片内部更高优先级仍可申请"的限制；并且某个中断退出服务前，CPU要软件检查"它是不是这个从片里唯一的中断"，确认没有遗留中断了才能真正结束主片对这个从片的服务。

顺着级联往下想，就会遇到一道经典计算题：**N片8259A级联最多能管多少个中断源**。主片自己有8个IR口，其中(N-1)个被拿去接从片了（这些口不再直接挂中断源），剩下 $8-(N-1)$ 个口还能直接挂中断源；而每个从片各自又能贡献8个，一共 $(N-1)\times 8$ 个。两部分相加：

$$总中断源数 = [8-(N-1)] + (N-1)\times 8$$

比如6片级联：$(8-5)+5\times8=3+40=43$。

**服务怎么结束——三种中断结束方式**：回到3.2节提过的ISR寄存器，它"被服务对象置1"之后，到底什么时候、靠什么方式清0，对应三种结束方式：①**自动结束**——靠硬件，第二个INTA负脉冲后沿自动清掉；②**一般中断结束命令**（非特殊EOI）——靠软件发命令，但只能清"当前ISR里优先级最高的那个非零位"，只适用于全嵌套这种"谁该被清"显而易见的场景；③**特殊中断结束命令**——命令里直接带编码指定要清哪一位，因为非全嵌套场景（比如特殊全嵌套）下"谁该被清"不再是显而易见的，必须显式指定。

### 3.5 初始化编程：把所有配置打包写进ICW1~ICW4

前面讲的级联结构、触发方式、类型码、工作方式选择……这些配置项最终都要通过四个初始化命令字（ICW1~ICW4）写进8259A。因为ICW1走偶地址、ICW2~ICW4都走奇地址需要靠顺序区分，写入顺序是固定的：

```
ICW1（必写）── 设置：是否级联、触发方式（边沿/电平）、是否需要ICW4
   │
   ▼
ICW2（必写）── 设置：中断类型码高5位
   │
   ▼
是级联方式？──Y──► ICW3 ── 主片：标明哪些IR接了从片
   │                       从片：标明自己接在主片哪个IR
   N
   │
   ▼
需要ICW4？──Y──► ICW4 ── 设置：特殊全嵌套/缓冲方式/自动结束方式/CPU类型
```

可以看到，ICW1实际上是整个流程的"入口判断"——它决定了后面要不要走ICW3、要不要走ICW4，这正好呼应了3.4节讲的"级联场景需要专门配置（ICW3）、自动结束方式需要专门配置（ICW4里的AEOI位）"。**初始化编程不是新知识，它是把3.2~3.4节学到的所有配置项打包归档的收尾动作。**

## 第四部分：让外设和内存"直接对话"——8237A与DMA

### 4.1 中断方式不够用在哪

中断方式解决了"CPU不用傻等"的问题，但它有一个绕不开的代价：**每次中断CPU都要保护现场、执行中断服务程序、再恢复现场**，这些附加动作本身要花时间，而且通常一次中断只传一两个字节。如果要传的是一整块数据（比如从磁盘读一个文件），中断次数会多到附加开销本身成了瓶颈。DMA方式的核心思路就是：**既然瓶颈在"CPU要跑一趟"，那就让CPU完全不参与传输过程**，由专门的硬件直接接管总线、让外设和内存点对点直传。

### 4.2 主从模态：8237A的双重身份

8237A最特别的地方在于它有**两种身份**，可以类比成"代驾"：平时8237A只是一个普通的I/O端口，乖乖被CPU读写配置（**从态方式**，此时CPU是老板，掌握总线方向盘）；但一旦外设有DMA请求、CPU也批准了，8237A就会"借走车钥匙"，自己掌握总线控制权，直接向地址总线发地址、向存储器和外设发读写信号（**主态方式**，此时8237A自己开车），传完之后再把总线控制权还给CPU。

```
平时（从态）：           DMA传送时（主态）：
  CPU ──读写──► 8237A      8237A ──直接控制──► 地址总线/数据总线/读写信号
  （8237A被动，像普通端口）  （8237A主动，像CPU一样掌握总线）
```

这也解释了一道常被问到的细节：**8237A的IOR、IOW信号线为什么是双向的**——因为它既要在从态下"被CPU读写"（此时信号是CPU发给8237A方向），也要在主态下"自己发读写信号控制外设/存储器"（此时信号是8237A发出方向），双重身份决定了信号线必须双向。

DMA传送启动靠两种方法（外设发DREQ请求 / 软件编程内部启动），结束也靠两种方法（字节计数器减1由0变FFFFH产生TC信号 / 外部输入$\overline{EOP}$负脉冲强制结束）——这两点课件里专门列出过，但题库没有单独出题，需要直接记住。

### 4.3 四种传送方式：借到总线后怎么用

借到总线控制权之后，"一次借多久、传多少"是一个需要灵活配置的问题，8237A提供了四种传送方式，可以类比成不同的"搬家服务模式"：

- **单字节传送方式**——每次只搬一件家具（一个字节）就归还总线，下次还要重新申请。最不占用总线资源，但效率最低，适合不想长期独占总线的场合。
- **数据块传送方式**——一次申请，把一整车家具（一整块数据）搬完才归还总线，靠字节计数器减到FFFFH（产生TC）或外部$\overline{EOP}$信号判断"搬完了"。效率最高，但占用总线时间长，会让CPU等得更久。
- **请求传送方式**（查询方式）——类似数据块传送，但每搬一件就回头看一眼"外设的门（DREQ）还开不开着"，DREQ无效就先停下，是单字节和数据块之间的折中。
- **级联传送方式**——把扩展芯片的HRQ、HLDA分别接到主片某个通道的DREQ、DACK上，相当于"分包给二级搬家公司"，用来扩展DMA通道数。

### 4.4 三种传送类型：数据往哪流

注意区分"传送**方式**"（4.3节讲的，描述怎么借用总线、借多久）和"传送**类型**"（这里讲的，描述数据具体往哪个方向走）——这是最容易混淆的一对概念：

- **DMA读**：8237A输出$\overline{MEMR}$和$\overline{IOW}$，把存储器的数据读到I/O设备；
- **DMA写**：8237A输出$\overline{IOR}$和$\overline{MEMW}$，把I/O设备的数据写到存储器；
- **DMA校验**：一种"伪传输"，走完整套地址、响应流程，但实际的读写控制线都无效，专门用来自检8237A本身的读写功能是否正常，不传真实数据。

### 4.5 时序：速度怎么对齐

DMA传送同样要面对"CPU和外设速度不匹配"这个第一部分提到的老问题，体现在8237A的三种总线周期时序上（课件未给出文字定义的细节，以下按通用知识补充，建议对照课件原文核实）：正常时序是标准基准节奏；压缩时序利用CPU空闲周期把一次传送压缩进更少的时钟周期，传得更快；扩展时序则相反，当外设/存储器较慢时插入等待状态（靠READY信号），把周期拉长以等慢设备跟上。

## 第五部分：把五种方式拉回同一张表

走到这里，第二部分提出的五种方式，已经有两种（中断、DMA）被拆解成了具体芯片的工作原理。现在可以把整章串成一张总览表：

| 方式 | 落地芯片/机制 | CPU是否参与传输过程 | 一次能传多少 |
|---|---|---|---|
| 无条件传送 | 无（端口直读写） | 不参与，但前提苛刻 | 不限 |
| 程序查询 | 无（软件循环） | 全程参与（盯着状态位） | 不限，但慢 |
| 程序中断 | 8259A | 仅响应瞬间参与 | 通常一两个字节/次 |
| DMA | 8237A | 几乎不参与 | 一大块数据/次 |
| I/O处理机 | 专用I/O处理机 | 完全不参与管理 | 不限 |

这张表也是本章选择题里常考的"哪个芯片能接管总线"这类识别题的答案来源——8259A、8255A、8253A都只能被CPU读写（从属地位），只有8237A具备"接管总线"的能力，因为只有DMA方式真正需要"借走总线控制权"这件事。

## 第六部分：整章知识连接图

```
接口（翻译官，解决CPU与外设互不匹配）
   │
   ▼
CPU该不该死等外设？──► 无条件传送（外设永远就绪，最理想情形）
   │
   ▼
外设不一定就绪 ──► 程序查询方式（CPU不停问，效率低）
   │
   ▼
CPU被查询占满 ──► 程序中断方式（外设主动喊，CPU并行工作）
   │                  │
   │                  ▼
   │           8259A：仲裁多源(IRR/ISR/IMR/PR)
   │              → 拼类型码(ICW2+IRx)
   │              → 适配场景(全嵌套/循环/特殊全嵌套/级联)
   │              → 打包初始化(ICW1~ICW4)
   ▼

                                │
                                ▼
                         8237A：借总线(主从模态)
                            → 借多久(四种传送方式)
                            → 传哪个方向(三种传送类型)
                            → 对齐速度(三种时序)
   │
   ▼

```

这条链最重要的特征是：**每往下走一步，都是在修正上一步暴露出来的具体缺陷**，而不是凭空多出一个新功能。中断方式不是"无缘无故"比查询方式高级，它是专门为了解决"CPU被查询占满"这个具体问题而存在的；DMA方式同理，是专门为了解决"中断方式附加开销大、传输量小"这个具体问题而存在的。理解了这条因果链，8259A和8237A里那些看起来繁琐的寄存器、命令字细节，都能在"它具体解决了链条上哪一环的问题"这个框架里找到位置，而不是孤立的记忆负担。

## 第七部分：本章要带走的几件事

1. **接口是翻译官，端口是窗口**——接口解决"CPU和外设语言不通"的问题，本质是4个寄存器（数据输出/数据输入/状态/控制）+地址译码逻辑；状态字、控制字这些"内容"都走数据总线，地址总线只负责"选谁"。
2. **五种传送方式是同一道题的五级解法**——这道题是"CPU该不该死等外设"，无条件传送→查询→中断→DMA→I/O处理机，参与程度逐级递减，每一级都在修正上一级暴露的具体缺陷，不是孤立的并列关系。
3. **8259A的核心是"仲裁+亮身份"**——IRR记录谁在排队、ISR记录谁在被服务、IMR记录谁被屏蔽，配合优先级分辨器完成仲裁；中断类型码＝ICW2程序设定的高5位+IRx硬件自动填的低3位，这个类型码再接到第2章的"向量地址=4×类型码"才能真正找到服务程序入口。
4. **8237A的核心是"借总线"**——从态时是普通端口被CPU管，主态时反客为主直接控制总线，这个双重身份解释了为什么IOR/IOW必须双向；四种传送方式的差异本质是"一次借多久"，三种传送类型的差异本质是"数据往哪流"。

本章讲的8259A和8237A，是"控制方式"这个抽象问题（中断怎么仲裁、DMA怎么借总线）的两套专用芯片实现，它们解决的是"CPU和外设之间数据怎么流动"这个调度层面的问题。但还有一个问题这两颗芯片都没回答：**具体一个外设（键盘、打印机、定时信号、串口）要怎么真正接到系统总线上、寄存器怎么编址、控制字怎么写**——这正是第7章可编程接口芯片（8255A通用并行接口、8253定时器/计数器、8251A串行通信接口）要解决的问题。可以把第6章理解成"骨架"（调度逻辑），第7章是"骨架上挂的肉"（具体怎么接、怎么编程）。
