# 第9章 人机交互接口 · 自学理解手册

> **这份资料的目标**：不是让你死记段码表和扫描算法，而是让你理解"计算机为什么需要这套接口、每个设计决定背后在解决什么问题"。读完之后，你应该能在脑子里建立一张从"用户按下一个键"到"数码管亮起对应数字"的完整信号流图，理解键盘消抖、行扫描、线反转、段码查表每一步存在的理由。

## 分层知识树

```
第9章 人机交互接口
│
├─ 【第一层】回答"人如何把意图传给计算机"的问题
│   ├─ 键盘是最基本的输入设备，按键触发电平变化，CPU读取后识别
│   ├─ 编码键盘：硬件自动产生ASCII码，使用方便但成本高
│   ├─ 非编码键盘：仅提供行列矩阵信号，软件负责全部识别逻辑
│   └─ 非编码键盘需要解决四个问题：去抖动、重键处理、按键识别、产生编码
│
├─ 【第二层】回答"为什么不能直接读引脚，一定要消抖"的问题
│   ├─ 机械触点在10～20ms内反复抖动，一次按键产生多次电平跳变
│   ├─ 硬件消抖：每键加RC滤波或RS锁存电路，成本高，占用电路板面积
│   ├─ 软件消抖：检测到低电平后延时20ms，再次采样确认，才读取键值
│   └─ 软件消抖的本质：让延时区间完全覆盖抖动区间，滤掉假信号
│
├─ 【第三层】回答"矩阵键盘如何知道哪个键被按下"的问题
│   ├─ 行线接输出口，列线接输入口；N行×M列支持N×M个按键，仅需N+M根线
│   ├─ 行扫描法：逐行输出低电平扫描字（11111110B→11111101B→…），读列值定位
│   ├─ 线反转法：先行输出全0读列值，再交换方向输出列值读行值，两步定位
│   └─ 两法对比：行扫描接口简单（固定方向）但软件复杂；线反转软件简单但需可编程方向接口
│
├─ 【第四层】回答"OS层如何让应用程序使用键盘"的问题
│   ├─ INT 09H（BIOS硬件中断）：有键按下时自动触发，读扫描码写入键盘缓冲区队尾
│   ├─ 键盘缓冲区：32字节循环队列（16次击键），建立在BIOS数据区，先进先出
│   ├─ INT 16H（BIOS软件调用）：从缓冲区队首读字符，AH=0阻塞等待，AH=1非阻塞检测
│   └─ INT 21H（DOS系统功能调用）：应用程序使用的高级接口，AH=1回显输入，AH=2显示字符
│
├─ 【第五层】回答"计算机如何把信息展示给人"的问题
│   ├─ LED七段显示器由a/b/c/d/e/f/g七段发光二极管组成，不同段组合点亮对应不同字形
│   ├─ 共阴极：COM接GND，要亮某段给高电平1；共阳极：COM接VCC，要亮某段给低电平0
│   ├─ 段码由字形决定，共阴与共阳互为反码（如"0"：共阴3FH，共阳C0H）
│   └─ 段码转换方法：硬件译码（专用芯片自动转换）或软件译码（程序查表XLAT指令）
│
├─ 【第六层】回答"多位显示器如何节省I/O口"的问题
│   ├─ 静态显示：每位独占一个8位输出口，段码持续输出，稳定但占用I/O口多
│   ├─ 动态显示：多位共用段码线，位码轮流选通，利用视觉暂留（约5ms/位）造成同时点亮假象
│   ├─ 动态显示两路控制线：段码口（决定显示什么字形）+ 位码口（决定哪位被选中）
│   └─ N位动态显示只需1个段码口+1个位码口（最少N位），而N位静态显示需N个段码口
│
└─ 【第七层】回答"8255A如何把键盘和显示器连起来"的问题
    ├─ 8255A三个8位并行I/O口（PA/PB/PC）可独立配置输入/输出方向
    ├─ 键盘接PC口：线反转法要求PC高4位与低4位方向能动态切换（重写控制字实现）
    ├─ 显示接PA/PB口：静态显示时A口接十位段码，B口接个位段码，双口同时输出
    └─ 8253配合：按键时用计数器产生1kHz方波驱动蜂鸣器，方式3（方波发生器），初值=CLK/1000
```

**各层串联逻辑**：第一层和第二层处理的是同一个问题——"读到的信号可信吗"——前者是结构层面的问题（硬件有几种），后者是信号层面的问题（怎么去除噪声）。第三层是真正的"识别"，依赖前两层建立的可信信号基础。第四层把硬件识别的键值交给操作系统管理，形成缓冲区机制，使应用程序不必直接轮询硬件。第五六七层是对称的输出侧：第五层定义了"能显示什么（字形与段码）"，第六层解决"怎么显示更多位（节省I/O）"，第七层把所有硬件用8255A的三个口统一组织起来。两条主线（键盘输入、LED输出）在第七层汇合，共用同一片8255A和同一套初始化程序。

## 第一部分：为什么需要"人机交互接口"这一章

前八章讲的都是计算机自身的事——CPU怎么组织寄存器，内存怎么分段分页，接口芯片怎么做定时和串并行通信。这些都是机器和机器之间的对话。

但计算机最终要服务于人。人如何把目标告诉计算机？计算机又怎么把结果展示给人？这就是第9章要解决的问题。

从硬件角度看，人机交互接口面临的核心矛盾是：**人的动作是模糊的、慢速的、机械的；而CPU的信号是精确的、高速的、数字的**。键盘上的机械触点无法像数字门电路那样干净地在0和1之间切换——它会抖动。LED显示器的发光原理和CPU的数字输出口也不能直接相连——需要段码转换和限流驱动。这一章的每一个设计，都是在弥合这条鸿沟。

---

## 第二部分：按键为什么会"抖动"，软件消抖又在干什么

想象一个弹簧门：你推门，门会向内打开，但如果弹力强，门会弹回来再推开你，来回几次才最终稳定。机械按键的触点也一样——按下瞬间，两个金属片接触，但由于弹性，会在10～20毫秒内反复分离和接触几次，才最终稳定在"闭合"状态。

这意味着，如果CPU在抖动期间不停读取引脚，它看到的是：低、高、低、高、低……而非一次干净的"低电平"。于是一次按键变成了多次按键事件——这不能接受。

**硬件消抖**是给每个按键加RC滤波电路，让电容吸收掉那些短暂的电平变化，相当于给弹簧门加了阻尼器。效果好，但每个按键都要一套电路，成本随按键数线性增加。

**软件消抖**换了个思路：我不试图过滤那些抖动，我只是等它结束。

```
检测到低电平（疑似有键按下）
         │
         ▼
  等待 20ms（CALL D20ms）
         │
         ▼
  再次读取引脚
         │
    ┌────┴────┐
  仍为低    已为高
    │          │
  确认有键  是干扰，
  按下，开   回头重
  始识别     新查询
```

这个策略的聪明之处在于：抖动时间最长约20ms，只要第一次检测和第二次检测之间的间隔超过抖动时间，两次都是低电平就可以确认是真实按键。程序代价极低——一个延时子程序和一次重新读取。

## 第三部分：矩阵键盘——用乘法代替加法来节省I/O线

最直观的键盘设计是每个键连一根I/O线，16个键就需要16根线。但矩阵键盘发现了一件事：**不需要每个键独占一根线，只需要独占一个"坐标"**。

把16个按键排成4行4列，行线4根、列线4根，共8根线就能支持16个按键。一般地，N行M列的矩阵键盘用N+M根线支持N×M个按键——这是加法和乘法的区别。

但这带来了一个代价：**CPU不能直接"问"某个按键是否被按下，而必须通过"扫描"来推断**。就像在一个二维迷宫里找出口，你不知道出口在哪，只能一行一行地探测。

### 行扫描法——老老实实逐行查

行扫描法是最直接的策略：一次只让一行为低电平，其余行都是高电平，然后看列线上有没有低电平。如果某列变低了，说明这行和这列的交叉点有按键被按下。

```
初始预检（行全0，判断有没有任何键）：
  行输出：0000
  读列线：若全1→无键；若有0→有键，消抖后开始扫描

逐行扫描（已确认有键，找是哪行哪列）：
  第一轮：行输出 11111110B（第0行低）→ 读列线
  第二轮：行输出 11111101B（第1行低）→ 读列线
  ...依此类推

找到列线有0的那一轮：
  行扫描值 → 行号（高4位）
  列线读值 → 列号（低4位）
  组合为位置码 → 查表得键号
```

行扫描法要求行线固定为输出，列线固定为输入，方向不变，接口设计简单。代价是程序需要一个循环逐行扫描，还需要一张键值对照表（TABLE）。

### 线反转法——用两步代替循环

线反转法是一个更聪明的算法，只需两步就能定位按键，不需要逐行循环。

**第一步**：行线方向=输出，列线方向=输入。行全输出0（所有行拉低）。这时如果某个按键被按下，对应的列线会被该行的低电平"拉低"。读列线，得到**列值**（哪几列有0）。

**第二步**：反转——行线变输入，列线变输出。把刚才读到的列值从列线输出出去。这时被按下的按键会把列的低电平"拉到"它所在的行。读行线，得到**行值**。

```
第一步（行→列）               第二步（列→行，"反转"）
┌──────────┐               ┌──────────┐
│ 行口(输出)│──全0──→键盘   │ 行口(输入)│←──读行值──键盘
└──────────┘               └──────────┘
┌──────────┐               ┌──────────┐
│ 列口(输入)│←──读列值──键盘 │ 列口(输出)│──上步列值──→键盘
└──────────┘               └──────────┘

两步读到：列值（哪列）+ 行值（哪行）→ 唯一确定按键位置
```

线反转法的前提是：接口必须能动态改变输入/输出方向——这正是8255A方式0的能力，通过重写控制字即可在一个时钟周期内切换PC口高低半字节的方向。

**两种方法的本质权衡**：行扫描把复杂度放在软件（循环+查表），硬件方向固定；线反转把复杂度放在硬件（需要可编程方向），软件只需两步读取。真题综合设计题明确要求"行列反转方式"，说明在拥有8255A的系统里，线反转是更受推荐的方式。

## 第四部分：键盘中断体系——从硬件到DOS的三个层次

按键识别完成后，键值要怎么交给应用程序？这里有一套分层的服务体系，三个中断号各司其职。

可以用一个快递系统来类比：

```
按键动作（快递员上门送包裹）
        │
        ▼
  INT 09H 硬件中断（收件员）
  ─────────────────────
  从键盘控制器60H端口读扫描码
  判断是哪种键（字符键/功能键/控制键）
  将ASCII码+扫描码写入键盘缓冲区队尾
        │
        ▼  （包裹放进小区快递柜）
  键盘缓冲区 32字节 FIFO循环队列
  （最多存16次击键，先进先出）
        │
        ▼
  INT 16H BIOS服务（你去取快递）
  ─────────────────────
  AH=0：从队首取字符，无字符就等（阻塞）
  AH=1：检测队列是否非空，不等待（非阻塞）
  出口：AL=ASCII码，AH=扫描码
        │
        ▼
  INT 21H DOS服务（物业帮你取快递）
  ─────────────────────
  AH=1：输入字符并回显（最常用）
  AH=7：输入字符不回显（密码输入）
  AH=2：显示DL中的字符
  AH=9：显示DS:DX指向的$结尾字符串
  AH=4CH：程序返回DOS
```

这三层的设计逻辑是：09H负责"硬件→缓冲区"的搬运，这是最底层的实时响应；16H负责"缓冲区→应用"的取用，可以阻塞也可以轮询；21H在16H之上再包一层DOS逻辑（比如回显、行编辑），让应用程序不必关心任何硬件细节。

从考试角度记忆：**09H写队尾（硬件写进来），16H读队首（应用读出去），21H是DOS高级调用**。

## 第五部分：LED七段显示器——把一个数字"翻译"成发光的几何形状

LED七段显示器的核心问题是：**数字"3"和字节值`4FH`之间没有任何直观联系，这个对应关系只能靠查表**。

理解为什么需要查表，先要理解七段显示器的结构。它有七个发光条（a～g）加一个小数点（h），每个条对应输出字节的一位：

```
  aaa
 f   b
 f   b      对应字节：D7 D6 D5 D4 D3 D2 D1 D0
  ggg                  h   g   f   e   d   c   b   a
 e   c
 e   c
  ddd  ●h
```

要显示"3"，需要让a/b/c/d/g这五段亮，f/e/h不亮。把这5段对应的位置1，其余清0，得到共阴极段码：

```
h g f e d c b a
0 1 0 0 1 1 1 1  =  4FH
```

这就是"3"的共阴极段码4FH的来历——不是计算出来的，是根据七段结构对照点亮哪些段"拼"出来的。

**共阴极 vs 共阳极**：共阴极的COM端接地，要点亮某段就给那段一个高电平（逻辑1）。共阳极的COM端接+5V，要点亮某段就把那段拉到低电平（逻辑0）——所以共阳极的段码是共阴极的**按位取反**。"0"的共阴极段码是3FH（00111111B），取反得C0H（11000000B）即共阳极段码。

这一点是本章最容易丢分的地方：段码表没有背错，但搞混了共阴/共阳，把3FH输出到共阳极显示器，结果显示的是"8"（全亮）。每次做题的第一件事：**确认是共阴还是共阳**。

## 第六部分：静态显示 vs 动态显示——用视觉暂留换I/O口数量

假设要显示一个三位数"153"。最直接的方法是三个显示位各连一个8位输出口，三个口分别持续输出段码——这就是静态显示。每位独立驱动，互不干扰，程序写起来极简单：

```asm
OUT PORT_A, '1'的段码   ; 百位一直亮
OUT PORT_B, '5'的段码   ; 十位一直亮
OUT PORT_C, '3'的段码   ; 个位一直亮
```

问题是：8位显示需要8个输出口，每个8255A只有3个口，显示8位数就要3片8255A。

动态显示发现了人眼的一个漏洞：**只要每位显示的频率超过每秒约25次，人眼就无法分辨它是在闪烁还是一直亮的**。于是可以让8个显示位共用一个段码口，每次只点亮一位，以约5ms/位的速度轮流扫描——每位每秒被点亮200次，人眼看到的效果是"8位同时亮"。

```
时间轴：
|--位0亮(5ms)--|--位1亮(5ms)--|--位2亮(5ms)--|--…--| ← 循环
人眼看到：所有位都亮（刷新率200Hz，远超视觉暂留阈值25Hz）
```

这样8位显示器只需：1个段码输出口（PA口）+ 1个位选输出口（至少8位，用PA口的N位或PB口）。代价是CPU必须持续不断地执行显示扫描循环，不能停。在实际系统中通常用定时中断（8253产生定时脉冲触发8259A中断）来周期性执行显示刷新，使其不占用主程序的时间。

 └─────────────────┘
        │ 得到按键位置码/键号

        ▼

        │
        ▼
【显示输出层：结果可视化】
 软件查表（XLAT指令）
   数字(0~9/A~F) → 段码(共阴3FH~71H / 共阳C0H~8EH)
        │
        ▼
 ┌─────────────────┐
 │   静态显示        │   每位独占端口，持续输出
 │   PA→十位段码    │   稳定，占I/O多
 │   PB→个位段码    │   ← 真题综合设计题要求此法
 └────────┬────────┘
          │ 两种显示方式

## 第八部分：本章要带走的五件事

**第一件：消抖是"两次采样"而不是"一次延时"。**  
延时本身不解决问题，必须在延时后重新采样确认。只延时不确认，等于什么都没做。

**第二件：线反转法的核心是"方向反转"，不是"电平反转"。**  
第一步是行输出列输入，第二步是行输入列输出——是接口方向的切换（重写8255A控制字），不是把电平取反。这个概念搞混会导致程序完全写错。

**第三件：共阴与共阳的段码互为反码，做题第一步先确认极性。**  
"0"在共阴极是3FH，在共阳极是C0H。真题综合设计题明确给出的是共阳极，用3FH会导致显示"8"（全亮）。

**第四件：三个键盘中断号的分工是"写→存→读"的流水线。**  
09H负责把扫描码写进缓冲区（硬件触发），16H负责从缓冲区读出（软件调用），21H在最上层包装了输入输出服务（应用使用）。它们之间不是替代关系，而是层叠关系。

**第五件：静态/动态显示的选择取决于位数和I/O资源。**  
位数少（如两位）且I/O充裕时选静态——程序简单稳定。位数多（如八位）时选动态——节省I/O但需要持续刷新。两者不是优劣关系，是不同约束下的不同解法。
