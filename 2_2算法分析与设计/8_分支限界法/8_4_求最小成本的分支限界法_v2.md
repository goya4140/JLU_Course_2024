# 8.4 求最小成本的分支限界法 — 复习资料 v2

> 本节的主线是一条递进的修补链：
> **BB算法直接截断（错）→ 改进判断时机（算法8.2）→ 对n-元组够用，对k-元组仍错
> → 引入U和u(X)（算法8.3/8.4/8.5）→ 对所有情况都正确，且效率更高**

---

## 一、出发点：为什么不能直接用 BB + LC-检索求最优解

### 1.1 原始算法 BB 的判断时机

算法 8.1（BB）在 `for E 的每个儿子 X` 循环**内部**，一旦发现某儿子是答案结点，**立即 `return`**。求可行解时没问题，但求**最优**解时会截断得太早。

**反例：**

```
              根 (ĉ=0)
        ┌──────────┴──────────┐
   L (ĉ=2)               R (ĉ=4)
   ┌────┴────┐            ┌────┴────┐
LL(cost=20) LR(∞)      RL(∞)   RR(cost=10)
  ★答案                           ★真正最优
```

BB + LC-检索执行：根展开 → L(ĉ=2)、R(ĉ=4) 入表 → LEAST 选 L →
展开 L 的儿子时**碰到 LL 是答案结点，立刻 `return`，输出 cost=20**。

R 的子树根本没有机会被搜索，真正最优的 RR(cost=10) 被永久错过。

---

## 二、第一次修补：算法 8.2——把判断时机挪到"选出 E 之后"

### 2.1 改动

不在生成儿子时判断，而是等 `LEAST` 把某结点选为新 E 之后，**循环顶端第一件事**就判断它是否是答案结点。

```
procedure LC(T, ĉ)
  E ← T;  将活结点表初始化为空
  loop
    if E是答案结点 then            ← ★判断在此：LEAST选出E之后
      输出从E到T的路径; return
    endif
    for E的每个儿子X do
      if B(X) then ADD(X); PARENT(X)←E endif
    repeat
    if 表为空 then print("no answer node"); return endif
    call LEAST(E)                  ← LEAST职责不变，只负责选出ĉ最小的E
  repeat
end LC
```

**与算法 8.1 的唯一区别**：`if X是答案结点` 从 `for` 循环内部移到了 `loop` 顶端。LEAST 的代码本身没有任何变化。

### 2.2 用同一棵反例树验证

| 步骤 | LEAST 选出 E | 是答案？ | 操作 |
|---|---|---|---|
| 1 | 根 | 否 | L(2)、R(4) 入表 |
| 2 | **L**（ĉ=2） | 否（非叶） | LL(20)、LR(∞) 入表 |
| 3 | **R**（ĉ=4） | 否（非叶） | RL(∞)、RR(10) 入表 |
| 4 | **RR**（ĉ=10） | **是！** | 输出 cost=10 ✓ |

### 2.3 正确性证明

设 LEAST 选出 E，E 被判定为答案结点，表中剩余任意结点记为 L：

$$\text{cost}(E) = c(E) = \hat{c}(E) \leq \hat{c}(L) \leq c(L)$$

- $c(E)=\hat{c}(E)$：来自 8.2 节第三条要求——**叶答案结点处** $\hat{c}=c=\text{cost}$
- $\hat{c}(E)\leq\hat{c}(L)$：LEAST 的选取保证（E 是表中 $\hat{c}$ 最小者）
- $\hat{c}(L)\leq c(L)$：来自 8.2 节第二条要求——$\hat{c}$ 是下界估计

因此 E 的真实代价不超过表中任何结点子树里能找到的最优答案代价，**E 就是全局最小成本答案结点**。$\blacksquare$

### 2.4 但是——算法 8.2 对 k-元组仍然不正确

上面证明有一个隐含前提：**E 是叶答案结点**，才能保证 $c(E)=\hat{c}(E)$（等号成立）。

k-元组中答案结点可能是**非叶结点**，此时只有 $\hat{c}(E)\leq c(E)\leq\text{cost}(E)$，等号不成立，整条证明链断裂。

**k-元组的反例：**

```
              根
         ┌────┴────┐
  A（答案，非叶）    B（非答案）
  ĉ=3  cost=8       ĉ=6
                     │
                D（答案，叶）
                ĉ=6  cost=6   ← 真正最优
```

算法 8.2 执行：根展开 → A(ĉ=3)、B(ĉ=6) 入表 → LEAST 选 **A** →
**A 是答案结点，立刻输出 cost=8，终止。**

B 的子树从未被搜索，真正最优的 D(cost=6) 被错过。

**错误根源**：$\hat{c}(A)=3$ 只是下界估计，A 自身的 $\text{cost}(A)=8$ 并不等于 $\hat{c}(A)$，
不能凭 $\hat{c}$ 最小断定 A 就是最优答案。

| 解的表达方式 | 答案结点位置 | 算法 8.2 保证最优？ |
|---|---|---|
| **固定长 n-元组** | 必为叶结点，$\hat{c}=c=\text{cost}$ 成立 | ✅ 保证 |
| **不定长 k-元组** | 可能为非叶结点，仅有 $\hat{c}\leq c\leq\text{cost}$ | ❌ 不保证 |

## 三、第二次修补：引入 $U$ 和 $u(X)$——修正 k-元组的正确性

### 3.1 核心思路转变

算法 8.2 的问题在于：**遇到答案结点就立刻终止**。

修正思路：**遇到答案结点时，不终止，而是把它的代价记录为当前已知最优值 $U$，然后继续搜索**，直到活结点表中所有剩余结点都不可能比 $U$ 更优时，才真正终止。

用上面的 k-元组反例重新追踪（引入 $U$ 后）：

| 步骤 | E | 操作 | U | ans |
|---|---|---|---|---|
| 1 | 根 | 展开 → A(ĉ=3)、B(ĉ=6) 入表 | ∞ | 0 |
| 2 | **A**（ĉ=3） | A 是答案，cost=8 < U=∞，**更新 U←8，ans←A，继续搜索** | 8 | A |
| 3 | **B**（ĉ=6） | 不是答案，展开 → D(ĉ=6) 入表 | 8 | A |
| 4 | **D**（ĉ=6） | D 是答案，cost=6 < U=8，**更新 U←6，ans←D，继续搜索** | 6 | D |
| 5 | 表为空 | 输出 U=6，ans=D ✓ | | |

找到了真正最优的 D。$U$ 的作用不只是剪枝——**它让算法在找到一个答案后不再停步，持续寻找更优的答案**。

> 注：对 n-元组，引入 $U$ 不影响正确性（算法 8.2 已经正确），只是加速剪枝、提高效率。
> **对 k-元组，引入 $U$ 是修正正确性的必要条件，而不只是优化手段。**

### 3.2 成本上界函数 $u(X)$

$U$ 可以通过两个渠道被收紧：
- 发现一个答案结点，用其**真实代价** $\text{cost}(X)$ 收紧
- 发现一个状态结点，用其**上界估计** $u(X)$ 收紧

$u(X)$ 是对 $c(X)$（结点 X 子树内最优答案成本）的上界估计，要求：

$$\hat{c}(X) \leq c(X) \leq u(X)$$

这是本节最重要的一条不等式。三点设计要求：

| 要求 | 说明 |
|---|---|
| 易于计算 | 类似 $\hat{c}$，通常由已走代价 + 对未来代价的粗略上界估计组成 |
| $c(X) \leq u(X)$ | 上界性质（与 $\hat{c}$ 的下界性质方向相反） |
| 叶答案结点处 $c(X)=u(X)$ | 精确收口 |

**区分 $U$ 与 $u(X)$：**

| | $U$ | $u(X)$ |
|---|---|---|
| 本质 | 算法运行中的**全局数值**，单调收紧 | 针对**特定结点 X** 的函数值 |
| 更新 | 被 $\text{cost}$ 或 $u(X)$ 拉低 | 由 X 自身在树中的位置决定，不随 $U$ 变化 |
| 作用 | 判断活结点死活（剪枝依据） | 提前收紧 $U$，加速剪枝 |

## 四、$U$ 的改值与剪枝规则（★易错考点★）

### 4.1 改值规则

| 情形 | 操作 |
|---|---|
| 发现答案结点 X，$\text{cost}(X) < U$ | $U \leftarrow \text{cost}(X)$，`ans ← X` |
| 发现状态结点 X，$u(X) < U$ | $U \leftarrow u(X)$，`ans ← 0`（子树里有更优的，但还没找到） |

### 4.2 剪枝规则——等号处理是核心易错点

| `ans` 状态 | $U$ 的来源 | 杀死条件 | 原因 |
|---|---|---|---|
| `ans` **有**结点 | 真实成本 | $\hat{c}(X) \geq U$（**含等号**） | 已有确认的最优答案，打平也没有意义，直接杀死 |
| `ans` **为空** | 上界估计 | $\hat{c}(X) > U$（**不含等号**） | 只知道"存在"不超过 U 的解，但还没找到；打平的 X 本身可能就是那个真正的答案 |

> **记忆口诀：有答案，等号杀；无答案，等号放。**

## 五、三个算法

### 5.1 算法 8.3：界函数 UB（被后两个算法调用）

```
procedure UB(X, U, ans)
  // ans: 当前已知最优答案结点，0 表示无
  // 当 X 是答案结点时 cost(X) 有意义，否则为 ∞
  if ĉ(X)>U or (ans有结点 and ĉ(X)=U)        ← 第一步：先判死活
    then return false
  endif
  if X是答案结点 and cost(X)<U                 ← 第二步：真实值收紧
    then U←cost(X); ans←X
  endif
  if u(X)<U then U←u(X); ans←0 endif          ← 第三步：估计值收紧
  return true
end UB
```

三步顺序固定：先判死活，再更新 $U$，最后通过返回 true。

### 5.2 算法 8.4：FIFOBB

```
procedure FIFOBB(T, ĉ, u, cost)
  E←T; U←∞; ans←0
  UB(E,U,ans); 初始化队列为空
  loop
    for E的每个儿子X do
      if B(X) and UB(X,U,ans) then ADDQ(X); PARENT(X)←E endif
    repeat
    loop                                       ← 内层循环
      if 队列为空 then 输出U和ans路径; return endif
      DELETEQ(E)
      if ĉ(E)<U or (ans无结点 and ĉ(E)=U)    ← 界定通过则跳出
        then exit
      endif
    repeat
  repeat
end FIFOBB
```

### 5.3 算法 8.5：LCBB

```
procedure LCBB(T, ĉ, u, cost)
  E←T; U←∞; ans←0
  UB(E,U,ans); 初始化活结点表（min-堆）为空
  loop
    for E的每个儿子X do
      if B(X) and UB(X,U,ans) then ADD(X); PARENT(X)←E endif
    repeat
    if 表为空 or 堆顶ĉ>U or (ans有结点 and 堆顶ĉ=U)  ← 界定杀死则终止
      then 输出U和ans路径; return
    endif
    call LEAST(E)
  repeat
end LCBB
```

### 5.4 FIFOBB 为什么多一层内层循环？

这是两个算法结构最显著的区别，根源在数据结构的性质不同：

**LCBB 用 min-堆**：堆顶永远是 $\hat{c}$ 最小的结点。只要堆顶都被判死，后面所有结点 $\hat{c}$ 只会更大，**全表皆死**，一次检查即可终止。

**FIFOBB 用队列**：队首只是"最早进队"的结点，与 $\hat{c}$ 大小无关。在它排队等待期间，$U$ 可能已经被其他分支收紧，导致它变成"过时的死结点"，但队列后面的结点 $\hat{c}$ 可能更小、仍然存活。因此必须用**内层循环**逐个弹出、逐个核验，直到找到第一个通过检验的结点（`exit` 跳出内层继续）或队列清空。

> 本质：队列不自排序 → 无法一次判定全表死活 → 必须逐个扫描。

## 六、两种剪枝的完整体系

| 剪枝类型 | 依据 | 剪掉的是 |
|---|---|---|
| **可行解剪枝** | 约束函数 $B$ | 不含任何可行解的子树 |
| **最优解剪枝** | $\hat{c}(X)$ 与上界 $U$ | 含可行解但不含最优解的子树 |

**剪枝发生的两个时间点：**
- X **入**活结点表时：接受 $B$ 检验 + UB 检验（双重过滤）
- X **出**活结点表时：再次接受 UB 检验（应对 $U$ 在 X 排队期间已被收紧的情况）

**算法终止条件（满足任一即可）：**
- 活结点表为空
- 活结点表中最小的 $\hat{c}$ 值也已不可能优于 $U$（全表皆死）

## 七、极大化问题

| 方法 | 做法 |
|---|---|
| **取负转换（推荐）** | 目标函数取相反数作为成本函数 $c$，直接复用本节所有算法 |
| **镜像修改** | 不等式方向全部反转：$u(X)\leq c(X)\leq\hat{c}(X)$；$U$ 变为下界（只增不减）；剪枝条件变为 $\hat{c}(X)\leq U$ 时杀死 |

## 八、易错点汇总

| 易错点 | 正确理解 |
|---|---|
| 以为算法 8.2 套上 LC-检索就能找到最优解 | 仅对 n-元组成立；k-元组（非叶答案结点）不成立，$\hat{c}=c$ 的等号无法保证 |
| 认为引入 $U$ 只是提高效率 | 对 n-元组是只提高效率；对 k-元组是**修正正确性**，没有 $U$ 算法本身就会给出错误答案 |
| 把 $\hat{c}$ 和 $u$ 的方向搞反 | $\hat{c}(X)\leq c(X)$（下界）；$u(X)\geq c(X)$（上界）；合起来 $\hat{c}(X)\leq c(X)\leq u(X)$ |
| 混淆 $U$ 与 $u(X)$ | $U$ 是全局数值（随时间收紧）；$u(X)$ 是针对特定结点的函数值（不随 $U$ 变化） |
| 等号杀死规则搞反 | `ans` 有结点时含等号杀死；`ans` 为空时不含等号（等号放行） |
| 以为 FIFOBB 和 LCBB 结构应该相同 | 队列无序所以多一层内层循环逐个核验；min-堆有序所以一次检查堆顶即可判定全表 |
| 发现答案结点后以为可以立刻输出 | 只是更新 $U$ 和 `ans`，必须继续搜索直到全表皆死才能确认这是全局最优 |

## 九、考点速查

| 考点 | 核心内容 |
|---|---|
| 算法 8.2 与 8.1 的区别 | 判断"是否答案"的时机从 `for` 内部移到 `loop` 顶端；LEAST 本身不变 |
| 算法 8.2 正确性的关键等式 | $\text{cost}(E)=c(E)=\hat{c}(E)\leq\hat{c}(L)\leq c(L)$，依赖叶答案结点处 $\hat{c}=c$ |
| 算法 8.2 对 k-元组失效的原因 | 非叶答案结点处 $\hat{c}\leq c\leq\text{cost}$，等号不成立，证明链断裂 |
| $U$ 对 k-元组的作用 | 不是只提高效率，而是**修正正确性**：找到答案不停，持续搜索直到确认全局最优 |
| 四函数不等式链 | $\hat{c}(X)\leq c(X)\leq u(X)$（$\text{cost}(X)$ 仅对答案结点有意义） |
| U 的两种改值来源 | 答案结点 $\text{cost}<U$：$U\leftarrow\text{cost}$，`ans←X`；状态结点 $u(X)<U$：$U\leftarrow u(X)$，`ans←0` |
| 等号杀死口诀 | 有答案含等号杀；无答案不含等号杀 |
| 算法 8.3（UB）三步顺序 | 先判死活 → 真实值收紧 $U$ → 估计值收紧 $U$ → 通过 |
| FIFOBB vs LCBB 结构差异 | 队列无序多内层循环；min-堆有序一次判定全表 |
| 两种剪枝 | $B$ 函数（可行解剪枝）+ $\hat{c}/U$（最优解剪枝） |
| 终止条件 | 表为空，或全表 $\hat{c}$ 均不优于 $U$ |
