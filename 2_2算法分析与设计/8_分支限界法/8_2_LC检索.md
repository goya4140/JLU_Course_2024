# 8.2 LC-检索 — 复习资料

> 本节承接 8.1：8.1 只是把"优先队列式"检索抽象地命名为 **LC-检索**，并未说明成本估计函数 $\hat{c}$ 该怎样设计；本节正式给出 $\hat{c}$ 的通用框架 $\hat{c}(X)=f(h(X))+\hat{g}(X)$ 与三点设计要求，是 **8.3（15-谜）、8.5（作业调度）、8.6（货郎担）** 三节具体 $\hat{c}$ 公式的共同理论源头，也是 8.4 节"求最小成本分支限界法"的铺垫。本节本身不直接对应某一道作业题，但是后续三节作业题解答的方法论基础。

---

## 一、LC-检索的优点 —— 为什么需要它

FIFO-检索的问题在于：**对下一个 E-结点的选择是死板的、盲目的**。它只看"谁先进队列"，对那些"更有希望快速通向答案结点"的活结点，没有给予任何优先权——这正是 8.1 节 4-皇后例子中 FIFO 处理 31 个结点（而回溯法只需 16 个）的根本原因：好的分支和坏的分支被一视同仁地排队等待。

**理想方案：** 给活结点表配一个"**有智力的**"成本函数 $c$，谁的 $c$ 值小（更接近答案、更有希望），就优先把谁取出来做下一个 E-结点。

> **一句话目标：** 给那些可能导致答案结点的活结点赋以优先次序。

这就是 LC（Least Cost）检索的出发点——本节接下来的全部内容，都是在回答"这个 $c$ 该怎么定义、怎么算"。

---

## 二、成本函数 $c$ 的量化方法

令 $X$ 是当前结点，$c(X)$ 关注**以 $X$ 为根的子树中的最小成本值**。根据问题性质不同，"成本"可以用三种思路量化：

| 方法 | 量化依据 | 适用场景 |
|---|---|---|
| **方法1** | 生成答案结点之前需要生成的**结点数** | 寻找"生成数目最少"的答案结点 |
| **方法2** | 答案结点的**路径长度** | 寻找"路径长度最短"的答案结点 |
| **方法3** | 问题描述中的**目标函数** | 寻找使目标函数取极值的答案结点（即最优解，组合优化问题） |

> **★本章约定★：** 当问题不存在目标函数时（即只是"找一个可行解"而非求最优解的场景，如 8.3 节 15-谜问题），本章统一采用**方法2**（路径长度）定义 $c$ 函数。而 8.5、8.6 节属于组合优化问题（有目标函数：罚款最小 / 周游成本最小），自然采用**方法3**。

---

## 三、成本函数 $c$ 的精确定义

成本函数的定义分两层：先给"解结点"定义**真实成本** $\text{cost}$，再借助 $\text{cost}$ 给**任意结点**定义 $c$。

### 3.1 真实成本函数 $\text{cost}(X)$（仅对解结点定义）

$$\text{cost}(X)=\begin{cases}\text{根到达 } X \text{ 的真实成本/代价（如路径长度、目标函数值等）} & X \text{ 是答案结点}\\ \infty & X \text{ 不是答案结点}\end{cases}$$

### 3.2 成本函数 $c(X)$（从"上帝视角"对任意结点定义）

$$c(X)=\begin{cases}\text{cost}(X) & \text{叶结点 } X \text{ 是答案结点}\\ \min\{\text{cost}(P)\mid \text{答案结点 } P\in\text{子树 } X\} & \text{子树 } X \text{ 中包含答案结点}\\ \infty & \text{子树 } X \text{ 不包含答案结点}\end{cases}$$

> **思考题：在什么样的状态空间树中会出现答案结点 $X$ 不是叶结点的情况？**
>
> **答案：** 当解空间用**不定长 $k$-元组** $(X_1,\ldots,X_k)$（$k\leq n$）表达时。此时树中任意一个满足约束条件的**中间结点**都可能本身就是一个完整可行解，不需要补全到第 $n$ 层才算数——典型例子是 8.5 节"带有期限的作业调度问题"：选中作业子集 $J=\{2,3\}$ 对应的结点（深度2）本身就是一个答案结点，即便树还可以继续往下扩展到更深的层。相对地，**固定长 $n$-元组**表达的树（如 15-谜、货郎担问题）中，只有走到第 $n$ 层的叶结点才能构成一个完整解，答案结点必为叶结点。

这一区分非常关键：它解释了为什么 $c(X)$ 的定义要同时覆盖"叶结点是答案结点"和"子树中（非叶）包含答案结点"两种情况——后一种正是为不定长元组问题预留的。

---

## 四、成本函数 $c$ 的例子（图解）

设 $c(X)$ = 到达答案结点的最短路径长度（即采用第二节的"方法2"）。

```
                    1 (c=3)
        ┌───────┬────┴────┬───────┐
      2(∞)    3(c=3)     4(∞)    5(∞)
            ┌────┼────┐
          6(∞)  7(∞)  8(c=3)
                    ┌────┴────┐
                 9(c=4)    10(c=3) ★答案结点
                    │
                 11(c=4) ★答案结点
```

- **蓝色（∞）结点**：2、4、5、6、7 —— 表示这些结点的子树**不能到达**任何答案结点
- **叶结点 10 和 11 是答案结点**：$\text{cost}(10)=3$（根→3→8→10，路径长度3），$\text{cost}(11)=4$（根→3→8→9→11，路径长度4）

**逐结点验算（体现 $c$ 的递归定义）：**

| 结点 | 子树中的答案结点 | $c(X)$ 计算 |
|---|---|---|
| 10（叶，答案） | 自身 | $c(10)=\text{cost}(10)=3$ |
| 11（叶，答案） | 自身 | $c(11)=\text{cost}(11)=4$ |
| 9（非叶，非答案） | 仅 11 | $c(9)=\min\{c(11)\}=4$ |
| 8 | 9 的子树、10 | $c(8)=\min\{c(9),\,c(10)\}=\min\{4,3\}=3$ |
| 3 | 6、7 子树为空，8 的子树含答案 | $c(3)=\min\{\infty,\infty,c(8)\}=3$ |
| 1（根） | 仅 3 的子树含答案 | $c(1)=\min\{\infty,c(3),\infty,\infty\}=3$ |

> **基于 $c$ 对活结点表检索的效果：** 若真能拿到精确的 $c$，从根出发每次都挑 $c$ 值最小的活结点：根(c=3)→只有结点3的 $c$ 有限，选3→只有结点8的 $c$ 有限，选8→{9(c=4), 10(c=3)} 中选**更小**的10→10 正是答案结点，**直接命中，不走 9→11 这条更长的弯路**。这就是"有智力的"检索效果。

## 五、成本函数 $c$ 的局限 —— 为什么还需要 $\hat{c}$

上一节的美好场景有一个致命问题：**$c$ 极难计算。**

- $c(X)$ 的定义建立在"答案结点的真实成本"之上，要严格求出 $c(X)$，必须先把以 $X$ 为根的整棵子树都生成出来，看看哪些是答案结点、它们的真实成本各是多少。
- 也就是说，**求 $c$ 的工作量与解决原问题本身具有相同的复杂度**——这等于说"为了搜索得快，先要把搜索做一遍"，自相矛盾。

**解决思路：** 基于 $c$，设计一个**易于计算**的成本估计函数 $\hat{c}$ 来替代 $c$ 对活结点表进行检索。设计 $\hat{c}(X)$ 时把握两个直觉：

- 很容易确定**根到 $X$ 已经产生的成本**（这部分是确定的、已发生的）
- 很容易**估计** $X$ 到一个答案结点**可能会产生**的成本（这部分允许是粗略估计）

> 这是一项富于**创造性**的工作——8.3、8.5、8.6 节分别针对 15-谜、作业调度、货郎担问题设计的具体 $\hat{c}$ 公式，正是这种"创造性"的三个范例。

---

## 六、成本估计函数 $\hat{c}$ 的定义与要求

$$\boxed{\hat{c}(X)=f(h(X))+\hat{g}(X)}$$

| 符号 | 含义 |
|---|---|
| $h(X)$ | 根结点到结点 $X$ 的成本（**已发生**的部分） |
| $\hat{g}(X)$ | $X$ 到最小成本答案结点的**估计**成本（**未来、待发生**的部分） |
| $f$ | 调整 $h$ 和 $\hat{g}$ 在 $\hat{c}$ 中**影响比例**的非负函数（最简单的取法是 $f(h(X))=h(X)$，即直接相加，下文 8.3/8.5/8.6 节的具体公式均采用这种简单形式） |

**对 $\hat{c}$ 的三点要求：**

1. **易于计算**，能够替代 $c$ 对活结点表进行检索（解决第五节的痛点）
2. $\hat{c}(X)\leq c(X)$（**下界性质**——保证 $\hat{c}$ 永远不会"高估"，从而不会让真正更优的分支被误判而提前出局）
3. 当答案结点 $X$ 是叶结点时，$c(X)=\hat{c}(X)$（**精确收口**——一旦真正走到答案结点，估计值必须等于真实值，不能再有误差）

---

## 七、LC-检索的定义；BFS 是 LC-检索的特例

**LC-检索（Least Cost search）：** 选取成本估计函数 $\hat{c}$ 值**最小**的活结点，作为下一个 E-结点。

**BFS-检索（广度优先）** 是 LC-检索的一种特殊情况：

$$f(h(X))=\text{根到结点 } X \text{ 的路径长度},\qquad \hat{g}(X)=0$$

也就是说，如果完全不去估计"未来还要付出多少代价"（$\hat{g}(X)\equiv 0$），只按"已经走了多深"来排序，LC-检索就退化成了按层展开的广度优先搜索。

> **思考题：只用 $\hat{g}(X)$ 为结点排序是否适合？即 $f(h(X))=0$。**
>
> 答案是**不适合**——这正是下一节要详细分析的"乐观的冒进"问题。

---

## 八、"乐观的冒进"—— 仅用 $\hat{g}(X)$ 排序的隐患（★重点考查★）

设 $\hat{c}(X)=\hat{g}(X)$（即完全丢掉 $f(h(X))$ 这一项，只看"未来还要花多少"）。

**问题机制：** 设子树 $X$ 的儿子结点是 $Y$，很容易出现 $\hat{g}(Y)\leq \hat{g}(X)$（直觉上，越往下走、离答案越近，"剩余代价"通常越小）。如果此时活结点表中其他结点的 $\hat{c}$ 值都比 $\hat{g}(X)$ 大，那么：

$$X \text{ 被选为 E-结点} \;\longrightarrow\; \text{生成 } Y,\ \hat{g}(Y)\leq \hat{g}(X) \;\longrightarrow\; Y \text{ 又被选为新的 E-结点} \;\longrightarrow\; \cdots$$

这个过程会**一路纵深检索下去**，直到子树 $X$ 整个被搜索完毕，才有可能转向树中其他分支。

**后果：** 算法只考虑了"未来可能的代价"（$\hat{g}$），完全忽略了"已经付出的代价"（$h$）。这使得算法**偏向纵深方向检索**——一旦踏入某个深邃但未必正确的分支，就会一条道走到底，可能**丢失掉那些更靠近根、本应优先被发现的答案结点**，导致搜索结果不理想。

> 这正是 $\hat{c}$ 的公式中**必须保留 $f(h(X))$ 这一项**的根本原因：不能只问"还要走多远"，也要问"已经走了多远"——两者加在一起，才能让"浅而优"的答案结点和"深而看似有希望"的分支得到公平的比较。

## 九、本节小结：$c$ 与 $\hat{c}$ 的关系、以及承上启下

### 9.1 关系小结

| 概念 | $T$ 是状态空间树，$X$ 是 $T$ 中任意结点 |
|---|---|
| $c(X)$ | 以 $X$ 为根的子树中**最小成本答案结点**的成本；难以**直接求出**（计算量与原问题同阶） |
| $\hat{c}(X)$ | 基于 $c$ 设计的**易计算**估计函数，$\hat{c}(X)=f(h(X))+\hat{g}(X)$，满足 $\hat{c}(X)\leq c(X)$，且叶答案结点处 $c=\hat{c}$ |
| LC-检索 | 用 $\hat{c}$ 对活结点表排序，**优先选择更靠近答案结点、同时又离根结点较近**的结点 —— 这正是 LC-检索相较 FIFO-检索的优点所在 |

### 9.2 承上启下：$\hat{c}$ 框架在后续小节中的实例化

8.3 / 8.5 / 8.6 三节的 $\hat{c}$ 设计，都是本节 $\hat{c}(X)=f(h(X))+\hat{g}(X)$（取 $f(h(X))=h(X)$）这一框架的具体实例：

| 问题 | $h(X)$（已发生的成本） | $\hat{g}(X)$（对未来代价的估计） |
|---|---|---|
| **8.3 15-谜问题** | $f(X)$：从初始排列到 $X$ 时棋牌已经移动的次数 | 不在目标位置的**非空白棋牌数目** |
| **8.5 带期限的作业调度** | （已选中作业不产生罚款，故这部分恒为0，体现为"不选才罚款"的特殊结构） | 下标 $<m$ 且**未被选中**的作业罚款之和 |
| **8.6 货郎担问题** | $f(X)=\sum a_i$：已选定路线 $1,X_1,\ldots,X_k$ 的成本和 | $l_{X_k}+\sum_{d\in S} l_d$：经过剩余结点回到起点的最短距离估计 |

> 三个具体设计都严格满足本节的两条硬性要求：**易计算**（都只需查表/求和，不需要搜索子树）、**$\hat{c}(X)\leq c(X)$**（都只统计"已知的下界"信息，绝不高估）。8.4 节会在此基础上再引入成本**上界** $U$，把"找一个答案结点"升级为"找**最小成本**的答案结点"。

## 十、易错点汇总

| 易错点 | 正确理解 |
|---|---|
| 混淆 $\text{cost}(X)$ 与 $c(X)$ | $\text{cost}$ 只对**解结点**（答案结点/非答案结点）定义；$c$ 是从"上帝视角"对**任意结点**（包括内部结点）定义，二者作用对象不同 |
| 认为 $c$ 函数可以直接拿来用 | $c$ 的计算量与原问题相同复杂度，**不可行**；实际算法用的永远是易计算的估计函数 $\hat{c}$ |
| 把 $\hat{c}(X)\leq c(X)$ 记反方向 | $\hat{c}$ 是**下界**估计（不能高估），而非上界；上界估计是 8.4 节的 $u(X)$，二者方向相反 |
| 忽视 $\hat{c}$ 在叶答案结点处必须等于 $c$ | 这一要求保证算法一旦真正找到答案结点，所得到的成本判断是**精确**而非估计的，否则无法判断是否真的找到了答案 |
| 认为 $\hat{c}=\hat{g}$（只看未来）总是更优 | 这恰恰是"乐观的冒进"的来源：忽略已发生代价 $h(X)$，会让算法偏向纵深检索，可能丢失更靠近根的答案结点 |
| 误以为 BFS 与 LC-检索是两种不相关的方法 | BFS 是 LC-检索在 $f(h(X))=$ 路径长度、$\hat{g}(X)=0$ 时的**特例**，二者共享同一个 $\hat{c}$ 检索框架 |
| 认为答案结点一定是叶结点 | 仅在**固定长 $n$-元组**表达的树中成立；**不定长 $k$-元组**（如 8.5 节作业调度问题）中，非叶结点也可能是答案结点 |

## 十一、考点速查

| 考点 | 核心内容 |
|---|---|
| LC-检索的动机 | FIFO 选取下一个 E-结点"死板、盲目"；用"有智力的"成本函数加快检索 |
| $c$ 的三种量化方法 | ① 结点数最少 ② 路径长度最短 ③ 目标函数取极值；本章无目标函数时用②（8.3）；有目标函数时用③（8.5、8.6） |
| $\text{cost}(X)$ 定义 | 答案结点：根到 $X$ 的真实成本；非答案结点：$\infty$ |
| $c(X)$ 定义（三分支） | 叶答案结点 $\to \text{cost}(X)$；子树含答案结点 $\to \min\{\text{cost}(P)\}$；子树不含答案结点 $\to \infty$ |
| 答案结点非叶结点的情形 | 仅出现在**不定长 $k$-元组**表达的解空间树中（如作业调度问题） |
| $c$ 的局限 | 求 $c$ 需要先生成整棵子树，计算复杂度与原问题相同 |
| $\hat{c}$ 公式 | $\hat{c}(X)=f(h(X))+\hat{g}(X)$ |
| $\hat{c}$ 三点要求 | 易计算；$\hat{c}(X)\leq c(X)$；叶答案结点处 $c=\hat{c}$ |
| LC-检索定义 | 选 $\hat{c}$ 值最小的活结点为下一个 E-结点 |
| BFS 与 LC-检索关系 | BFS 是 LC-检索的特例：$f(h(X))=$ 路径长度，$\hat{g}(X)=0$ |
| "乐观的冒进" | 仅用 $\hat{g}(X)$（$f(h(X))=0$）排序时出现，导致算法偏向纵深检索，可能丢失靠近根的答案结点 |
| 后续小节 $\hat{c}$ 实例化 | 15-谜：移动次数+错位棋牌数；作业调度：未选作业罚款和；货郎担：已走成本+剩余最短距离估计 |
